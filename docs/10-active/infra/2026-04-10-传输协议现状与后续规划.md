## 1. 当前实现现状

## 1.1 一句话结论

截至 **2026-04-10**，当前线上 gate 协议实际是：

- **自定义二进制消息格式**
- **基于 `{packet, 4}` 的长度前缀 TCP**
- **TCP 主通道 + UDP fast-lane 路径**
- **UDP attach 握手、Movement 上行与 `PlayerMove` 下行广播都已落地**
- **没有启用中的 KCP 路径**
- **gate 运行时核心路径已经不走 protobuf**

所以当前状态可以明确表述为：

> **自定义二进制协议 + TCP 主通道已经上线。**
>
> **UDP fast-lane attach bootstrap、Movement 上行和 `PlayerMove` 下行广播都已落地，并且已增加重附着替换、空闲过期、TCP 关闭清理，以及“失败后 TCP fallback + backoff 重试 + cooldown”恢复策略；更强的 snapshot / resync 机制仍未完成。**

---

## 1.2 当前实际使用的传输

`gate_server` 通过 `:gen_tcp.listen/2` 监听，并开启 Erlang 的 packet framing：

- `apps/gate_server/lib/gate_server/worker/tcp_acceptor.ex`

当前行为：

- 传输层：**TCP**
- 分帧方式：**4 字节大端长度前缀**
- socket 模式：`[:binary, packet: 4, active: true, reuseaddr: true]`

这与 `2026-04-10-线协议规范.md` 中的现行定义一致。

---

## 1.3 当前实际使用的消息编码

`GateServer.Codec` 是当前 gate 收发消息实际使用的 codec：

- `apps/gate_server/lib/gate_server/codec.ex`
- `apps/gate_server/lib/gate_server/worker/tcp_connection.ex`

当前实现特征：

- 消息头形态：`<<msg_type::8, payload::binary>>`
- 数值字段使用 BEAM 二进制模式匹配编码/解码
- 向量使用 `float-64`
- 每种消息都有显式的 decode / encode 逻辑

这意味着：

- 项目已经完成“从 protobuf 转向自定义二进制”的关键迁移
- 当前线上 gate 路径不再依赖生成式 protobuf struct

## 1.4 当前消息集合

### 客户端 → 服务端

| ID | 名称 | 用途 | 当前传输 |
|---|---|---|---|
| `0x01` | `Movement` | 高频角色移动输入 | TCP / UDP fast lane（已附着会话优先 UDP） |
| `0x02` | `EnterScene` | 请求进入场景 | TCP |
| `0x03` | `TimeSync` | 请求时间同步 | TCP |
| `0x04` | `Heartbeat` | 心跳 / 保活 | TCP |
| `0x05` | `AuthRequest` | 用户名 + token/code 鉴权 | TCP |
| `0x06` | `FastLaneRequest` | 请求 UDP attach ticket | TCP |
| `0x07` | `FastLaneAttach` | 用 ticket 绑定 UDP peer | UDP |

### 服务端 → 客户端

| ID | 名称 | 用途 | 当前传输 |
|---|---|---|---|
| `0x80` | `Result` | 通用结果 / ack | TCP；`Movement` ack 可走 UDP |
| `0x81` | `PlayerEnter` | AOI 进入广播 | TCP |
| `0x82` | `PlayerLeave` | AOI 离开广播 | TCP |
| `0x83` | `PlayerMove` | AOI 移动广播 | TCP fallback / UDP fast lane |
| `0x84` | `EnterSceneResult` | 进入场景响应 | TCP |
| `0x85` | `TimeSyncReply` | 时间同步响应 | TCP |
| `0x86` | `HeartbeatReply` | 心跳响应 | TCP |
| `0x87` | `FastLaneResult` | 返回 UDP 端口与 ticket | TCP |
| `0x88` | `FastLaneAttached` | UDP attach ACK | UDP |

更详细字段布局请看 `2026-04-10-线协议规范.md`。

## 1.5 与旧迁移目标相比，当前进度到了哪里

`2026-04-07-增量迁移计划.md` 里的方向大致是：

- 先摆脱 protobuf
- 切到自定义二进制消息
- 可靠流量先继续走 TCP
- 高频移动流量以后再考虑走 KCP 或原生 UDP

这说明：

- **协议格式迁移已经完成**
- **传输层初步分流已经完成**
- **当前真正未完成的是恢复/纠偏与运行时治理**

也就是说，项目已经走到了：

> “TCP + 自定义二进制 + UDP fast lane” 已经是现实；
> “更强的恢复/纠偏/治理能力” 才是下一阶段目标。

---

## 1.6 目前还没有实现的东西

当前仓库运行时**尚未实现**：

- movement / broadcast 的快照纠偏与恢复机制
- 不可靠移动通道的 sequence / ack / 丢包恢复逻辑
- 面向丢包场景的 movement resync / snapshot recovery
- 更细的 stale peer / 网络抖动 / 运营观测治理
- 是否继续引入 KCP 的最终决策

## 2. 当前架构总结

今天的传输栈可以概括成：

```text
client
  -> TCP control plane
  -> UDP fast lane

TCP:
  auth / enter-scene / time-sync / heartbeat / reliable control flow

UDP:
  movement uplink / player-move downlink
```

这个结构已经比旧 protobuf/raw framing 路径清晰很多，但它仍然还有一个明显缺口：

> **fast lane 虽然已经承载 movement 热路径，但恢复/纠偏和长期治理能力还不够强。**

当前实现已经补上了一个重要的运行时保护：

- UDP fast-lane 失败时，客户端会**立即回退 TCP**
- 随后不会无限重试，而是按固定 backoff 窗口重新申请 fast-lane
- 连续失败达到阈值后，会进入 cooldown，再开启下一轮恢复

这让 demo 和本地运行时在弱网/端口抖动场景下更接近“先保活、再恢复低延迟”的实际工程策略。

这正是当前实现与更完整 MMO 风格传输架构之间的主要差距。

---

## 3. 建议的目标架构

建议未来拆成两类通道：

### 可靠通道

初期继续保留在 **TCP**：

- auth / login
- enter-scene 请求
- 通用结果 / 错误响应
- 会话初始化
- 聊天、背包、战斗、事务型操作

### 高频通道

在后续分流完成后迁移到 **UDP 系列传输**：

- movement uplink（`Movement`）
- movement broadcast downlink（`PlayerMove`）
- 可选的低延迟 time sync / latency measurement

目标形态大致是：

```text
client
  -> TCP reliable channel
  -> UDP/KCP fast channel

TCP:
  auth, scene entry, 可靠控制流, 事务型操作

UDP/KCP:
  movement input, movement broadcast, 低延迟状态同步
```

## 4. 后续实施建议

## Phase 0 —— 先冻结并说清楚“当前只有 TCP”

目标：

- 明确告诉所有客户端/服务端开发者：当前线上行为是**仅 TCP + 自定义二进制**

建议动作：

1. `2026-04-10-线协议规范.md` 继续作为**当前线格式**的事实来源
2. 本文继续作为**现状 + 演进计划**的事实来源
3. 不要把未来 UDP 设计和当前线上规范混写在一起，除非代码真的已经变成双栈

完成标准：

- 团队内部任何人都能明确回答“今天线上到底跑什么”

---

## Phase 1 —— 先决定高频通道到底是 KCP 还是原生 UDP

目标：

- 在真正实现前先选定 fast lane 技术路径

两个可选方向：

### 方案 A —— KCP over UDP

优点：

- 比自己手写可靠层更稳妥
- 更容易得到有序/半可靠行为
- 如果 movement 仍希望保留部分恢复语义，会更省事

缺点：

- 协议复杂度更高
- 调参成本更高
- 如果 movement 本身主要是“可丢弃状态”，可能会偏重

### 方案 B —— raw UDP

优点：

- 延迟最低、结构最简单
- 如果 movement 就是瞬时状态，它更接近 MMO 常见做法
- 序列、插值、回滚、纠正逻辑都能自己定义

缺点：

- 要自己做更多会话与丢包恢复逻辑

建议的决策标准：

- 如果团队想先降低实现风险：优先 **KCP**
- 如果团队希望更 MMO-native 且愿意自己做同步/纠正：优先 **raw UDP**

完成标准：

- 文档里明确写清楚 fast lane 最终选型

---

## Phase 2 —— 加入双通道会话握手

2026-04-10 实施进展：

- gate 已能在已认证 TCP 会话上签发 fast-lane ticket
- gate 已提供 UDP attach 握手入口
- movement uplink 已可通过已附着的 UDP peer 发送
- `PlayerMove` 下行广播已迁入 UDP
- 已补充重附着替换、空闲过期与 TCP 关闭清理等基础治理


目标：

- 由 TCP 负责建立/认证会话
- 由 UDP/KCP 安全地附着到这个已存在的会话上

建议动作：

1. 客户端先走 TCP 完成 auth
2. 服务端返回一个 session 标识 / ticket
3. 客户端携带该 ticket 走 UDP/KCP 建立 fast lane
4. gate 将 UDP/KCP channel 绑定到既有 TCP 会话

这个阶段的关键不是“先跑 UDP 包”，而是：

> **先确保 UDP/KCP 永远不是匿名裸连接。**

---

## Phase 3 —— 迁移 movement uplink

2026-04-10 实施进展：

- attached UDP peer 已可发送 `Movement`
- gate 会按已附着会话把 UDP movement 投递到 scene
- TCP 主通道仍保留可靠控制流

目标：

- 先把客户端高频输入从 TCP 挪到 fast lane

建议顺序：

1. 先保留 TCP movement 路径作为 fallback
2. 新增 UDP/KCP movement 上行
3. gate 按 session 将 movement 投递到 scene
4. 稳定后再逐步下掉 TCP movement 热路径

这样做的好处是：

- 可以做灰度切换
- 出问题时可以回退到 TCP
- 客户端演进成本更低

---

## Phase 4 —— 迁移 `PlayerMove` 高频下行广播

当前阶段结论：

- uplink 已经迁移
- `PlayerMove` 下行广播也已经迁移
- `MovementResult` 已开始回传服务端权威位置，作为最小纠偏信号
- 下一步重点应转向 snapshot / resync / sequence 等更强恢复能力

目标：

- 把其他玩家的高频位置更新广播也迁到 fast lane

原因：

- 单纯把 uplink 迁到 UDP/KCP 还不够
- 如果下行广播仍全走 TCP，热点场景下仍可能被可靠流量队头阻塞

建议做法：

- 高价值事务消息继续留在 TCP
- 高频位置广播走 UDP/KCP
- 必要时由周期 snapshot 做纠偏

---

## Phase 5 —— 增加同步恢复与纠偏机制

目标：

- 允许高频通道丢包，但整体体验仍然稳定

建议补的能力：

- 位置快照
- 周期性 authoritative correction
- 基于序号的过期包丢弃
- 必要时从 TCP 通道补发关键同步信息
- 更细的 retry/jitter/observability 策略（当前已具备 fallback + backoff + cooldown 基础版）

这个阶段决定系统最终是否能承受：

- 高频输入
- 广播热点
- 网络抖动
- 弱网丢包

---

## 5. 关键前提：不要带着“未收口的协议语义”去做双通道

在真正继续扩大 UDP/KCP 能力之前，有一个前置条件必须满足：

> **先把当前 TCP 协议契约做稳。**

也就是应该先解决：

- 状态机约束
- auth / scene 依赖 fail-closed
- cid / session 绑定
- request/reply 关联语义
- time sync 契约
- fast lane 的基础生命周期治理（重附着、空闲过期、TCP 关闭清理）

否则一旦开始双通道，原本就不清晰的东西会同时跨越：

- TCP
- UDP/KCP
- session
- scene
- 客户端状态机

到时候复杂度会指数级上升。

这个问题已经在 `2026-04-10-gate_server协议补全计划.md` 中做了更细的拆分。

---

## 6. 当前最合理的判断

截至现在，这个项目在传输层上的真实状态是：

- **自定义二进制 + TCP** 已完成
- **UDP fast lane 分流** 已经开始并承载了 movement 热路径
- **snapshot / resync / sequence 治理** 仍未完成

所以当前最准确的对外表述应该是：

> **现在已经完成协议格式迁移，并且已经开始传输层分流；当前真正的重点是把 fast lane 的恢复/纠偏能力补完整。**

下一步最稳妥的推进方式不是马上继续扩大协议面，而是：

> **在当前 UDP fast lane 基础上，继续补 snapshot / resync / sequence 等恢复治理能力。**

这是当前风险最低、返工最少、长期维护性最好的路线。
