# 移动同步扩展设计：跳跃 / 位移技能 / UE5 迁移（2026-04-20）

> 基于 `docs/2026-04-20-移动同步架构实现.md` 的当前架构，规划三个尚未落地的扩展方向。本文档为**设计稿**，代码尚未实现。

> **2026-07-14 客户端口径全文覆盖（覆盖本文所有相反表述）**：本文所有 Bevy、Bevy + UE5 双客户端并行、先做 Bevy 再迁移 UE5 的内容只作历史迁移证据，不是 active 目标、默认读取对象、CI / smoke 门禁或待决策任务。唯一现役 UE5 客户端是 `clients/Voxia`。服务端协议增量约束与 movement / reconcile 设计事实仍可继续参考；任何现役客户端落地必须以 Voxia 当前代码和 current-truth 为准另行排期，不得继续执行本文的 Bevy 任务。

## 1. 设计目标

在不推翻现有"预测 + 对账 + 视觉平滑"架构的前提下，扩展到：

1. **跳跃**：带空中状态的离散弹道。
2. **位移技能**：由技能触发的脚本化移动（闪现 / 冲锋 / 跳斩 / 钩锁）。
3. **UE5 客户端**：把当前 Bevy 客户端替换为 Unreal 5，服务端二进制协议不动。

三者共享一个核心判断：**任何位置变化都必须过 `LocalPredictionRuntime` + `reconcile` 流水线，不能绕过**。绕过就产生第二套位置真值，和走路打架。

## 2. 跳跃（Jump）

### 2.1 定位

跳跃 = 一个瞬时输入（起跳）+ 一段弹道（升空-顶点-落地）。适配现有流程只需**三个小扩展**，不需要新子系统。

### 2.2 输入帧

复用 `MoveInputFrame.flags`：
```rust
const MOVEMENT_FLAG_BRAKE: u8 = 0x01;
const MOVEMENT_FLAG_CROUCH: u8 = 0x02;
const MOVEMENT_FLAG_JUMP: u8 = 0x04;   // ← 新增
```

起跳帧由客户端按键时写入一次，后续空中帧仅携带方向输入（air control）。

### 2.3 状态扩展

```rust
enum MovementMode {
    Grounded,
    Falling,    // ← 新增：正在上升或下落
    Landing,    // ← 新增（可选）：着地瞬间的 0.1 s 窗口，用于硬着陆判定
}
```

`PredictedMoveState.movement_mode` 本来就预留了这个枚举，现在填充后续变体。

### 2.4 积分

`predictor::step` 在空中分支要改：

```rust
fn step(prev, input, profile) -> PredictedMoveState {
    match prev.movement_mode {
        MovementMode::Grounded => {
            if input.has_flag(MOVEMENT_FLAG_JUMP) && prev.is_on_ground() {
                let v = prev.velocity.with_z(profile.jump_impulse);
                return PredictedMoveState {
                    velocity: v,
                    movement_mode: MovementMode::Falling,
                    ..step_grounded(prev, input, profile)
                };
            }
            step_grounded(prev, input, profile)
        }
        MovementMode::Falling => step_air(prev, input, profile),
        // ...
    }
}

fn step_air(prev, input, profile) -> PredictedMoveState {
    let dir = input.input_dir * profile.air_control;  // air_control ≈ 0.3
    let desired_v = dir.extend(0.0) * profile.max_speed;
    let v = clamp(prev.velocity + (desired_v - prev.velocity).scale(profile.air_accel * dt), profile.max_speed);
    // 重力
    let v = v.with_z(v.z - profile.gravity * dt);
    let pos = prev.position + v * dt;
    PredictedMoveState { position: pos, velocity: v, movement_mode: MovementMode::Falling, ... }
}
```

`MovementProfile` 需要加 `jump_impulse / gravity / air_control / air_accel` 参数。

### 2.5 对账差异

**跳跃对账比走路严格一个数量级**，因为：
- Z 轴速度是**离散事件**：起跳瞬间 `v.z = jump_impulse`，错过一个 ack 就差一大截。
- 落地的 Z 坐标由服务器 rapier 决定，客户端无碰撞体，猜不准落点。

策略：
- `soft_threshold` 保持 2.0 u。
- **当 `movement_mode == Falling` 且 drift > 1.0 u 时提前走 Replayed 分支**（灵敏度 2 倍）。
- 落地 tick 的 ack 一到，**强制 push_sample** 并 `pending_correction = new - old`（走视觉平滑），避免悬浮/穿地。

### 2.6 动画协同

- 起跳：客户端按键即播起跳动画，`flags |= JUMP`。
- 上升：动画根据 `velocity.z` 决定"上升/顶点/下落"的 blend。
- 落地：服务器下发 `movement_mode = Grounded` 的 ack 后，客户端切着地动画。硬着陆（`v.z < -threshold`）可以播额外的屈膝 + 摄像机抖。

## 3. 位移技能（Displacement Skill）

### 3.1 为什么不是"纯技能"，也不是"纯移动"

位移技能是**混合物种**：

| 维度 | 技能特征 | 移动特征 |
|---|---|---|
| 触发 | 按键 + CD + 费蓝 + 目标校验 | 无 |
| 表现 | 动画、特效、UI 反馈 | 位置变化 |
| 执行 | 离散事件、可拒绝、可中断 | 连续状态、可回滚 |
| 权威 | 技能校验在服务端 | 位置权威在 rapier |

**结论**：**在移动体系内开一个 `MovementMode::Scripted` 子模式，由技能体系触发**。两个体系不合并，但有清晰的单向接口：技能 → 授权一段 scripted 移动 → 移动体系执行。

### 3.2 核心数据结构

```rust
// sim/types.rs
enum MovementMode {
    Grounded,
    Falling,
    Scripted(ScriptedSegment),   // ← 新增
}

struct ScriptedSegment {
    source_skill: u32,         // 技能 id，客户端据此找动画
    start_tick: u32,           // 服务器授权的起始 tick
    duration_ticks: u16,       // 总时长，0 = 瞬移
    trajectory: Trajectory,
    interruptible: bool,
    input_override: InputOverride,
}

enum Trajectory {
    Blink    { target: Vec3 },                           // 闪现
    Linear   { start: Vec3, end: Vec3 },                 // 冲锋
    Parabolic{ start: Vec3, end: Vec3, peak: f32 },      // 跳斩
    Homing   { target_id: u64 },                         // 盲僧 Q 二段
}

enum InputOverride {
    Locked,          // 完全脚本化，禁止转向（闪现、冲锋）
    SteerableAngle,  // 可转向但不可中止（踏前斩）
    Cancellable,     // 可按键取消
}
```

### 3.3 确定性积分

```rust
fn step_scripted(prev, input, seg) -> PredictedMoveState {
    let t = ((input.client_tick - seg.start_tick) as f32)
            / (seg.duration_ticks.max(1) as f32);
    let pos = match seg.trajectory {
        Trajectory::Blink { target }              => target,
        Trajectory::Linear { start, end }         => start.lerp(end, t.clamp(0.0, 1.0)),
        Trajectory::Parabolic { start, end, peak }=> parabola_lerp(start, end, peak, t),
        Trajectory::Homing { target_id }          => resolve_homing(target_id, t),
    };
    let mode = if t >= 1.0 { MovementMode::Grounded } else { prev.movement_mode };
    PredictedMoveState { seq: input.seq, tick: input.client_tick, position: pos, velocity: derive_v(prev, pos, dt), movement_mode: mode, ... }
}
```

**关键性质**：`step_scripted` 是纯确定性函数。只要两端拿到同一个 `ScriptedSegment`，每 tick 的预测位置逐字节相同——对账自然走 Accepted 分支，现有 reconcile 逻辑**无需改动**。

### 3.4 协议增量

**上行**：技能走现有 `CastSkill` 消息，不变。

**下行**：`CastAck` 追加字段：
```
CastAck {
  seq: u32,
  accepted: bool,
  reason: u8,                         // 0=ok / 1=CD / 2=oom / 3=oor / 4=interrupted
  scripted: Option<ScriptedSegment>,  // ← 授权的位移段
}
```

`MovementAck` 本身**不需要改动**——位置还是位置，`movement_mode` 字段能传 `Scripted` 变体即可。

codec 需要 bump 协议 minor version，但老客户端可以通过忽略未知字段兼容。

### 3.5 客户端预测策略

分两档：

| 类型 | `duration_ticks` | 预测策略 | 理由 |
|---|---|---|---|
| Blink | 0 | **乐观**：按键即传送 | 服务器几乎总是接受，回滚肉眼不可见 |
| Dash / Leap / Charge | > 0 | **悲观**：先播前摇，等 `CastAck` 再执行 scripted 段 | 300 ms 的滑铲被拉回视觉上藏不住 |
| Pull / Push（被动位移） | > 0 | **只能等授权**：由服务器主动推送 | 客户端无法预测被钩 |

**前摇失败容错**：客户端发 `CastSkill` 后启动 300 ms 定时器，超时自动打断前摇回 idle，防止丢包导致动画卡死。

### 3.6 打断 / 叠加（免费获得）

这是把位移技能塞进移动体系的最大收益：

| 场景 | 处理 |
|---|---|
| 冲锋中被眩晕 | 服务器下发 `MovementAck { movement_mode: Grounded, position: X }`，客户端 reconcile 发现 mode 切换 → 自动中止 Scripted 段 → 视觉 `pending_correction` 接管余下位置差 |
| 冲锋中被钩拉走 | 服务器下发新的 `Scripted(Homing)` segment，`start_tick` 覆盖前一段 → predictor 自然改向 |
| 冲锋撞墙 | 服务器在 tick N 产生 rapier 碰撞 → 下发 segment 的 `truncated_at_tick = N`，客户端 reconcile 把冲锋截断 |
| 跳跃中施放闪现 | `Falling` → `Scripted(Blink)`，落地点由 Blink.target 决定 |

**全部走同一条 reconcile 路径**，不需要为"技能位移"写独立的插值 / 快照 / 回滚。

### 3.7 效果副作用的边界

以下**严禁**在前摇期或预测执行期就作用：
- 扣蓝、扣 CD（必须等 `CastAck.accepted = true`）
- 伤害、debuff（必须等 `SkillHit` 服务端广播）
- 位移带来的状态变化（如"进入隐身"）必须等 `Scripted` segment 的 ack

客户端前摇动画 = **纯表现**，不改任何 gameplay state。这是整个系统能安全预测的基石。

## 4. UE5 客户端迁移

### 4.1 核心判断

**对账协议 ≠ 物理引擎**。只要线上协议不变，客户端实现用 Bevy / UE5 / Unity 都行。

保留不变：
- `MoveInputFrame { seq, client_tick, input_dir, dt_ms, flags, speed_scale }` 上行
- `MovementAck { auth_tick, ack_seq, position, velocity, movement_mode }` 下行
- `CastAck.scripted: Option<ScriptedSegment>` 下行（未来扩展后）
- codec（见 `docs/2026-04-10-线协议规范.md`）

服务端 scene_server 完全不动。

### 4.2 三条迁移路径

#### 路径 A：UE5 `CharacterMovementComponent`（CMC）+ 自定义网络层（推荐）

CMC 自带"客户端预测 + 服务器权威 + 回放 + 视觉 smoothing"，和我们现有架构**完全同构**：

| 我们的 | UE5 CMC |
|---|---|
| `MoveInputFrame` | `FSavedMove_Character` |
| `MovementAck` | `FClientAdjustment` / `ServerMove_Corrections` |
| `LocalRenderPrediction.pending_correction` | `NetworkSmoothingMode::Exponential`（原生） |
| `ScriptedSegment` | `FRootMotionSource`（原生） |
| 重放 `step_kinematic` | `CombineWith` replay |

适配点：
1. 写一个自定义 `UNetConnection` / `UNetDriver`，把 CMC 的 `FSavedMove` 序列化成我们的二进制 `MoveInputFrame`。
2. 把 scene_server 下行的 `MovementAck` 翻译成 CMC 期望的 `ServerMove_Corrections`。
3. `ScriptedSegment::Linear / Parabolic / Homing` 映射到 `FRootMotionSource_MoveToForce / JumpForce / MoveToDynamicForce`。
4. 动画、蓝图、特效生态**全部免费继承**。

**推荐度：★★★★★**。UE5 原生支持的东西太多了，自己重写不划算。

#### 路径 B：绕开 CMC，自写 `UCharacterMovement`

把 `sim/predictor.rs` + `sim/reconcile.rs` 1:1 移植成 C++ Component。

| 优点 | 缺点 |
|---|---|
| 行为完全可控，与 Bevy 客户端同源 | 失去 UE 生态：root motion、动画同步、lag compensation 辅助都要自己重写 |
| 迁移 risk 最低 | 代码量翻倍 |

**推荐度：★★**。除非 CMC 的某个行为特性绝对不能接受，否则不选。

#### 路径 C：两端都改用 rapier3d-f64 + 手写网络层

把 UE5 整个替换成 Rust 渲染内核，或者嵌入 rapier。

**推荐度：★**。不实际，工程代价爆炸。

### 4.3 迁移分阶段路线

1. **阶段 0 ~ 2 周**：在 UE5 搭空工程，对接 `auth_server` HTTP + `gate_server` TCP，实现登录 + 进入场景（只有一个方块）。
2. **阶段 1 ~ 4 周**：把 CMC 的 default 实现跑通，用 CMC 原生 replication 打到 scene_server。差距用"网络层翻译"桥接，先不关心优化。
3. **阶段 2 ~ 2 周**：跑 `scripts/e2e-live-movement` 的 UE5 版本，确认 hard_snap = 0、drift < 8 u。
4. **阶段 3 ~ 4 周**：加跳跃（同步扩展 `MovementProfile` + `MOVEMENT_FLAG_JUMP`）。
5. **阶段 4 ~ 6 周**：加位移技能（同步扩展 `CastAck.scripted`）。
6. **阶段 5 ~ 持续**：接入 GAS（Gameplay Ability System）做完整技能/Buff 栈。

每个阶段都能独立灰度，不依赖后续阶段。

### 4.4 兼容期双客户端并行

过渡期 Bevy + UE5 必须能同时连同一个服务器：
- 服务端不识别客户端类型，只认协议字段。
- 协议扩展必须**增量**（新增字段用 Option 或 tag 扩展），不能改现有字段语义。
- CI 增加一条"同一场景 1 × Bevy + 1 × UE5 对穿跑 1 分钟"的 smoke。

## 5. 落地建议顺序

下面是若要立刻开工的优先级：

| 阶段 | 目标 | 代价 | 收益 |
|---|---|---|---|
| 1 | `MovementMode::Falling` + `MOVEMENT_FLAG_JUMP`（跳跃最小实现） | 2~3 天 | 游戏性明显提升 |
| 2 | `MovementMode::Scripted(Blink)`（瞬移） | 1 周 | 为位移技能打底，零弹道风险 |
| 3 | `ScriptedSegment::Linear`（冲锋） | 1 周 | 完整验证 scripted pipeline |
| 4 | `CastAck.scripted` 协议扩展 + 技能服务端 DSL | 2 周 | 真正打通"技能 → 移动" |
| 5 | UE5 客户端阶段 0 ~ 2 | 6~8 周 | 生态跃迁 |
| 6 | `Parabolic / Homing` + Cancellable + 被击退 | 3 周 | 完整位移技能覆盖 |

每阶段完成都要重跑：
- `cargo test --lib --no-fail-fast`（单元回归）
- `scripts/e2e-live-movement.ps1`（真实链路回归）
- 指标门槛不变：`total_hard_snaps = 0`，`drift ≤ 8 u`。
