# 订阅活性根因分析 + 正交修复设计稿（原 workflow 产出，根因部分权威）

> 状态：设计稿（开工前必读，呼应 `docs/2026-06-27-架构设计指导思想-系统正交.md`）
> 症状：玩家挖/放方块被服务端 `accepted`、`chunk_version` 在涨，但客户端 spawn 后收到 0 个 `0x62/0x63/0x69`，编辑不可见（订阅静默失效）。当前临时修复=客户端每 10s 全半径重订阅（band-aid，浪费且不正交）。
> 目标：把"订阅"这块地基做正交、做扎实——**连接驱动活性**，edit 只依赖"订阅区块的变更必回推"这一稳定契约。

## 0. 关键代码事实核验（本稿据此推导，已逐条读码确认）

下面几条是判定根因的"地基事实"，我直接读码确认（不是转述研究）：

1. **`push_chunk_delta` 对订阅者 fan-out 时，零 lease/epoch 校验**。`chunk_process.ex:4378` 是 `Enum.each(state.subscribers, fn {subscriber, _} -> send(subscriber, {:voxel_chunk_delta_payload, ...}) end)`，只看 `state.subscribers` 这张本进程内存表，不读 lease、不比 owner_epoch。**结论:lease 续约抬 epoch 本身，绝不会让一个"已经注册在该 ChunkProcess 上的活订阅者"收不到 delta。** 这推翻了 lease-region-rebind 研究里"续约抬 epoch → 旧订阅静默失效"的推测。

2. **`lease_stale?` 只在 `run_simulation_tick` 跳过该 tick（`chunk_process.ex:1597`），从不丢订阅者、不停进程、不发 0x69**。stale lease 仅暂停模拟,**编辑路径(apply_intent→push_chunk_delta)与订阅 fan-out 完全不受影响**。

3. **ChunkProcess 没有 `terminate/2`**（grep 零命中）。0x69 唯一发出点是 `handle_call({:invalidate_subscribers})`（`chunk_process.ex:1049`），唯一外部调用链是 World migration cutover → `ChunkDirectory.invalidate_chunk` → `ChunkProcess.invalidate_subscribers`。**任何自身死亡路径(crash/idle :stop/supervisor shutdown)都不发 0x69。**

4. **ChunkDirectory 不维护订阅者表**。它只 `Process.monitor` 每个 ChunkProcess pid，`{:DOWN}` 仅从 `chunks`/`monitors` 表删行（`chunk_directory.ex:628`），无从广播 0x69（它根本不知道谁订阅了）。

5. **Gate `SubscriptionWorker` 只 monitor `connection_pid`**（`subscription_worker.ex:107`），不 monitor ChunkProcess pid。ChunkProcess 死亡在 Gate 侧不产生任何 `:DOWN`，订阅静默变死。

6. **RouteCache 把"lease 临近过期"当 miss**（`route_cache.ex:101` `fresh? = expires_at_ms > now + refresh_window`）。**只要连接真的去 route 那个 region**（subscribe/AOI 扩展走 `route_cached`），就会 miss→打 World→触发 `renew_region_lease` 续约。**问题恰恰在"连接此刻不去 route"**——站着不动、AOI 不变、不重订，就永远不 miss、永远不续。

7. **`sweep_expired_regions`（`map_ledger.ex:312`）回收过期 region 时,只删目录行+内存 assignment/lease,从不调 `scene_invalidator`、从不通知 Scene ChunkProcess**——它根本不知道 Scene 侧有没有活订阅者。

## 1. 根因判定

### 1.1 确切根因链（**确认**——与设计哲学文档 §27-50 的真因一致，并经上述读码坐实）

> **近邻订阅是 spawn 时一次性发出，没有任何活性维护；服务端 lease 有 TTL，玩家站着不动也会在 TTL 后过期；过期后没有任何机制把"订阅活性"接回来。**

精确机理分两段，要分清"目前为何坏"和"未来会怎样坏":

**(A) 当前 Voxia 的实际故障路径(spawn 后收 0 个 0x62/0x63)——确认:**

这其实**不是 lease 过期**(spawn 后几秒内 lease 远未到期),而是 Voxia 客户端侧的**订阅一次性闩 + 不重订**:

- Voxia `DriveBootstrap` 用一次性 bool `bSubscribed`,首订阅一次后永不重订(`VoxiaPawn.cpp:399`);这些 bool **从不被重置**。
- 若首次 `SubscribeAround` 发出时 Phase/连接时序未就绪、或那一帧 subscribe 帧丢失/被服务端因任何原因拒绝(`subscribe_one` 失败只回一帧 0x68),`bSubscribed` 仍被置 true,**之后再不重订**。
- 收 `0x63` delta 但本地无该 chunk(`VoxelStore.cpp:61`)或 base 版本不匹配(`:68`)时,Voxia 只打日志含 `re-subscribe` 文本,**代码不真重订**。
- 收 `0x69` 只删 store chunk、不重订(`VoxiaTransportSubsystem.cpp:555`)。

→ **客户端订阅系统完全没有"活性"维度,只有不完整的"一次性覆盖"。** 这是当前症状的直接原因(philosophy doc §44 的"连'订阅=位置驱动的覆盖'心智模型本身都不完整")。

**(B) 服务端侧的时间性静默失效(确认会发生,只是 Voxia 当前的 10s band-aid 暂时盖住了它)——确认:**

即便客户端订阅成功,只要玩家**长期站着不动**:

- ChunkProcess 的 `state.subscribers` 里有该连接 → 编辑 fan-out 正常(事实1/2:fan-out 不校验 lease),**所以"编辑→delta 回推"本身在 ChunkProcess 活着时是稳的**。
- 但 region lease TTL(默认 2h)到期后,若该连接长期不 route(站着不动、RouteCache 不 miss),World 永不被触发 `renew_region_lease` → lease 过期超 30min grace → `sweep_expired_regions` 把 region 回收(事实7),**且不发 0x69、不通知 Scene**。
- 此后该 region 再被 route 会 `:region_without_lease`,新编辑/写令牌签发失败;而**已存在的 Scene ChunkProcess 订阅者既收不到 0x69 也收不到错误**——订阅静默死,直到客户端因别的原因(移动跨界)重订。

### 1.2 次级/条件根因（**确认存在路径，但非当前主症状**）

| 编号 | 路径 | 是否确认 | 影响 |
|---|---|---|---|
| R-1 | **ChunkProcess crash→restart** 后新进程 `subscribers: %{}`,旧订阅者无 0x69、Gate 无 :DOWN(事实3/4/5) | 确认 | 崩溃即静默死订阅;靠 AOI 移动兜底 |
| R-2 | **region GC 回收活 region 不发 0x69**(事实7) | 确认 | 长期不 route 的活订阅静默死 |
| R-3 | **migration cutover 漏发/吞错 0x69**:`safe_invoke_scene_invalidator` 吞单 chunk 失败、或 `scene_invalidator=nil`(部分部署/测试) | 确认 (代码存在),生产是否触发未核 | cutover 后部分 chunk 静默死订阅 |
| R-4 | **订阅/编辑路由解耦**:订阅走 worker 30s 缓存路由,编辑走每次新鲜路由;迁移后 worker 缓存未失效 → 编辑落 scene_node A、订阅注册在 B,A 的 fan-out 名单无该连接,且无 0x69 | 推测 (需迁移频繁才成立),dev 单 World 不暴露 | 编辑 accepted 但订阅者收不到 delta |
| R-5 | **reason 0x02/0x03 从不发出**:codec 定义了 `region_removed`/`catalog_changed`,无任何生产路径发 | 确认 | region 显式移除/catalog 变更时无 0x69 |

### 1.3 **被推翻的推测**(避免顺着第一直觉,呼应自查清单 §4)

- ❌ "lease 续约抬 owner_epoch 让旧订阅静默失效":**伪**。fan-out 不校验 epoch(事实1)。续约对"已注册订阅收 delta"完全透明。
- ❌ "lease 过期 → ChunkProcess 立即丢订阅者":**伪**。`lease_stale?` 只跳 sim tick(事实2),ChunkProcess 不会因 lease 过期主动丢订阅者。真正丢订阅者的是 **region GC 回收整个 region**(R-2,比"lease 过期"晚 30min grace)以及 **进程死亡**(R-1)。
- ❌ "移动导致挖放失效":**红鲱鱼**(philosophy doc §38)。站着不动 18 分钟也坏。移动与故障无因果。

> **一句话根因**:订阅系统漏了"活性(liveness)"这一正交维度——客户端只做了不完整的"一次性覆盖",服务端把订阅活性隐式挂在"连接会不断主动 route"这个没人维护的假设上。edit 的可见性默默依赖"订阅活着",而没人维护"订阅活着"。这是隐藏耦合(自查清单 §3),也是 §25 描述的"最糟的第三种"。

## 2. 正交设计原则

呼应 `docs/2026-06-27-架构设计指导思想-系统正交.md` §一:

### 2.1 订阅系统的两个正交职责（必须分开建模）

1. **覆盖(coverage）**:位置驱动——玩家移动→订阅哪些区块进/出 AOI。(已有:`do_reconcile` 差集)
2. **活性(liveness)**:时间驱动——订阅一旦建立,**在连接在场期间必须始终活着**,绝不因 lease TTL / region GC / 进程重启而静默失效。(**缺失的一维**)

### 2.2 活性该由谁拥有?——**连接驱动活性(服务端权威侧维护)**

设计哲学文档 §55 把这个明确列为"仍未决,需团队拍板":客户端轮询(现状 band-aid)vs 服务端按连接活性维护。**本稿拍板:活性归服务端,以"连接 liveness"为权威信号。** 理由:

- **正交**:连接活则订阅活,无需客户端轮询,也无 10s 全半径重订的重网格风暴(Voxia 每 10s → 5x5x5=125 chunk 全快照 → 反复全量 RebuildMesh)。
- **承诺方维护不变量**(§14 自维护不变量):订阅系统自己保证"在场期间订阅始终活着",而不是把活性外包给客户端轮询或"玩家恰好会移动"。
- **edit 只依赖稳定契约**:"订阅区块的变更必回推",edit 一行不动、不感知订阅机制(§49)。

### 2.3 双保险(§21-24 两条路都要,互为兜底)

- **承诺方强制维护(主)**:服务端保证活订阅不静默失效——要么续住 lease/不回收有连接的 region,要么**所有订阅失效路径必发 0x69**(显式契约,不留隐式假设)。
- **依赖方对破坏鲁棒(兜底自愈)**:客户端收 0x69 即**精准重订阅**(单 chunk,radius-0,known 空→强制全量),去掉/大幅降频轮询。

> 关键不变量(服务端对外承诺):**只要一个连接仍持有某 chunk 的活订阅,该 chunk 的任何真值变更要么被回推(0x62/0x63),要么该订阅被显式 0x69 失效。绝不存在"订阅还在、变更不回推、也无 0x69"的第三态。**

## 3. 正交修复方案

分**服务端(承诺方维护)**与**客户端(依赖方自愈)**两侧。

### 3.A 服务端:消除"订阅静默失效"

核心思路:**让 Scene 侧成为"订阅活性"的权威维护者**,并补齐所有失效路径的 0x69 契约。分三个机制(S1 是主,S2/S3 补漏)。

#### S1 —— ChunkProcess 死亡 / region 失活 → **必发 0x69**(补齐显式契约,堵 R-1/R-2/R-3/R-5)

这是最正交、最小侵入的修复:**不改 lease 语义,只保证"所有订阅失效路径必发 0x69"**,然后让客户端自愈(3.B)。

具体:

1. **ChunkProcess crash → 让订阅者收到失效信号(堵 R-1)。**
   - **方案**:在 `ChunkDirectory` 引入一张 `chunk_subscribers` 表(`{logical_scene_id, chunk_coord} → MapSet<subscriber_pid>`),在 `{:subscribe}`/`{:unsubscribe}` 转发时同步维护。当 `{:DOWN, ref, pid, reason}` 且 `reason not in [:normal,:shutdown]`(crash)时,对该 chunk 表里每个 subscriber `send {:voxel_chunk_invalidate_payload, encode_chunk_invalidate(reason: 0x04 :process_restart)}`。
   - 文件:`apps/scene_server/lib/scene_server/voxel/chunk_directory.ex`(扩 state 加 `chunk_subscribers`,改 `{:subscribe}`/`{:unsubscribe}`/`{:DOWN}` handler)、`apps/scene_server/lib/scene_server/voxel/codec.ex`(加 reason `0x04 :process_restart`)。
   - **注意正交边界**:这正是 `docs/2026-06-25-voxel-world-production-architecture.md` §525 记为"遗留:崩溃无主动 0x69,靠 directory 侧订阅者表"的那块——本稿把它落地。subscriber 是连接 pid,directory 侧表与 ChunkProcess `state.subscribers` 是**镜像**(directory 转发即写),不引入新所有者,符合阶段4"单一所有者"教训(directory 的表只用于 crash 广播,不参与正常 fan-out)。

2. **region GC 回收 → 发 0x69(堵 R-2/R-5)。**
   - `sweep_expired_regions` 回收 region 时,对该 region bounds 内每个 chunk 调 `scene_invalidator.(attrs)`,reason=`0x02 :region_removed`。这复用 cutover 已有的 `scene_invalidator` 闭包(`authority_observe.ex` 已接到 `ChunkDirectory.invalidate_chunk`),只是新增一个调用点。
   - 文件:`apps/world_server/lib/world_server/voxel/map_ledger.ex`(`sweep_expired_regions` 回收前 emit invalidations)。
   - **更稳的做法(可选,优先)**:GC 不该回收"Scene 侧仍有活订阅"的 region。但 World 不知道 Scene 订阅数——所以更正交的是 **S2 的连接驱动续租**让活 region 永不进入 reapable;GC 发 0x69 作为兜底。

3. **cutover 吞错/未配置 → 显式化(堵 R-3)。**
   - `safe_invoke_scene_invalidator` 单 chunk 失败时,除 `error_count` 外应 emit 可观测事件(已 emit),并考虑对失败 chunk 重试一次;`scene_invalidator=nil` 在生产 boot 必须 assert 非 nil(加 boot 期校验,测试环境显式 opt-out)。
   - 文件:`apps/world_server/lib/world_server/voxel/map_ledger.ex`、`authority_observe.ex`(boot assert)。

#### S2 —— 连接驱动续租(让活订阅的 lease 永不过期,堵 R-2 的根)

S1 是"失效就通知";S2 是"根本别让它失效"。两者叠加=双保险。

- **机制**:Gate `SubscriptionWorker` 是 per-connection、已 monitor 连接、且是订阅集唯一所有者——它天然知道"这个连接当前订阅了哪些 region"。给它加一个**低频续租 tick**(如 30s,对齐 RouteCache refresh_window):对当前订阅集涉及的每个 region,主动走一次 `Routing.route_chunk`(绕过缓存或用"强制刷新"标志),**触发 World `renew_region_lease`**。这把"续租"从"恰好玩家移动会 route"变成"连接活就周期续",活性绑连接活性。
  - 关键:这**不是**客户端轮询,是 **Gate 进程(服务端侧)** 的续租,不发任何网络帧给客户端、不触发重快照、不重网格。它只刷新 World lease + Gate RouteCache。
  - 文件:`apps/gate_server/lib/gate_server/voxel/subscription_worker.ex`(加 `:renew_tick` 定时器,遍历订阅集去重 region,调 `Routing.route_chunk`/或新增 `Routing.renew_region`)。`apps/gate_server/lib/gate_server/voxel/route_cache.ex`(可加 `touch`/`force_refresh`)。
- **效果**:玩家站着不动 18 分钟,Gate 仍每 30s 续租 → lease 永不过期 → region 永不进 GC reapable → R-2 根除。连接死(tcp_closed)→ worker stop → 续租停 → lease 自然过期 → region 被 GC 回收(正确行为)。

#### S3 —— 不动的部分(明确不改,避免踩回老坑)

- **不改 fan-out 加 epoch 校验**:fan-out 无 epoch 校验是**对的**(事实1),它让续约对 delta 透明。加 epoch 校验反而会引入"续约即静默"的新坑。
- **不改 `lease_stale?` 让它丢订阅者**:stale lease 只暂停 sim 是对的;改成丢订阅者会让"短暂续租窗口未覆盖"误杀活订阅。
- **不改 idle eviction**:已被 0-subscriber guard 保护,且默认 disabled。

> **S1+S2 的正交关系**:S2 维护"活性不变量"(承诺方主路径,活订阅永不失效);S1 保证"万一仍失效必发 0x69"(承诺方契约兜底)。二者都不让 edit 感知订阅机制。

### 3.B 客户端(Voxia):收 0x69 即精准重订阅 + 去 band-aid

下述 Bevy 范式（`authority_plugin.rs`：Dropped→resync_in_flight 去重→`SubscribeChunks{radius:0, known:[]}`）仅作归档历史参考，不是现役依赖；Voxia 实现以当前服务端订阅契约和 `clients/Voxia` 当前代码为准，默认不依赖读取归档客户端：

1. **收 0x69 → 精准重订阅该 chunk(堵客户端侧不自愈)。**
   - `VoxiaTransportSubsystem.cpp:555` 的 `case 0x69`:`VoxelStore.Invalidate` 后,**追加** `SubscribeChunk(Cx,Cy,Cz, radius=0, want_snapshot=true, Known=空)`(单 chunk 定向重订,强制全量快照)。
   - 加去重闸（行为可参考归档 Bevy 的 `resync_in_flight` 历史范式，但不得把归档实现当当前真值）：同一 chunk 重订在途时不重复发，收到新 0x62 快照即清闸。避免 0x69 风暴→重订风暴。
   - 文件:`clients/Voxia/Source/Voxia/Net/VoxiaTransportSubsystem.cpp`(0x69 handler 加重订)、`VoxiaTransportSubsystem.h`(加在途重订集合 `TSet<FIntVector> ResyncInFlight`)。

2. **收 0x63 delta-miss(无 chunk / base 版本不匹配)→ 同样触发 radius-0 重订。**
   - `VoxelStore.cpp:61/68` 返回 false + reason 含 `re-subscribe` 时,调用方(`Transport.cpp:534`)**真的去重订**(走同一去重闸),而不是只打日志。堵"版本 gap 永久 drift"。

3. **大幅降频/去掉 band-aid 轮询。**
   - `MaybeRefreshSubscription`(`VoxiaPawn.cpp:509`)保留**跨 chunk 重订(覆盖,这是对的)**;把 10s 无条件 keep-alive 重订**去掉**(活性已由服务端 S2 维护 + 0x69 自愈接管),或降到很低频(如 60s)作纯保险并且**只在确实跨 region 时**才发,避免全半径重快照。
   - 文件:`clients/Voxia/Source/Voxia/Gameplay/VoxiaPawn.cpp`。

4. **修一次性闩 + 重连恢复。**
   - `bSubscribed`/`bConnectRequested`/`bEnterSceneRequested` 在 `Disconnect('reconnect')` 时必须重置,否则重连后 bootstrap 不重订(death_path 已确认)。
   - 文件:`clients/Voxia/Source/Voxia/Gameplay/VoxiaPawn.cpp`、`VoxiaPawn.h`、`VoxiaTransportSubsystem.cpp`(Disconnect 路径)。

### 3.C 双保险闭环

```
                ┌─────────────────── 承诺方维护(主) ───────────────────┐
连接活 ──monitor──► Gate SubscriptionWorker ──30s renew──► World lease 永不过期
                                                              │
                              edit ──► ChunkProcess ──fan-out(无 epoch 校验)──► 0x63 回推
                                                              │
                ┌─────────────────── 承诺方契约(兜底) ───────────────────┐
crash/GC/cutover ──► 必发 0x69 ──► Gate forward ──► 客户端 ──radius-0 重订(自愈,去重)──► 0x62
```

edit 永远只依赖"订阅区块的变更必回推"。它不知道 lease、不知道续租、不知道 0x69——完全正交。

## 5. 风险 / 未决（gaps，及与既有"dev voxel region 脆弱性"修复的关系）

1. **directory 侧订阅者表 vs 阶段4"单一所有者"教训(最需小心)**。阶段4 血泪是"订阅集分散在双所有者致乱序静默死订阅"(`docs/2026-06-25-...` §615)。S1.1 的 directory `chunk_subscribers` 表**必须只用于 crash 广播,绝不参与正常 fan-out / 不作权威**——它是 ChunkProcess `state.subscribers` 的镜像(转发即写、:DOWN 即清)。若它被当成第二个权威来源,会重蹈乱序覆盖坑。**设计上严格限定其用途为"crash 时找谁广播"。**

2. **S2 续租 tick 的频率 vs lease TTL/refresh_window 不变量**。既有坑:续约窗口曾 15min > 60s 测试租约致误续约(§490)。S2 加的 30s renew tick 必须保持"续租间隔 < 现实最短 lease TTL"且对齐 RouteCache refresh_window(30s),否则要么漏续(过期)要么 World 续约风暴。**测试注入短 TTL 时,renew tick 也要相应调短,保持不变量。**

3. **GC 发 0x69 与"GC 删行后 chunk 再被 route 失败"的时序**。GC 删 region 行后,客户端收 0x69 重订会立刻 route 该 region → 触发懒物化重建(route-miss materialize)。需确认:重订 race GC 删行,materialize 能正确重建 region(应能,§懒物化设计),否则重订会拿 `:region_without_lease` 一帧 0x68。**Step 5 实跑须覆盖此 race。**

4. **R-4(订阅/编辑路由解耦)在单 World dev 不暴露,多 World/迁移频繁才成立**。本稿 S1/S2 不直接修 R-4,但 S1.1(crash 0x69)+ cutover 0x69 + 客户端重订能**兜底**它(迁移 cutover 必发 0x69 → 客户端重订到新 scene_node)。前提是 cutover 0x69 不漏发(S1.3)。**R-4 的彻底修(订阅/编辑共用同一路由解析或迁移强制 rebind)留 backlog,不在本轮。**

5. **未落地的更深脆弱性(与既有 gaps 一致,本轮不碰)**:MapLedger 仍单 World 本地具名 GenServer(W4,Horde 分片未落地);scene_node `:nodedown` failover(W5)未落地——死节点 region 被 route 到则订阅/编辑静默超时。本稿假设单节点 dev;多节点的活性需 W4/W5 落地后再评估。**别在本轮顺手动这些,会扩大爆炸半径。**

6. **WorldGen opt-in / 出生点被埋**(设计性,非订阅 bug)。Step 5 实跑必须开 `:worldgen`,否则"地形空"会被误判为订阅失效(表象相同,根因不同)。

7. **Voxia headless Automation 构造网络帧的可行性未实测**(voxia-ue58-client 记忆:headless 回路靠 Build.bat + UnrealEditor-Cmd Automation)。若构造伪 0x69 帧过难,Step 3 退化为对解码后 handler 函数的直接单测——仍确定性,但端到端覆盖弱一档,Step 5 实跑补足。

8. **保留 band-aid 的低频兜底要不要彻底删?** 建议 Step 3 先**降频(60s)而非全删**,Step 5 实跑确认 S1+S2 闭环稳定后,再单独 commit 删除 10s 轮询——避免一次性去掉所有兜底导致 dev 体验回退。
