# 线协议规范

## 概述

当前客户端与服务端之间的通信使用 **TCP + 自定义二进制协议**。运行时主路径已经不再使用 protobuf。

本规范描述的是 **当前生效的主协议格式**，不再保留旧请求布局的兼容说明。

> **部署纪律（v1）**：本协议为 v1，**字段只追加、不重排**，且 client / server **必须同版本同时部署**——任一侧落后都会因长度 mismatch 直接断线。后续如需 wire-breaking 改动，请走"新 message type / 新 codec 版本号"路径，不要原地改字段。

## Pillar 1.1 wire 升级说明（2026-05-28）

支柱 1.1 在以下三条热点帧加入了版本守卫字段，同时新增 `EnterSceneResult` 结构。**三个平行发送端**均已同步升级：`apps/gate_server/lib/gate_server/worker/tcp_connection.ex`（TCP）、`apps/gate_server/lib/gate_server/worker/udp_acceptor.ex`（UDP，仅 MovementAck）、`apps/gate_server/lib/gate_server/worker/ws_connection.ex`（WebSocket）。

### 新增字段语义

| 字段 | 类型 | 含义 |
|------|------|------|
| `schema_version` | `u8` | **热点帧逐帧守卫**：每条 Movement / PlayerMove / MovementAck 帧的第 2 字节；服务端 codec 按当前 `@movement_wire_schema` 填写，客户端解码时逐帧核验，不匹配则 fail-fast 断线。区别于握手级 `protocol_version`，它是运行时每帧的快速守卫，防止滚动升级中间态产生静默错位 |
| `server_send_ms` | `u64` | **服务端 wall-clock 发送时刻**（毫秒 Unix 时间戳）：随 PlayerMove(0x83) 和 MovementAck(0x8b) 下行；供 pillar 1.3 客户端插值时间轴对齐——客户端可以用 TimeSync 算出的 `serverClockOffsetMs` 将其折算为本地时钟时间，驱动真实服务端时间线上的插值，消除"tick 计数 × 固定间隔"近似带来的时间轴跳变 |
| `protocol_version` | `u16` | **握手版本断言**：随 EnterSceneResult(0x84) 一次性下行；客户端收到后与编译期常量 `PROTOCOL_VERSION` 比较，不匹配则立即 `markDisconnected`，阻止后续所有帧解码（避免静默错位） |

## 分帧

每个 TCP 消息都带有一个 4 字节大端长度前缀，该前缀由 Erlang socket 的 `{packet, 4}` 选项自动处理。

```text
┌──────────────────┬─────────────────────────┐
│ length (4 bytes) │ body (variable length) │
│ big-endian u32   │ msg_type (1) + payload │
└──────────────────┴─────────────────────────┘
```

其中：

- `length` 表示 `body` 的字节数，不包含它自身
- 客户端发送时需要附带这个长度头
- 服务端使用 `{packet, 4}` 后会自动按该格式收发

## 消息体格式

```text
body = <<msg_type::8, payload::binary>>
```

- `msg_type`：1 字节消息类型 ID
- `payload`：各消息专属字段

## 数据类型定义

| 类型 | 大小 | 编码方式 |
|------|------|----------|
| `u8` | 1 字节 | 无符号大端 |
| `u16` | 2 字节 | 无符号大端 |
| `i64` | 8 字节 | 有符号大端 |
| `u64` | 8 字节 | 无符号大端 |
| `f64` | 8 字节 | IEEE 754 double，大端 |
| `string` | 2 + N 字节 | `<<length::16-big, data::binary-size(length)>>` |
| `vec3` | 24 字节 | `<<x::f64, y::f64, z::f64>>` |

所有向量统一使用 **double precision (`f64`)**，以保持与内部物理引擎（Rust NIF）一致。

## 客户端 → 服务端消息（0x01–0x7F）

除 `0x60..0x7F` 体素保留段外，当前主线仍按低位 opcode 上行、高位 opcode 下行组织。
体素段采用同一段内双向消息；字段格式以
`docs/2026-04-29-server-authoritative-voxel-data-protocol-design.md` 为准。Phase 1.4
（2026-05-13）起体素段从 `0x60..0x6F` 扩展到 `0x60..0x7F`，加入
`0x70 VoxelEditIntent` 与 `0x71 CatalogPatch`。

### 0x01 —— Movement

玩家移动时高频发送。**Pillar 1.1 升级**：第 2 字节插入 `schema_version` 守卫，总长度由 25 字节升至 26 字节。

```elixir
<<0x01, schema_version::u8, seq::u32, client_tick::u32, dt_ms::u16,
  input_dir_x::f32, input_dir_y::f32, speed_scale::f32, movement_flags::u16>>
```

| 字段 | 类型 | 偏移 | 含义 |
|------|------|------|------|
| `opcode` | `u8` | 0 | `0x01` |
| `schema_version` | `u8` | 1 | 热点帧逐帧守卫，见 Pillar 1.1 说明 |
| `seq` | `u32` | 2 | 客户端输入序号；服务端 ack 的主对账键 |
| `client_tick` | `u32` | 6 | 客户端 fixed-tick 序号；服务端会重编号为权威 tick |
| `dt_ms` | `u16` | 10 | 客户端采样间隔；服务端会钳制为权威 fixed tick |
| `input_dir_x` | `f32` | 12 | 水平输入方向 X |
| `input_dir_y` | `f32` | 16 | 水平输入方向 Y |
| `speed_scale` | `f32` | 20 | 客户端请求速度缩放；服务端按 profile 钳制 |
| `movement_flags` | `u16` | 24 | 输入意图 bitfield |

`movement_flags` 当前位义：

| 位 | 名称 | 含义 |
|----|------|------|
| `0x0001` | `Run` | 预留跑步修饰 |
| `0x0002` | `Brake` | 制动 / 零输入减速 |
| `0x0004` | `Jump` | 一次性 grounded 起跳请求；客户端只在按键边沿发送一帧 |

**总长度**：26 字节（1 + 1 + 4 + 4 + 2 + 4 + 4 + 4 + 2）

### 0x04 —— Heartbeat

心跳 / 保活消息。

```elixir
<<0x04, timestamp::u64>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `timestamp` | `u64` | 客户端时间戳（毫秒） |

**总长度**：9 字节

### 0x05 —— AuthRequest

使用用户名和 token 做认证。

```elixir
<<0x05, request_id::u64, username::string, code::string>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `request_id` | `u64` | 请求关联 ID |
| `username` | `string` | 用户名（UTF-8） |
| `code` | `string` | token / 验证码 |

**总长度**：可变

当前运行时语义补充：

- 服务端会将 `username` 与 token claims 中的用户名做一致性校验
- token claims 当前至少可携带：`username`、`source`、`session_id`
- token claims 还可以携带：`cid` 或 `allowed_cids`，用于约束后续 `EnterScene`

### 0x06 —— FastLaneRequest

```elixir
<<0x06, request_id::u64>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `request_id` | `u64` | 请求关联 ID |

**总长度**：9 字节

### 0x07 —— FastLaneAttach

客户端通过 UDP 发送 attach 请求，把自己附着到已经认证的 TCP 会话。

```elixir
<<0x07, request_id::u64, ticket::string>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `request_id` | `u64` | 请求关联 ID |
| `ticket` | `string` | 由 TCP `FastLaneRequest` 返回的短期 ticket |

**总长度**：可变

### 0x08 —— ChatSay

可靠聊天消息。当前 v1 语义是 **server-authoritative world channel**：
由当前已进场连接发起，经 Gate 转给 `ChatServer.Runtime`，再由 Chat runtime
按服务端会话元数据投递并回推 `ChatMessage(0x89)`。Scene AOI 不再拥有 Gate
聊天发送路径。需要 `region` / `local` 频道时使用 `ChatSayScoped(0x0A)`；客户端
仍不能自报 `region_id`、`chunk_coord` 或位置作为聊天真相。

```elixir
<<0x08, request_id::u64, text::string>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `request_id` | `u64` | 请求关联 ID |
| `text` | `string` | 聊天正文 |

**总长度**：可变

### 0x0A —— ChatSayScoped

带作用域的可靠聊天消息。客户端只声明想使用的频道作用域；Gate 必须从当前连接的
`partition_context` / `chat_context` 派生实际 `world` / `region` / `local` channel。
该帧不携带 `region_id`、`chunk_coord`、半径或位置，避免客户端伪造分区权威。

```elixir
<<0x0A, request_id::u64, scope::u8, text::string>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `request_id` | `u64` | 请求关联 ID |
| `scope` | `u8` | `0=world`、`1=region`、`2=local` |
| `text` | `string` | 聊天正文 |

**总长度**：可变

### 0x09 —— SkillCast

简单技能施放请求。当前实现只支持最小可联调技能切片：
`skill_id = 1` 表示一个不带伤害结算的 pulse 技能广播。

```elixir
<<0x09, request_id::u64, skill_id::u16>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `request_id` | `u64` | 请求关联 ID |
| `skill_id` | `u16` | 技能 ID |

**总长度**：11 字节

### 0x60..0x7F —— Voxel reserved segment

体素世界使用 `WorldServer` 作为全局地图账本、租约、地块和跨 scene 事务协调者；
`SceneServer.Voxel.*` 作为租约区域内的 hot authority。`WorldServer` 不参与
chunk/object 状态热路径 tick，Scene 的写入必须来自当前 World lease / owner epoch。

体素段继续复用本协议的 `{packet, 4}` frame 与 `msg_type::8` body 头；payload 内所有多字节字段
统一使用 **big-endian / network byte order**，不得再使用另一套 payload 字节序。

同步原则：玩家相关表现可以本地预测或预演；体素数据必须服务器先计算，客户端只从
`ChunkSnapshot / ChunkDelta / ObjectStateDelta / VoxelIntentResult` 更新 confirmed voxel store。

| opcode | 方向 | 名称 | 当前用途 |
| --- | --- | --- | --- |
| `0x60` | C→S | `ChunkSubscribe` | 订阅 scene 内 chunk AOI |
| `0x61` | C→S | `ChunkUnsubscribe` | 取消订阅 |
| `0x62` | S→C | `ChunkSnapshot` | 下发 canonical chunk truth |
| `0x63` | S→C | `ChunkDelta` | 下发 chunk 增量 |
| `0x64` | C→S | `VoxelImpactIntent` | 可选：技能/工具请求地形影响，通常由技能系统内部触发 |
| `0x65` | C→S | `BuildReservationIntent` | 请求地块/区域内施工排他保留 |
| `0x66` | C→S | `BlueprintCreate` | 预留：创建 prefab/blueprint 定义 |
| `0x67` | C→S | `PrefabPlaceIntent` | 提交 blueprint_id + anchor intent，服务端重算 raster |
| `0x68` | S→C | `VoxelIntentResult` | 返回 accepted/deferred/rejected/stale |
| `0x69` | S→C | `ChunkInvalidate` | 服务端要求客户端丢弃/重订阅 |
| `0x6A` | C→S | `ParcelQuery` | 查询地块/权限/建设状态 |
| `0x6B` | C→S | `ObjectAction` | 预留：object/part 级交互 |
| `0x6C` | S→C | `ObjectStateDelta` | 预留：object 状态更新 |
| `0x6D` | S→C | `TagCatalogSnapshot` | Phase 5.B：全局 tag catalog 全量快照（见下文 `0x6D` 字段定义） |
| `0x6E` | S→C | `AttributeCatalogSnapshot` | Phase 5.A：全局 attribute catalog 全量快照（见下文 `0x6E` 字段定义） |
| `0x6F` | both | `VoxelDebugProbe` | 本地/dev 调试保留 |
| `0x70` | C→S | `VoxelEditIntent` | typed 客户端编辑通道，见 §13.6.1 |
| `0x71` | S→C | `CatalogPatch` | Phase 1.4 envelope only；attribute/tag catalog 增量 patch（payload 内容 Phase 5 解释） |
| `0x72` | S→C | `EnvironmentUpdated` | Phase 5.F：温湿度 simulator 输出的 macro-level 环境增量（见下文 `0x72` 字段定义） |
| `0x73` | S→C | `FieldRegionSnapshot` | Phase 6：本地场（temperature / electric_potential / ionization）按 region 的全量稀疏快照（见下文 `0x73` 字段定义） |
| `0x74` | S→C | `FieldRegionDestroyed` | Phase 6：本地场 region 终结通知（expired / lease_revoked / explicit / chunk_crash） |
| `0x75` | C→S | `FieldConductIntent` | Phase 7：请求服务端权威导电/放电求解 |
| `0x76` | C→S | `VoxelChunkAck` | 客户端确认已应用并保留 chunk 版本，供后续重订阅复用 |

体素 v1 canonical 参数：

| 参数 | 值 |
| --- | --- |
| `chunk_size_in_macro` | `16` |
| `micro_resolution` | `8` |
| refined occupancy | `512 bits = u64[8]` |
| normal block fixed header | `20 bytes` |

体素消息必须携带足够版本信息：`schema_version`、`chunk_version`、`cell_version`、`cell_hash`。
客户端可以做 build preview 和玩家侧技能表现预演；体素 confirmed truth 只能来自服务器下行。

#### 0x76 `VoxelChunkAck` payload

`VoxelChunkAck` 是可靠连接上的客户端接收确认。Gate 只接受不超过本连接已成功
下发版本的 ACK；超前、未知或倒退 ACK 会被拒绝并写入 observe 日志。被接受的 ACK
表示客户端已应用并保留该 chunk 版本，直到服务器发送 `ChunkInvalidate` 或连接结束。
后续 `ChunkSubscribe` 只会把这些经服务端验证的 ACK 作为 `known_version` 传给 Scene；
Gate 的 forwarded cache 不能再单独代表客户端已收到。若 Gate 的 live delivery queue
已经把某 chunk 标记为需要 resync，普通退订不会解除该保护；只有替换 snapshot 或已送达
的 `ChunkInvalidate` 才能恢复该 chunk 的 ACK 复用资格。

```text
VoxelChunkAck
  request_id:       u64
  logical_scene_id: u64
  ack_count:        u16
  acks[ack_count] {
    chunk_coord:    i32 x 3
    chunk_version:  u64
  }
```

#### 0x6E `AttributeCatalogSnapshot` payload（Phase 5.A）

全局 attribute catalog 全量快照，作为客户端冷启动 / 重连 / catalog 大幅变更时
的"基线"通道。增量更新走 `0x71 CatalogPatch`（schema_kind=0x01 attribute）。
本消息 **payload 只**，不含 opcode byte（在 `{packet,4}` frame 内紧跟 `msg_type::8`）。

Wire layout（一旦发出即冻结）：

```text
AttributeCatalogSnapshot
  catalog_version: u64         # monotonic；增量 patch 累加同一计数
  definition_count: u32
  definitions[definition_count] {
    id:            u32         # 全局 attribute_id；Phase 1.2 chunk-local `AttributeEntry.key_id`
                                # 在 Phase 5.A 之后语义升级为本 id（wire 字段不变）
    name_len:      u16
    name:          bytes(name_len)   # UTF-8，非空
    unit_len:      u16
    unit:          bytes(unit_len)   # UTF-8，允许为空（unitless attribute，如 boolean / enum）
    value_type:    u8                # 0x01 i16 / 0x02 u16 / 0x03 fixed32 (Q16.16) /
                                     # 0x04 enum8 / 0x05 bitset32（与 §12 Phase 1.2 一致）
    default_value: bytes(N)          # N = value_type 字节长度：i16/u16=2, fixed32/bitset32=4, enum8=1
    min_value:     bytes(N)
    max_value:     bytes(N)
    merge_rule:    u8                # 0x01 override / 0x02 add_delta / 0x03 max /
                                     # 0x04 min / 0x05 material_default
    dynamic:       u8                # 0 / 1（运行时可变 hint）
  }
```

字节量估算：单 `AttributeDefinition` 假设 `name="temperature"`(11B) + `unit="°C"`(3B UTF-8):
`4 + 2+11 + 2+3 + 1 + 2+2+2 + 1 + 1 = 31 B`；5 个内置 attribute 约 150~200B。

校验规则：
- `name` 必须非空合法 UTF-8；`unit` 允许为空（unitless）。
- `value_type` ∈ {0x01..0x05}（与 Phase 1.2 `AttributeEntry.value_type` 完全一致）。
- `default_value` / `min_value` / `max_value` 按 `value_type` 字节长度编码；必须满足
  `min_value <= default_value <= max_value`。
- `merge_rule` ∈ {0x01..0x05}；未知值在 normalize / decode 阶段都 raise（不做 forward-compat skip）。
- `dynamic` ∈ {0, 1}。
- `definitions` 内按 `id` 升序、不重复（producer canonicalize，consumer 在 wire boundary 再校验一次）。

Phase 5.A 仅落地服务端 wire 类型 `SceneServer.Voxel.AttributeCatalogSnapshot` /
`SceneServer.Voxel.AttributeDefinition`。Gate codec 集成、客户端 TS decoder、catalog
持久化（DataService schema）推到 Phase 5.C / 5.D。

#### 0x6D `TagCatalogSnapshot` payload（Phase 5.B）

全局 tag catalog 全量快照，作为客户端冷启动 / 重连 / catalog 大幅变更时
的"基线"通道。增量更新走 `0x71 CatalogPatch`（schema_kind=0x02 tag）。
本消息 **payload 只**，不含 opcode byte（在 `{packet,4}` frame 内紧跟 `msg_type::8`）。

与 `0x6E AttributeCatalogSnapshot` 对称但更简单：`TagDefinition` 只携带
`id + name`，无 `value_type / default / min / max / merge_rule / dynamic`
（Phase 1.3 T-2 决策"不携带 value"——要 value 走 `AttributeSet` / `AttributeCatalog`）。

Wire layout（一旦发出即冻结）：

```text
TagCatalogSnapshot
  catalog_version: u64         # monotonic；增量 patch 累加同一计数
  definition_count: u32
  definitions[definition_count] {
    id:        u32             # 全局 tag_id；Phase 1.3 chunk-local `TagSet.tag_ids`
                                # 中的每个 u32 元素在 Phase 5.B 之后语义升级为本 id
                                # （wire 字段不变）
    name_len:  u16
    name:      bytes(name_len) # UTF-8，非空
  }
```

字节量估算：每条 `TagDefinition` wire 字节数 = `4 + 2 + name_byte_len`，
例如 `name="flammable"`(9B) → 15 B/definition；10 个内置 tag 约 150B。

校验规则：
- `name` 必须非空合法 UTF-8。
- `id` 必须在 u32 范围（0..0xFFFF_FFFF）。
- `definitions` 内按 `id` 升序、不重复（producer canonicalize，consumer 在 wire boundary 再校验一次）。

Phase 5.B 仅落地服务端 wire 类型 `SceneServer.Voxel.TagCatalogSnapshot` /
`SceneServer.Voxel.TagDefinition`。Gate codec 集成、客户端 TS decoder、第一批
typed tag 注入（flammable / conductive / etc.）由 Phase 5.C 接续。

#### 0x72 `EnvironmentUpdated` payload（Phase 5.F）

低频规则帧 simulator 计算后的 macro-level 环境增量（温度 / 湿度）。每个
ChunkProcess 在每个 simulation tick 完成且 simulator 返回非空 env_delta 时，按
本格式编码一份 payload 并 fanout 给本 chunk 的所有 subscribers。本消息 **payload
只**，不含 opcode byte（在 `{packet,4}` frame 内紧跟 `msg_type::8`）。

Wire layout（一旦发出即冻结）：

```text
EnvironmentUpdated
  logical_scene_id:     u64
  chunk_coord:          i32 cx, i32 cy, i32 cz
  base_chunk_version:   u64
  new_chunk_version:    u64
  update_count:         u16
  updates[update_count] {
    macro_index:  u16            # 0..4095
    field_mask:   u8             # 0x01 temperature / 0x02 moisture / 其它预留
    temperature:  i16            # 仅 field_mask 含 0x01 时存在
    moisture:     i16            # 仅 field_mask 含 0x02 时存在
    source_hash:  u32            # simulator 输入快照 hash，便于客户端去重 / 回放校验
  }
```

字段语义：

- `chunk_coord` —— 与 `ChunkSnapshot` / `ChunkDelta` 同坐标含义。
- `base_chunk_version` / `new_chunk_version` —— Phase 5.F 本 commit 服务端不
  改写 `environment_summaries` 的 canonical truth（仅推 wire），两者通常相等于
  当前 storage `chunk_version`；Phase 5.F.runtime 真正写回时会按既有 chunk_version
  单调递增的纪律 bump。
- `field_mask` —— bitmap，目前只接受 `0x01` (temperature) / `0x02` (moisture) /
  `0x03` (两者同发)。其它 bit 位预留：与 `0x71 CatalogPatch` envelope 同款
  "未知 schema_kind 硬错误"纪律 —— encoder / decoder 都拒绝。`field_mask = 0`
  亦视为非法。
- `temperature` / `moisture` —— i16 raw delta（按 §12 Phase 1.2 `AttributeEntry`
  fixed32 Q16.16 的 i16 子范围解释，表达相对 catalog default 的偏移），与
  `MacroEnvironmentSummary.current_temperature` / `current_moisture` 字段语义
  完全一致。仅在 `field_mask` 含相应 bit 时存在，**严格按 temperature → moisture
  顺序串联**。
- `source_hash` —— simulator 计算本 op 时输入快照（macro_index + 自身 + 6 邻居 +
  α/dt + attribute_name）的 64-bit hash 取低 32 位；同输入 → 同 hash，便于
  客户端做幂等 / 回放校验。

校验规则：

- `macro_index` 必须在 `0..4095`。
- `field_mask` 必须落在 `{0x01, 0x02, 0x03}` —— 否则 decoder 抛 `ArgumentError`。
- `temperature` / `moisture` 必须落在 i16 范围 `-0x8000..0x7FFF`。
- `updates` 内不强制按 `macro_index` 升序（simulator 自然按 dirty bounds 三维
  迭代序输出）；consumer 不依赖顺序。

Phase 5.F 仅落地服务端 codec + ChunkProcess fanout；客户端 TS decoder 推后续
（草案 F-7 推荐方案）。

#### 0x73 `FieldRegionSnapshot` payload（Phase 6）

Phase 6 局部场最小目标：把绑定到某个 chunk 的活跃场区域（temperature /
electric_potential / ionization）每 tick 的稀疏快照下推到该 chunk 的所有
subscribers。`FieldTickWorker` 是 per-region GenServer，独立调度 10 Hz tick，
通过 `ChunkProcess.push_field_snapshot_payload/2`（GenServer.cast）把
encode 完的 payload 投到 chunk fanout 通道。

temperature 层从 2026-05-14 起采用环境基线 + 整数异常 delta：服务端只保存并下发
偏离 20°C 环境基线的 macro cell，wire 中的 `temperature_values` 仍是绝对温度值
（f32 little-endian），用于保持现有客户端 decoder 兼容。

**注意：本消息 payload 在生产路径上 _包含_ opcode byte `0x73`**——与其它体素
消息不同。`SceneServer.Voxel.Field.FieldCodec.encode_snapshot_payload/2` 输出
的二进制以 `<<0x73, ...>>` 起始；gate 端 `tcp_connection.send_frame/2` 走
`:gen_tcp.send/2` 直接写 socket，由 `{packet, 4}` 在 gen_tcp 层补 4 字节长度
前缀，不再过 `GateServer.Codec.encode/1`。

Wire layout（一旦发出即冻结）：

```text
FieldRegionSnapshot
  opcode:               u8 = 0x73
  logical_scene_id:     u64 (big endian)
  chunk_coord:          i32 cx, i32 cy, i32 cz (big endian)
  region_id:            u64 (big endian)
  tick_count:           u32 (big endian)
  field_mask:           u8
    bit0  0x01 temperature
    bit1  0x02 electric_potential
    bit2  0x04 ionization
    其它 bit 预留 / 必须为 0
  cell_count:           u16 (big endian)
  macro_indices[cell_count]:         u16 big endian × cell_count
  temperature_values[cell_count]:    f32 little endian × cell_count
    （iff field_mask 含 0x01）
  electric_potential_values[cell_count]: f32 little endian × cell_count
    （iff field_mask 含 0x02）
  ionization_values[cell_count]:     u8 × cell_count
    （iff field_mask 含 0x04，值已 clamp 到 0..255）
```

字段语义：

- `chunk_coord` —— FieldRegion 必须完全落在该 chunk 内（AABB 各轴 0..15）。
- `region_id` —— 由 `ChunkProcess.create_field_region/2` 在
  `System.unique_integer([:positive, :monotonic])` 分配，u64 单调递增。
- `tick_count` —— FieldTickWorker 在本次发送之前已经 increment，是 cumulative
  tick 数（首次推送为 1）。
- `field_mask` —— bitmap，可同时携带多个字段；目前只接受
  `0x01 / 0x02 / 0x04 / 0x03 / 0x05 / 0x06 / 0x07` 的组合。**未知 bit 必须保留为 0**。
- `cell_count` —— 三个字段值数组的统一长度，对应 `macro_indices` 中"任一字段
  active"的 cell 集合（取并集；temperature active 表示相对 20°C 的整数 delta
  绝对值 ≥ 1；某个字段在该 cell 没有数据时仍发 `0.0` / `0`，
  以保持三个数组按 index 对齐）。
- `macro_indices` —— 严格升序，无重复，每个值 `0..4095`。
- `f32 little-endian` —— wire 兼容格式。`FieldLayer` 内部现在是稀疏 delta map；
  temperature 会在写入时整数化，但仍按 f32 发送绝对温度值。与 `ChunkSnapshot`
  macro/normal block 字段的 big-endian 整数路径不冲突，因为前者是浮点数组、后者是
  定长定点结构。

校验规则：

- `field_mask` 必须落在 `{0x01, 0x02, 0x03, 0x04, 0x05, 0x06, 0x07}`；其它 bit
  位（含 0x00）解码视为非法。
- `macro_indices` 必须严格升序、无重复，且每个值 `0..4095`。
- 三个值数组的长度必须严格等于 `cell_count`；若 `field_mask` 含对应 bit 但
  剩余字节不足则解码失败。

#### 0x74 `FieldRegionDestroyed` payload（Phase 6）

某个 FieldRegion 终结时（max_ticks 触发 / 调用方显式 destroy / lease 失效 /
绑定 chunk 进程崩溃），由 ChunkProcess fanout 给 subscribers。和 `0x73` 一样
**payload 包含 opcode byte**。

Wire layout：

```text
FieldRegionDestroyed
  opcode:               u8 = 0x74
  logical_scene_id:     u64 (big endian)
  chunk_coord:          i32 cx, i32 cy, i32 cz (big endian)
  region_id:            u64 (big endian)
  destroy_reason:       u8
    0x00 expired         (FieldRegion.tick_limit_reached?/1)
    0x01 lease_revoked   (绑定 chunk lease token 变化)
    0x02 explicit        (ChunkProcess.destroy_field_region/2)
    0x03 chunk_crash     (绑定 chunk 进程 DOWN)
    其它值 decoder 视为 :unknown
```

校验规则：

- `destroy_reason` 未知字节不抛错，decoder 落到 `:unknown`，供 server-side
  audit 用；wire 端是否拒绝由消费者自行决定。
- 同一 `region_id` 在一个 chunk 生命周期内 0x73 / 0x74 严格按"先 push 至少
  一次 snapshot，最后一次 destroyed"的顺序出现（FieldTickWorker 在 `:tick`
  handler 内先 push snapshot 再判 limit；lease_revoked 路径在 ChunkProcess
  `{:apply_lease, ...}` 内同步 stop worker 并 fanout destroyed）。

Phase 6 仅落地服务端 codec / FieldTickWorker / FieldTickSupervisor /
ChunkProcess 集成 + gate 端 tcp_connection 转发；客户端 TS decoder 推后续。

## 当前连接状态约束（2026-04-10）

| 连接状态 | 允许消息 |
|----------|----------|
| `waiting_auth` | `AuthRequest`、`Heartbeat` |
| `authenticated` | `EnterScene`、`TimeSync`、`Heartbeat` |
| `in_scene` | `Movement`、`TimeSync`、`Heartbeat`、`ChatSay`、`ChatSayScoped`、`SkillCast`、voxel `0x60..0x7F` 中允许的 C→S 消息 |

当前约束说明：

- 未认证时发送 `EnterScene` 会返回 `EnterSceneResult error`
- 未进入场景时发送 `Movement` 会返回通用 `Result error`
- `TimeSync` 在 `authenticated` 和 `in_scene` 状态都允许
- `auth_server` 不可用时，`AuthRequest` 会 fail closed，返回通用 `Result error`
- `scene_server` 或场景玩家进程不可用时，`EnterScene` 会 fail closed，返回 `EnterSceneResult error`
- 如果 token claims 中的用户名与 `AuthRequest.username` 不一致，认证会失败
- 如果 token claims 中声明了 `cid` 或 `allowed_cids`，`EnterScene(cid)` 必须满足该约束，否则失败
- `EnterScene(cid)` 还会经过 `auth -> data_service` 的真实角色归属校验；仅凭 token claim 不足以通过最终准入
- `ChatSay` / `ChatSayScoped` 与 `SkillCast` 只允许在 `in_scene` 状态下使用；否则返回通用 `Result error`
- `ChatSayScoped` 的 `region` / `local` 作用域必须由 Gate 当前服务器上下文派生频道；客户端 payload 中没有分区权威字段
- voxel C→S 消息只允许在 `in_scene` 状态下使用，并且必须由 gate 按当前连接的 `scene_ref`
  路由到对应 `SceneServer.Voxel.*` authority；客户端 payload 中的 `scene_id` 只能用于校验，不能用于绕过连接态路由
- UDP fast lane 现在具备显式生命周期语义：同一 TCP 会话只保留一个活跃 UDP peer，重附着会替换旧 peer，空闲 peer 会自动过期

## 运行时实现位置

- **服务端 codec**：`apps/gate_server/lib/gate_server/codec.ex`
- **voxel canonical codec（未来）**：`apps/scene_server/lib/scene_server/voxel/codec.ex`
- **三个平行发送端（Pillar 1.1 均已升级）**：
  - **TCP**：`apps/gate_server/lib/gate_server/worker/tcp_connection.ex`

- **字节序**：统一使用大端（network byte order）

- **客户端协议版本常量**：`clients/web_client/src/infrastructure/net/protocolVersion.ts`（`PROTOCOL_VERSION = 1`）
