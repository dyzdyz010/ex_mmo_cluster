# 线协议规范

## 概述

当前客户端与服务端之间的通信使用 **TCP + 自定义二进制协议**。运行时主路径已经不再使用 protobuf。

本规范描述的是 **当前生效的主协议格式**，不再保留旧请求布局的兼容说明。

> **部署纪律（v1）**：本协议为 v1，**字段只追加、不重排**，且 client / server **必须同版本同时部署**——任一侧落后都会因长度 mismatch 直接断线。后续如需 wire-breaking 改动，请走"新 message type / 新 codec 版本号"路径，不要原地改字段。

## 分帧

每个 TCP 消息都带有一个 4 字节大端长度前缀，该前缀由 Erlang socket 的 `{packet, 4}` 选项自动处理。

```text
┌──────────────────┬─────────────────────────┐
│ length (4 bytes) │ body (variable length) │
│ big-endian u32   │ msg_type (1) + payload │
└──────────────────┴─────────────────────────┘
```

其中：

- `length` 表示 `body` 的字节数，不包含它自身
- 客户端发送时需要附带这个长度头
- 服务端使用 `{packet, 4}` 后会自动按该格式收发

## 消息体格式

```text
body = <<msg_type::8, payload::binary>>
```

- `msg_type`：1 字节消息类型 ID
- `payload`：各消息专属字段

## 数据类型定义

| 类型 | 大小 | 编码方式 |
|------|------|----------|
| `u8` | 1 字节 | 无符号大端 |
| `u16` | 2 字节 | 无符号大端 |
| `i64` | 8 字节 | 有符号大端 |
| `u64` | 8 字节 | 无符号大端 |
| `f64` | 8 字节 | IEEE 754 double，大端 |
| `string` | 2 + N 字节 | `<<length::16-big, data::binary-size(length)>>` |
| `vec3` | 24 字节 | `<<x::f64, y::f64, z::f64>>` |

所有向量统一使用 **double precision (`f64`)**，以保持与内部物理引擎（Rust NIF）一致。

## 客户端 → 服务端消息（0x01–0x7F）

除 `0x60..0x7F` 体素保留段外，当前主线仍按低位 opcode 上行、高位 opcode 下行组织。
体素段采用同一段内双向消息；字段格式以
`docs/2026-04-29-server-authoritative-voxel-data-protocol-design.md` 为准。Phase 1.4
（2026-05-13）起体素段从 `0x60..0x6F` 扩展到 `0x60..0x7F`，加入
`0x70 VoxelEditIntent` 与 `0x71 CatalogPatch`。

### 0x01 —— Movement

玩家移动时高频发送。

```elixir
<<0x01, seq::u32, client_tick::u32, dt_ms::u16,
  input_dir_x::f32, input_dir_y::f32, speed_scale::f32, movement_flags::u16>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `seq` | `u32` | 客户端输入序号；服务端 ack 的主对账键 |
| `client_tick` | `u32` | 客户端 fixed-tick 序号；服务端会重编号为权威 tick |
| `dt_ms` | `u16` | 客户端采样间隔；服务端会钳制为权威 fixed tick |
| `input_dir_x` | `f32` | 水平输入方向 X |
| `input_dir_y` | `f32` | 水平输入方向 Y |
| `speed_scale` | `f32` | 客户端请求速度缩放；服务端按 profile 钳制 |
| `movement_flags` | `u16` | 输入意图 bitfield |

`movement_flags` 当前位义：

| 位 | 名称 | 含义 |
|----|------|------|
| `0x0001` | `Run` | 预留跑步修饰 |
| `0x0002` | `Brake` | 制动 / 零输入减速 |
| `0x0004` | `Jump` | 一次性 grounded 起跳请求；客户端只在按键边沿发送一帧 |

**总长度**：25 字节（1 + 4 + 4 + 2 + 4 + 4 + 4 + 2）

### 0x02 —— EnterScene

请求进入场景。

```elixir
<<0x02, request_id::u64, cid::i64>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `request_id` | `u64` | 请求关联 ID |
| `cid` | `i64` | 需要进入场景的角色 ID |

**总长度**：17 字节

### 0x03 —— TimeSync

请求做连接级时间同步。

```elixir
<<0x03, request_id::u64, client_send_ts::u64>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `request_id` | `u64` | 请求关联 ID |
| `client_send_ts` | `u64` | 客户端发送该请求时的本地时间戳 |

**总长度**：17 字节

### 0x04 —— Heartbeat

心跳 / 保活消息。

```elixir
<<0x04, timestamp::u64>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `timestamp` | `u64` | 客户端时间戳（毫秒） |

**总长度**：9 字节

### 0x05 —— AuthRequest

使用用户名和 token 做认证。

```elixir
<<0x05, request_id::u64, username::string, code::string>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `request_id` | `u64` | 请求关联 ID |
| `username` | `string` | 用户名（UTF-8） |
| `code` | `string` | token / 验证码 |

**总长度**：可变

当前运行时语义补充：

- 服务端会将 `username` 与 token claims 中的用户名做一致性校验
- token claims 当前至少可携带：`username`、`source`、`session_id`
- token claims 还可以携带：`cid` 或 `allowed_cids`，用于约束后续 `EnterScene`

### 0x06 —— FastLaneRequest

在已认证 TCP 会话上请求一个 UDP fast-lane attach ticket。

```elixir
<<0x06, request_id::u64>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `request_id` | `u64` | 请求关联 ID |

**总长度**：9 字节

### 0x07 —— FastLaneAttach

客户端通过 UDP 发送 attach 请求，把自己附着到已经认证的 TCP 会话。

```elixir
<<0x07, request_id::u64, ticket::string>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `request_id` | `u64` | 请求关联 ID |
| `ticket` | `string` | 由 TCP `FastLaneRequest` 返回的短期 ticket |

**总长度**：可变

### 0x08 —— ChatSay

场景内可靠聊天消息。当前语义是 **scene-local / AOI-local**：
由当前已进场连接发起，经 gate 转给当前玩家进程，再广播给 AOI 内的其他连接。

```elixir
<<0x08, request_id::u64, text::string>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `request_id` | `u64` | 请求关联 ID |
| `text` | `string` | 聊天正文 |

**总长度**：可变

### 0x09 —— SkillCast

简单技能施放请求。当前实现只支持最小可联调技能切片：
`skill_id = 1` 表示一个不带伤害结算的 pulse 技能广播。

```elixir
<<0x09, request_id::u64, skill_id::u16>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `request_id` | `u64` | 请求关联 ID |
| `skill_id` | `u16` | 技能 ID |

**总长度**：11 字节

### 0x60..0x7F —— Voxel reserved segment

体素世界使用 `WorldServer` 作为全局地图账本、租约、地块和跨 scene 事务协调者；
`SceneServer.Voxel.*` 作为租约区域内的 hot authority。`WorldServer` 不参与
chunk/object 状态热路径 tick，Scene 的写入必须来自当前 World lease / owner epoch。

体素段继续复用本协议的 `{packet, 4}` frame 与 `msg_type::8` body 头；payload 内所有多字节字段
统一使用 **big-endian / network byte order**，不得再使用另一套 payload 字节序。

同步原则：玩家相关表现可以本地预测或预演；体素数据必须服务器先计算，客户端只从
`ChunkSnapshot / ChunkDelta / ObjectStateDelta / VoxelIntentResult` 更新 confirmed voxel store。

| opcode | 方向 | 名称 | 当前用途 |
| --- | --- | --- | --- |
| `0x60` | C→S | `ChunkSubscribe` | 订阅 scene 内 chunk AOI |
| `0x61` | C→S | `ChunkUnsubscribe` | 取消订阅 |
| `0x62` | S→C | `ChunkSnapshot` | 下发 canonical chunk truth |
| `0x63` | S→C | `ChunkDelta` | 下发 chunk 增量 |
| `0x64` | C→S | `VoxelImpactIntent` | 可选：技能/工具请求地形影响，通常由技能系统内部触发 |
| `0x65` | C→S | `BuildReservationIntent` | 请求地块/区域内施工排他保留 |
| `0x66` | C→S | `BlueprintCreate` | 预留：创建 prefab/blueprint 定义 |
| `0x67` | C→S | `PrefabPlaceIntent` | 提交 blueprint_id + anchor intent，服务端重算 raster |
| `0x68` | S→C | `VoxelIntentResult` | 返回 accepted/deferred/rejected/stale |
| `0x69` | S→C | `ChunkInvalidate` | 服务端要求客户端丢弃/重订阅 |
| `0x6A` | C→S | `ParcelQuery` | 查询地块/权限/建设状态 |
| `0x6B` | C→S | `ObjectAction` | 预留：object/part 级交互 |
| `0x6C` | S→C | `ObjectStateDelta` | 预留：object 状态更新 |
| `0x6D` | S→C | `TagCatalogSnapshot` | 预留：tag catalog 下发 |
| `0x6E` | S→C | `AttributeCatalogSnapshot` | Phase 5.A：全局 attribute catalog 全量快照（见下文 `0x6E` 字段定义） |
| `0x6F` | both | `VoxelDebugProbe` | 本地/dev 调试保留 |
| `0x70` | C→S | `VoxelEditIntent` | typed 客户端编辑通道，见 §13.6.1 |
| `0x71` | S→C | `CatalogPatch` | Phase 1.4 envelope only；attribute/tag catalog 增量 patch（payload 内容 Phase 5 解释） |

体素 v1 canonical 参数：

| 参数 | 值 |
| --- | --- |
| `chunk_size_in_macro` | `16` |
| `micro_resolution` | `8` |
| refined occupancy | `512 bits = u64[8]` |
| normal block fixed header | `20 bytes` |

体素消息必须携带足够版本信息：`schema_version`、`chunk_version`、`cell_version`、`cell_hash`。
客户端可以做 build preview 和玩家侧技能表现预演；体素 confirmed truth 只能来自服务器下行。

#### 0x6E `AttributeCatalogSnapshot` payload（Phase 5.A）

全局 attribute catalog 全量快照，作为客户端冷启动 / 重连 / catalog 大幅变更时
的"基线"通道。增量更新走 `0x71 CatalogPatch`（schema_kind=0x01 attribute）。
本消息 **payload 只**，不含 opcode byte（在 `{packet,4}` frame 内紧跟 `msg_type::8`）。

Wire layout（一旦发出即冻结）：

```text
AttributeCatalogSnapshot
  catalog_version: u64         # monotonic；增量 patch 累加同一计数
  definition_count: u32
  definitions[definition_count] {
    id:            u32         # 全局 attribute_id；Phase 1.2 chunk-local `AttributeEntry.key_id`
                                # 在 Phase 5.A 之后语义升级为本 id（wire 字段不变）
    name_len:      u16
    name:          bytes(name_len)   # UTF-8，非空
    unit_len:      u16
    unit:          bytes(unit_len)   # UTF-8，允许为空（unitless attribute，如 boolean / enum）
    value_type:    u8                # 0x01 i16 / 0x02 u16 / 0x03 fixed32 (Q16.16) /
                                     # 0x04 enum8 / 0x05 bitset32（与 §12 Phase 1.2 一致）
    default_value: bytes(N)          # N = value_type 字节长度：i16/u16=2, fixed32/bitset32=4, enum8=1
    min_value:     bytes(N)
    max_value:     bytes(N)
    merge_rule:    u8                # 0x01 override / 0x02 add_delta / 0x03 max /
                                     # 0x04 min / 0x05 material_default
    dynamic:       u8                # 0 / 1（运行时可变 hint）
  }
```

字节量估算：单 `AttributeDefinition` 假设 `name="temperature"`(11B) + `unit="°C"`(3B UTF-8):
`4 + 2+11 + 2+3 + 1 + 2+2+2 + 1 + 1 = 31 B`；5 个内置 attribute 约 150~200B。

校验规则：
- `name` 必须非空合法 UTF-8；`unit` 允许为空（unitless）。
- `value_type` ∈ {0x01..0x05}（与 Phase 1.2 `AttributeEntry.value_type` 完全一致）。
- `default_value` / `min_value` / `max_value` 按 `value_type` 字节长度编码；必须满足
  `min_value <= default_value <= max_value`。
- `merge_rule` ∈ {0x01..0x05}；未知值在 normalize / decode 阶段都 raise（不做 forward-compat skip）。
- `dynamic` ∈ {0, 1}。
- `definitions` 内按 `id` 升序、不重复（producer canonicalize，consumer 在 wire boundary 再校验一次）。

Phase 5.A 仅落地服务端 wire 类型 `SceneServer.Voxel.AttributeCatalogSnapshot` /
`SceneServer.Voxel.AttributeDefinition`。Gate codec 集成、客户端 TS decoder、catalog
持久化（DataService schema）推到 Phase 5.C / 5.D。

## 服务端 → 客户端消息（0x80–0xFF）

### 0x80 —— Result

通用结果消息。不同上下文下可能携带不同负载。

**基础结果（ok/error）**：

```elixir
<<0x80, packet_id::u64, status::u8>>
```

**Movement ack（带角色位置）**：

```elixir
<<0x80, packet_id::u64, status::u8, cid::i64, location::vec3>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `packet_id` | `u64` | 对应请求的关联 ID |
| `status` | `u8` | `0x00 = ok`，`0x01 = error` |
| `cid` | `i64` | （可选）角色 ID |
| `location` | `vec3` | （可选）服务端权威位置，用于 uplink 纠偏 |

### 0x81 —— PlayerEnter（广播）

有其他玩家进入你的 AOI。

```elixir
<<0x81, cid::i64, location::vec3>>
```

**总长度**：33 字节

### 0x82 —— PlayerLeave（广播）

有玩家离开你的 AOI。

```elixir
<<0x82, cid::i64>>
```

**总长度**：9 字节

### 0x83 —— PlayerMove（广播）

附近玩家发生移动。

```elixir
<<0x83, cid::i64, server_tick::u32, location::vec3,
  velocity::vec3, acceleration::vec3, movement_mode::u8>>
```

其中 `server_tick` 是该 actor 的权威 movement tick。`movement_mode` 编码：
`0=grounded`、`1=airborne`、`2=disabled`、`3=scripted`。

**总长度**：86 字节

### 0x84 —— EnterSceneResult

`EnterScene` 的响应。

**成功**：

```elixir
<<0x84, packet_id::u64, 0x00, location::vec3, expected_seq::u32-big>>
```

`expected_seq` 是 server 期望的下一个 movement input 的 `seq`。客户端进场后必须把本地输入计数器对齐到该值（见 `world/local_player.rs::reset_with_seq`）。当前 server 端无 session 复用，每次进场新建 `PlayerCharacter`，因此值为 `1`；该字段把"双方都从 1 开始"的隐式契约显式化，并为未来的 session 复用留出对齐点。

**失败**：

```elixir
<<0x84, packet_id::u64, 0x01>>
```

### 0x85 —— TimeSyncReply

`TimeSync` 的响应。

```elixir
<<0x85, packet_id::u64, client_send_ts::u64, server_recv_ts::u64, server_send_ts::u64>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `packet_id` | `u64` | 回显客户端请求 ID |
| `client_send_ts` | `u64` | 回显客户端发送时的时间戳 |
| `server_recv_ts` | `u64` | 服务端收到该请求时的时间戳 |
| `server_send_ts` | `u64` | 服务端发送该响应时的时间戳 |

客户端可据此计算：

- RTT
- clock offset
- jitter 趋势

**总长度**：33 字节

### 0x86 —— HeartbeatReply

`Heartbeat` 的响应。

```elixir
<<0x86, timestamp::u64>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `timestamp` | `u64` | 服务端时间戳（毫秒） |

**总长度**：9 字节

### 0x87 —— FastLaneResult

`FastLaneRequest` 的 TCP 响应。

**成功**：

```elixir
<<0x87, packet_id::u64, 0x00, udp_port::u16, ticket::string>>
```

**失败**：

```elixir
<<0x87, packet_id::u64, 0x01>>
```

### 0x88 —— FastLaneAttached

UDP attach 握手的响应。当前 attach 成功后，客户端就可以把高频 `Movement` 上行和 `PlayerMove` 下行广播都切到 UDP fast lane。

**成功**：

```elixir
<<0x88, packet_id::u64, 0x00>>
```

**失败**：

```elixir
<<0x88, packet_id::u64, 0x01>>
```

### 0x89 —— ChatMessage

场景内可靠聊天广播。

```elixir
<<0x89, cid::i64, username::string, text::string>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `cid` | `i64` | 发言角色 ID |
| `username` | `string` | 发言用户名 |
| `text` | `string` | 聊天正文 |

### 0x8A —— SkillEvent

简单技能广播事件。当前由施法者的 AOI 连接共享，用于客户端显示最小技能表现。

```elixir
<<0x8A, cid::i64, skill_id::u16, location::vec3>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `cid` | `i64` | 施法角色 ID |
| `skill_id` | `u16` | 技能 ID |
| `location` | `vec3` | 技能触发位置 |

### 0x8B —— MovementAck

控制客户端的权威 movement 对账 payload。

```elixir
<<0x8B, ack_seq::u32, auth_tick::u32, cid::i64, location::vec3,
  velocity::vec3, acceleration::vec3, movement_mode::u8, correction_flags::u32,
  fixed_dt_ms::u16-big>>
```

| 字段 | 类型 | 含义 |
|------|------|------|
| `ack_seq` | `u32` | 被确认的客户端输入 seq |
| `auth_tick` | `u32` | 服务端权威 tick |
| `cid` | `i64` | 角色 ID |
| `location` | `vec3` | 服务端权威位置 |
| `velocity` | `vec3` | 服务端权威速度 |
| `acceleration` | `vec3` | 服务端权威加速度 |
| `movement_mode` | `u8` | 同 `PlayerMove` mode 编码 |
| `correction_flags` | `u32` | 对账语义位：Teleport / CollisionPush / StatusOverride / AntiCheatReject |
| `fixed_dt_ms` | `u16-big` | server 当前 `MovementProfile.fixed_dt_ms`（审计 B-M2，用于客户端 profile 漂移检测） |

**总长度**：96 字节（1 字节 msg_type + 95 字节 body：4+4+8+24+24+24+1+4+2）

## 当前连接状态约束（2026-04-10）

| 连接状态 | 允许消息 |
|----------|----------|
| `waiting_auth` | `AuthRequest`、`Heartbeat` |
| `authenticated` | `EnterScene`、`TimeSync`、`Heartbeat` |
| `in_scene` | `Movement`、`TimeSync`、`Heartbeat`、`ChatSay`、`SkillCast`、voxel `0x60..0x7F` 中允许的 C→S 消息 |

当前约束说明：

- 未认证时发送 `EnterScene` 会返回 `EnterSceneResult error`
- 未进入场景时发送 `Movement` 会返回通用 `Result error`
- `TimeSync` 在 `authenticated` 和 `in_scene` 状态都允许
- `auth_server` 不可用时，`AuthRequest` 会 fail closed，返回通用 `Result error`
- `scene_server` 或场景玩家进程不可用时，`EnterScene` 会 fail closed，返回 `EnterSceneResult error`
- 如果 token claims 中的用户名与 `AuthRequest.username` 不一致，认证会失败
- 如果 token claims 中声明了 `cid` 或 `allowed_cids`，`EnterScene(cid)` 必须满足该约束，否则失败
- `EnterScene(cid)` 还会经过 `auth -> data_service` 的真实角色归属校验；仅凭 token claim 不足以通过最终准入
- `ChatSay` 与 `SkillCast` 只允许在 `in_scene` 状态下使用；否则返回通用 `Result error`
- voxel C→S 消息只允许在 `in_scene` 状态下使用，并且必须由 gate 按当前连接的 `scene_ref`
  路由到对应 `SceneServer.Voxel.*` authority；客户端 payload 中的 `scene_id` 只能用于校验，不能用于绕过连接态路由
- UDP fast lane 现在具备显式生命周期语义：同一 TCP 会话只保留一个活跃 UDP peer，重附着会替换旧 peer，空闲 peer 会自动过期

## 运行时实现位置

- **服务端 codec**：`apps/gate_server/lib/gate_server/codec.ex`
- **voxel canonical codec（未来）**：`apps/scene_server/lib/scene_server/voxel/codec.ex`
- **连接级协议逻辑**：`apps/gate_server/lib/gate_server/worker/tcp_connection.ex`
- **TCP 分帧**：`gen_tcp` socket 上的 `{packet, 4}`
- **字节序**：统一使用大端（network byte order）

## 当前说明

这份规范描述的是当前有效的**主线协议格式**。  
当前已经具备 UDP fast-lane 的 ticket bootstrap/attach 握手，`Movement` uplink 与 `PlayerMove` downlink 都已迁入 UDP fast lane；同时已增加重附着替换、空闲过期和 TCP 关闭清理等基础治理能力。后续继续推进 snapshot / resync / sequence 等恢复机制时，应继续通过 dated docs 或同步更新本文，避免“线上行为”和文档描述脱节。
