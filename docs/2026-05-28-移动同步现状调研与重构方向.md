## 0. 调研方法与核心结论

**方法**：4 个并行子代理分层深挖现状 + codex（gpt-5.5）独立架构评审质证。

| 子代理 | 调研范围 |
|---|---|
| probe-authority | 服务端权威移动仿真（`scene_server` + `movement_core`/`scene_ops` Rust NIF） |
| probe-aoi-protocol | AOI、状态广播、线协议（`octree` + `gate_server` codec + 协议文档） |
| probe-client | 客户端预测/插值/和解/渲染（`clients/web_client`） |
| probe-docs | 设计文档、历史决策、已知痛点考古 |

**核心结论（与"系统落后待推倒"的预设相反）**：

> `ex_mmo_cluster` 的移动同步**已经实现了当下业界最先进的方案（Valve/Source 式 CSP + Server Reconciliation + Entity Interpolation 三件套），并已有完整的工程与验证基础设施**。它不是落后系统，而是"架构正确、实现基本到位、但存在 3 处架构级缺口 + 一批实现 bug"的系统。
>
> **因此正确的"自顶向下"不是推倒重来，而是：保留并强化已有框架，对 3 个真有架构缺陷的子系统做架构级重构，同时清掉实现 bug。** 推倒重来会摧毁一套已 work 且有量化验收门槛（`total_hard_snaps=0, drift≤8u`）的系统，引入大量回归，是错误且有害的方向。

## 2. 服务端 authority substrate 重构（自顶向下的真正对象）

> **经 codex 质证后的 reframe**：先进的是客户端预测栈（CSP/reconciliation/interpolation），落后的是**服务端 authority substrate**。下列即"自顶向下"真正要重构的层。新增最根本的"场景权威仿真"，原"3 个目标"升级为 §6 的四支柱。

### ★ 最根本：场景权威仿真所有权（scene authority ownership）
- 当前是 **per-player authoritative actor**（每个 `PlayerCharacter` 自管权威态与定时器），不是 **scene-authoritative simulation**。
- 后果：跨玩家碰撞、AOI、lag compensation、scene/node handoff 都缺少**场景级统一时间与状态历史契约**；"拆 mailbox"只是降 jitter 的表层手段，治标不治本。
- 这是支柱 2 的真正内核（codex 指出的遗漏）。

### A. 统一时间模型（最根本，A1/A2/D1 共同根因）
- **全栈把 `server_tick`（计数器）当时间用**：`snapshotTimeSecs = serverTick * tickDurationSecs`。
  - `remotePlayer.ts:156`
- **PlayerMove(0x83) 无 wall-clock 时间戳**，客户端无法重建真实服务器时间线，只能用"tick 计数 × 固定间隔"近似；tick 不规律（丢包/重启）时插值时间轴跳变。
- **时钟同步算出却断路**：客户端用标准 NTP 半差正确算出 `serverClockOffsetMs`，但**从未用于插值时间轴**（`sampleMotion` 仍用 `performance.now()`）。
  - `remotePlayerController.ts:124-130, 190-195`（仅暴露给 debug snapshot）
- **方向**：引入 wall-clock + tick 双时基；广播携带服务器时间戳；时钟偏移真正接入插值时钟。

### B. 进程 / tick 模型（远端抖动根因 R1）
- **`PlayerCharacter` 单 GenServer mailbox 混合职责**：`movement_tick` 与技能/聊天/伤害共享一个 mailbox，慢消息（技能 5–20ms）阻塞移动 tick → snapshot 间隔抖动。
  - `docs/voxel-server-authority/2026-05-10-multiplayer-hot-path-redesign.md` §1.1（根因 R1）
- **无场景统一 tick barrier**：每玩家独立定时器，玩家间 tick 不对齐，无法精确同帧碰撞/战斗判定。
- **10Hz tick 偏低**：单 tick 最大位移 600cm，可穿薄墙；现代 MMO 普遍 20–30Hz。
- **方向**：分离 movement tick 与慢消息（独立移动进程 / 场景统一 fixed-tick sim loop）；tick 提至 20–30Hz。

### C. 服务端权威碰撞融合（"碰撞由服务器决定"这条原则目前实为空）
- **体素地形未注册为碰撞体**，`rapier` 物理步进（`physics_pipeline.step()`）**从未被调用**——rapier 仅当 collider 注册表 + `move_shape()` teleport 用。
  - `scene_ops` PhySys 初始化了管线但无 step 调用；`scene_ops/src/character/*`
- **两套碰撞路径未融合**：rapier `KinematicCharacterController` 与 Elixir `VoxelCollision` 各行其是，结果不互通，有撕裂风险。
- **批量 replay 碰撞语义错误**：`replay_queued_inputs_with_collision` 对整段多帧位移只做**一次**碰撞解算，中途障碍会被穿透。
- **客户端预测无碰撞**：客户端预测不查体素缓存，预测与权威在碰撞维度必然分歧。
- 现状桥接点：`movement/VoxelCollision`（stateless read-only adapter）已存在；坐标约定已定（`movement_cm → voxel_micro`）。
  - `apps/scene_server/lib/scene_server/movement/README.md`、`docs/2026-05-24-server-authoritative-voxel-collision-plan.md`（MVP 规划，未实施）
- **方向**：体素 terrain 作为权威碰撞源；逐帧碰撞解算；客户端预测查体素缓存；明确 rapier 的去留。

## 4. 设计意图 vs 现状落差（probe-docs）

设计者原始目标（`docs/2026-04-12-移动同步架构设计.md`）：本地手感快、服务器权威、纠错不抖、远端平滑、可增量迁移、可扩展到加速度/jerk。目标态明确包含 CSP+SR、snapshot interpolation、presentation smoothing、输入流协议演进（Phase A→B）。

**已落地（扎实）**：固定 tick 权威 + latched input、CSP + reconcile 三档 + ReplayGovernance、远端 ring buffer + 插值延迟 + 有界外推、三层状态分离、`movement_core` 统一内核、UDP fast lane、AOI priority 分级、跳跃 airborne 同步。

**仍是缺口**：`correction_flags` 语义化未接入；服务端权威体素碰撞未实现；`PlayerCharacter` 单 mailbox 问题（已有重设计决策稿未实施）；输入流协议终态（Phase B，仍传 position/velocity）；Sync Budget v1 节流未打通；客户端插值不感知 `delivery_interval`；`SceneOps` vs `Movement.State` 双重表示未最终收敛。

**最大架构缺口**：移动同步与体素碰撞的融合——"碰撞由服务器决定"在碰撞维度上实际为空。

## 5. codex 独立架构评审结论（gpt-5.5，已整合）

codex 在本地 Windows 沙箱失败后改用 GitHub connector 读 `dyzdyz010/ex_mmo_cluster@5a0d2e7` 源码核验。**最终判定与本判断一致：CONSOLIDATE+COMPLETE（不推倒重来）**。三点关键深化/修正：

**1. 核心判断 reframe（重要）**：不应表述为"整个系统已 SOTA、只补小洞"。更准确：**CSP / reconciliation / interpolation 客户端主链条先进、值得保留；真正落后的是服务端 authority substrate——时间基准、进程调度、场景所有权、碰撞、协议契约**。这部分印证了"需要重构"的直觉，只是重构对象精确锁定在服务端 authority 层。

**2. 遗漏的最根本问题：scene authority ownership**。当前是 *per-player authoritative actor*，不是 *scene-authoritative simulation*。拆 mailbox 只能降 jitter，**不能**为跨玩家碰撞、AOI、lag compensation、scene/node handoff 提供统一的时间与状态历史顺序。这比"单 mailbox / 拆 movement lane"更深，是支柱 2 的真正内核（已据此升级，见 §2、§6）。

**3. 两处 mislabel 已纠正（bug → 架构债）**：
- `movement_state.tick` 取自 `client_tick`（`integrator.rs:93-96,115-118`）是**安全/authority 边界问题**，非普通 bug——server 必须分配 authoritative tick，client tick 仅作输入元数据。上移到支柱 1。
- 协议硬编码 offset / 无 magic / 无 version（`gateProtocol.ts:197-244`）是**协议契约架构债**，非实现 bug。上移到支柱 1。

**本地交叉核实（解决 codex 与子代理的两处矛盾）**：
- **碰撞 replay**：codex 称"每帧 resolve"——经**本地**核实**不成立**。`player_character.ex:833-850 replay_queued_inputs_with_collision` 实为 `Enum.reduce` 逐帧纯运动学积分得最终 `proposed_state`，再对 `anchor→proposed` **整段位移做一次** `resolve_voxel_collision`。**中途穿透风险真实存在**（probe-authority 正确；codex 引错行号 `786-803`〔实为 observe emit〕，因 GitHub 版本差异误判）。
- **delivery_interval awareness**：codex 正确。客户端 `remotePlayer.ts:78,160-173` 已有 `resolveInterpolationDelaySecs(snapshot.deliveryInterval)` 自适应逻辑。原 b6 据此降级（见 §3）。

> 教训：本地工作区是真相源；codex 基于 GitHub 提交版的核验有版本差异局限（已预警），交叉核实双向纠错有效。

## 6. 重构方向与排序（整合 codex 后，待 present design 定稿）

四支柱（取代原"3 块"），按 codex 建议的依赖序：

**支柱 1 — 协议契约 + 统一时间基准（地基，技术最可控）**
- 协议加 version/magic/schema guard，消除硬编码 offset 脆弱性（b4 升级）。
- `PlayerMove(0x83)` 加 server wall-clock 时间戳。
- server 分配 authoritative tick；`client_tick` 降为输入元数据（堵 authority 漏洞，b3 升级）。
- 客户端把 `serverClockOffsetMs` 真正接入插值时间轴（A2）。
- 修 jitter estimator（b2）。

**支柱 2 — 场景权威仿真 + 进程/tick 模型（核心，最根本，风险最高）**
- per-player actor → **scene-authoritative simulation**：引入 scene-owned fixed sim tick/barrier。
- `PlayerCharacter` 退化为 command/session adapter；movement tick 脱离慢消息 mailbox。
- tick 升 20–30Hz（放本阶段**后半**，不要盲目先调频）。

**支柱 3 — 碰撞权威融合**
- 定**单一**碰撞 truth：voxel 为主，或 voxel chunk materialize 成 rapier static collider；不容两套并存。
- 修复"整段位移一次碰撞"→ 逐帧碰撞（本地已确认中途穿透风险真实，见 §5）。
- 客户端预测查 voxel cache（在服务端碰撞权威稳定**之后**）。

**支柱 4 — AOI / lag compensation / cluster handoff**
- AOI 刷新绑定 movement/spatial 更新（弃 1Hz，b5/b7）。
- scene state ring buffer 支撑 lag compensation。
- 跨节点/场景 handoff。

**风险序**：支柱 2 最高（改状态所有权、消息顺序、handoff 边界）> 支柱 3 > 支柱 1（兼容性有风险但技术最可控）。

> 验收沿用既有量化门槛（`total_hard_snaps=0`、`drift≤8u`）+ 新增时间轴对齐 / 抖动 / 逐帧碰撞 / AOI 延迟指标，全部走 CLI observe + 结构化日志（遵 AGENTS.md）。
