# 移动同步架构实现（2026-04-20）

> 记录 `clients/bevy_client` + `apps/scene_server` 当前已落地的客户端预测 + 服务器对账 + 视觉平滑三层同步架构。可作为继续扩展（跳跃、位移技能、UE5 客户端迁移）的起点文档。
>
> 配套设计文档：`docs/2026-04-20-移动同步扩展设计.md`。

## 1. 策略定位

- **血统**：Valve/Quake 2001 客户端预测 + 服务器对账，叠加 Unreal `NetworkSmoothingMode::Exponential` 的视觉平滑层。
- **定位**：工业界已验证二十余年的最佳实践，非自创算法。我们选择站在巨人肩上，将工程精度放在参数调优 + 降级分层上。
- **范围**：当前只覆盖"平地位移"。跳跃、位移技能、被击退等脚本化移动尚未落地，见扩展设计文档。

## 2. 分层架构

```text
┌──────────────────────────────────────────────────────┐
│  Render Lane   (60 Hz)                               │
│    app.rs::LocalRenderPrediction                     │
│    pending_correction → exp(-rate·dt) 视觉衰减       │
├──────────────────────────────────────────────────────┤
│  Predict Lane  (10 Hz 固定 tick)                     │
│    sim/predictor.rs + world/local_player.rs          │
│    本地近似运动学积分，领先服务器 ≈ RTT/2            │
├──────────────────────────────────────────────────────┤
│  Authority Lane (10 Hz)                              │
│    apps/scene_server/native/scene_ops (rapier3d-f64) │
│    rapier 物理/碰撞权威，广播 MovementAck            │
└──────────────────────────────────────────────────────┘
```

两端**不共享**物理代码：

| | 服务端 | 客户端（当前 Bevy） |
|---|---|---|
| 引擎 | rapier3d-f64 0.16 完整刚体/碰撞 | 无 |
| 模型 | 完整物理步进 | 纯运动学：`v' = clamp(v + a·dt)`，`p' = p + v·dt` |
| 代码 | `apps/scene_server/native/scene_ops/` | `clients/bevy_client/src/sim/predictor.rs` |
| 共享点 | 只共享一份 `MovementProfile`（`max_speed / max_accel / max_decel / max_jerk` 常量） | 同上 |

这是业界通行的**近似预测器（Approximate Predictor）**模式——客户端只需要"像"服务器，不需要"是"服务器。平地差异小于 soft 阈值时不触发重放，一旦遇到墙/坡/台阶，100 ms 内由服务器 ack 拉回。

## 3. 核心算法五环节

### 3.1 确定性定步仿真

- Tick 频率 10 Hz（100 ms 一 tick）。
- 输入帧 `MoveInputFrame { seq: u32, client_tick: u32, input_dir: Vec2, dt_ms: u16, flags: u8, speed_scale: f32 }`。
- `seq` 客户端侧单调递增，作为对账的**主键**；`client_tick` 作为时间键。
- `flags` 目前定义了 `MOVEMENT_FLAG_BRAKE` 等位，预留跳跃/技能扩展位。

### 3.2 客户端预测（Predict）

位置：`clients/bevy_client/src/world/local_player.rs::LocalPredictionRuntime`。

流程：
1. `push_input(frame)` 把输入 append 到 `InputHistory`。
2. 立即调用 `predictor::step(previous, input, profile)` 推进一帧，结果 append 到 `PredictedHistory`。
3. 玩家视觉**立即**更新到新位置——感知零延迟。

`InputHistory`/`PredictedHistory` 为环形缓冲，容量由 governance 参数控制。

### 3.3 服务器对账（Reconcile）

位置：`clients/bevy_client/src/sim/reconcile.rs`。

服务器 ack 携带 `{ auth_tick, ack_seq, auth_state }`。

**按 seq 匹配**（优先）：
```text
let matched = history.state_at_seq(ack.ack_seq);
```
匹配到本地那一帧预测后，比对 `auth_state.position` 与本地预测位置：

| 差距 | 动作 | 代码分支 |
|---|---|---|
| `< soft_threshold` (2.0u) | 信任本地，只推进 anchor | `ReconcileOutcome::Accepted` |
| `< hard_snap_distance` (256.0u) | 保留 ack 之后的未确认输入，从 `auth_state` 起**重放**所有 future inputs | `ReconcileOutcome::Replayed` |
| `≥ hard_snap_distance` | HardSnap：清空历史直接传送 | `ReconcileOutcome::HardSnap` |

**按 tick 回退**（seq 缺失时）：
```text
if state_at_seq 未命中:
    state_at_tick(ack.auth_tick) + drift 评估 → 走同一套阈值
```
覆盖客户端重启 / 丢包 / 乱序导致 seq 空洞的场景。

**Accepted 分支特例**：当 `authoritative.tick >= latest.tick` 时，仍要 `push_sample(authoritative)`——否则同一 seq 的连续 ack 会让 anchor 停滞。

### 3.4 视觉平滑（Visual Smoothing）

位置：`clients/bevy_client/src/app.rs::LocalRenderPrediction`。

关键不变式：**`pending_correction` 是替换不是累加**（否则会重复计入上一次修正）。

```rust
// sync_full_state：每次对账后重算
pending_correction = old_rendered_pos - new_anchor_pos;
if pending_correction.length() > VISUAL_HARD_SNAP_DISTANCE {
    pending_correction = Vec3::ZERO;  // 视觉 HardSnap
}

// advance_local_render_prediction：每渲染帧衰减
pending_correction *= (-smoothing_rate_hz * dt).exp();
if pending_correction.length() < epsilon {
    pending_correction = Vec3::ZERO;
}
rendered_pos = anchor_pos + pending_correction;
```

默认参数：
- `DEFAULT_VISUAL_SMOOTHING_RATE_HZ = 15.0`
- `VISUAL_SNAP_DISTANCE = 96.0`
- `VISUAL_HARD_SNAP_DISTANCE = 256.0`

玩家看到的是修正量**平滑漂移回零**，而非瞬移。

### 3.5 治理 / 降级（Governance）

位置：`clients/bevy_client/src/sim/governance.rs`。

默认阈值：

| 参数 | 值 | 作用 |
|---|---|---|
| `soft_threshold` | 2.0 u | 低于此值信任本地，只更新 anchor |
| `hard_snap_distance` | 256.0 u | 超过此值直接 HardSnap |
| `max_replay_frames` | 32 | 重放帧数上限，超过即 HardSnap |
| `max_pending_inputs` | 64 | 输入缓冲上限，超过即截断 |

`ReplayGovernance` 在对账过程中实时统计 `total_corrections / total_replays / total_hard_snaps / total_window_trims`，通过 stdio `reconcile_stats` 事件暴露给 CLI 诊断。

### 3.6 远端玩家插值（Remote Interpolation）

位置：`clients/bevy_client/src/world/remote_player.rs`。

与本地预测互补：对**其他玩家**采用延迟插值 + 短期外推。

- `INTERPOLATION_DELAY_SECS = 0.1`（保持 100 ms 缓冲，允许乱序 ack 仍能插值）。
- `MAX_REMOTE_EXTRAPOLATION_SECS = 0.25`（超过就停在最后已知位置，不瞎猜）。

## 4. 传输层

| 通道 | 协议 | 用途 |
|---|---|---|
| 控制面 | TCP `127.0.0.1:20002`（packet:4 分帧） | 登录、进入场景、技能、广播事件 |
| 快车道 | UDP `127.0.0.1:20003` | 高频 movement upstream + ack |

- `MoveInputFrame` 上行走 UDP，允许丢包（反正有 seq + reconcile 兜底）。
- `MovementAck` 下行同样走 UDP。
- 其他消息仍走 TCP，保证有序可靠。
- 客户端启动后先 TCP 握手 + 认证，再自动 attach UDP 快车道，失败则自动降级回 TCP（`movement_transport` 字段对外可见）。

线协议细节见 `docs/2026-04-10-线协议规范.md`。

## 9. 关键工程教训

1. **视觉修正是替换不是累加**。`old_rendered` 本身就包含了上一次的修正，再累加会重复计入，导致抖动不衰减。只有 exp 衰减归零才能保证连续性。
2. **Accepted 分支必须在 `auth_tick` 前进时 `push_sample`**。否则同一 seq 的多次 ack 会让 anchor 停在旧位置，视觉 anchor 不动，pending_correction 也就永远不更新。
3. **Idle brake 填充**。`extend_prediction_through(auth_tick)` 对未发送任何输入的间隙要用 `MOVEMENT_FLAG_BRAKE` + ZERO 方向的虚拟输入帧，而非"惯性滑行"，否则停住后还会预测出漂移。
4. **seq 优先，tick 回退**。纯 tick 对账在丢包/乱序场景下会频繁误判；纯 seq 又无法容忍客户端重启。两者联合才完整覆盖。
5. **`mix demo.run` / `mix demo.observe` 已归档**。任何新 smoke 应走 `scripts/e2e-live-movement.*` 或 `cargo test --lib` 代理，不再依赖已归档 mix task。

## 10. 参数调优指南

| 参数 | 当前值 | 调小会 | 调大会 |
|---|---|---|---|
| `soft_threshold` | 2.0 u | 更多 replay，CPU 开销↑ | 位置漂移更大，手感变差 |
| `hard_snap_distance` | 256.0 u | 更容易瞬移 | 重放窗口过长，可能失控 |
| `max_replay_frames` | 32 | 更早 HardSnap | 长时间卡顿后仍尝试重放 |
| `smoothing_rate_hz` | 15.0 | 修正拖尾时间↑，视觉"肉" | 修正更快归零，但可能看到抖动 |
| `INTERPOLATION_DELAY_SECS` | 0.10 | 远端位置更即时但抖动大 | 远端稳但视觉延迟↑ |
| `MAX_REMOTE_EXTRAPOLATION_SECS` | 0.25 | 丢包时远端停得更早 | 丢包时远端可能"滑出去" |

调整前务必重新跑 `scripts/e2e-live-movement.ps1` 确认指标仍在阈值内。

## 11. P2 统一架构（2026-04-20）

### 11.1 为什么再动一次

P1 把预测、重放、插值三件事做对了，但算法内核被**复制了三份**：

- `clients/bevy_client/src/sim/predictor.rs`（f32 Bevy 版）
- `apps/scene_server/native/movement_engine/src/integrator.rs`（f64 NIF 版）
- `apps/scene_server/lib/scene_server/movement/integrator.ex`（Elixir 参考版）

一次调参要改三次、测三次、合三次，长期必然漂移。P2 不改算法、不改线协议，只把算法内核抽到单一 crate，把 `MovementMode` 扩出 `Scripted` 插槽，把远端插值对齐 Valve 默认。**最保险路线，兼收未来位移技能与 150ms 业界共识。**

### 11.3 MovementMode 状态机

四个变体，当前仅 `Grounded` 真正有积分路径，其余为挂载点：

| 变体 | 触发 | 本轮行为 | 未来 |
|------|------|----------|------|
| `Grounded` | 默认 | jerk-limited walking kinematics | 持续 |
| `Airborne` | 跳跃（预留） | 复用 `grounded_step`（不跳） | 替换为弹道 step |
| `Scripted` | 位移技能（预留） | no-op（保持 pos/vel/accel） | 由技能系统写入 root motion |
| `Disabled` | CC / 眩晕 | 清零 vel/accel，位置冻结 | 持续 |

`MovementMode::transition(prev, &input) -> Self` 当前永远返回 `prev`；未来的技能 / CC / 跳跃系统在调用 `step` 前先修改 `previous.movement_mode`，transition 留作未来输入驱动的模式切换挂钩。

### 11.4 bit-exact 契约

- **NIF ↔ Elixir 参考实现**：完全 bit-exact，由 `integrator_golden_test.exs` 逐 tick 断言 `< 1e-9`
- **movement_core ↔ Bevy 客户端**：`f64 → f32` 单向量化，往返精度预算 `≤ 1e-4`，由 `sim/predictor.rs::tests::core_bridge_round_trip_matches_core_reference_values` 把守
- **算法黄金值**（idle + (1,0) 方向 + dt=100ms + 默认 profile）：`acceleration.x = 900`，`velocity.x = 90`，`position.x = 9`，三处独立测试互为锚点

### 11.5 远端插值：对齐 Valve 2001 默认

```
INTERPOLATION_DELAY_SECS      0.10  →  0.15   （Bernier 2001 / cl_interp 默认）
MAX_REMOTE_EXTRAPOLATION_SECS 0.25         （保持；Unreal NetworkSmoothingMode 参考）
```

100ms tick cadence 下，0.15s 延迟保证 buffer 里始终有 ≥2 个 snapshot 可插值；单包丢包不再掉进 extrapolation 路径。

### 11.6 参数路线图标记

`movement_core/src/profile.rs` 的 `Default` 实现每个字段都有行内注释：引用 Unreal CMC / Valve / Amazon New World 默认值，并标注"C 路线"（调参阶段）备选值，方便未来 per-class / per-mount profile 落地。
