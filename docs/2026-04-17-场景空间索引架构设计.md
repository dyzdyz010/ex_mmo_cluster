# 2026-04-17 场景空间索引架构设计

> 本文档是 `/autopilot` 场景索引选型讨论的落盘结论。用于后续实施与 benchmark
> 对比时的参考基线，不代表已落地代码。

## 1. 背景

`scene_server` 当前有三套空间相关实现：

- `native/scene_ops`：基于 `rapier3d-f64` 的物理系统（collision / 移动 tick），内部
  自带 BVH，用于**物理模拟**
- `native/octree`：自写的八叉树，用于**AOI 邻近查询**（玩家看见谁、向谁广播）
- `native/coordinate_system`：遗留坐标系统，与 octree 功能重叠，准备退出

2026-04-17 对 `octree` 做了正确性审阅，发现 3 处严重缺陷：

1. `merge()` 仅统计直接子节点的 objects，若孙层有数据会被整体丢弃
2. `max_depth` 实际未在 insert/split 中校验，共点样本超过 capacity 会栈溢出
3. `insert` 的 check-then-act 在 `drop(read) → acquire(write)` 之间存在 race，
   新 item 可能落到已 split 的中间节点的 objects 数组中而变得不可达

此外它只对外暴露树状锁、性能随并发度退化，且 `Octree::update_position` 未导
出为 NIF。

同时业务侧提出两条新约束：

- **常规吞吐要更高**（10k 玩家均匀分布的普通场景）
- **密集场景要能扛**（主城 1000+ 玩家同场景限时活动）

因此本次不是"修 octree"，而是**重新评估整个 AOI 索引选型**。

## 2. 核心约束

| 约束 | 含义 | 量化参考 |
|------|------|---------|
| 高吞吐 | insert / update / range query 的稳态 QPS | 10k entity × 10Hz tick = 100k ops/s |
| 密集承载 | 单场景热点区域 1000 人 AOI 不崩 | 主城活动 500m² 内 1000 玩家 |
| 并发友好 | 能与 rayon / 多 scheduler 共存，不被整树锁卡住 | 多 scene 节点并行 |
| 代码可维护 | Rust crate 规模与调试难度 | 目标 < 300 LOC 核心逻辑 |

## 3. 候选方案对比

| 方案 | 常规吞吐 | 密集承载 | 并发性 | 代码量 | 备注 |
|------|---------|---------|--------|--------|------|
| **Uniform Spatial Hash Grid (SHG)** | ★★★★★ | ★★★ | ★★★★ | ~120 LOC | 热点单元线性扫描，但常数低 |
| Hierarchical Hash Grid (HHG) | ★★★★ | ★★★★ | ★★★ | ~280 LOC | 多分辨率，热点区域升级细粒度 |
| Loose Octree（修 bug 后）| ★★★ | ★★★★ | ★★ | ~600 LOC | 自适应细分，但整树锁或路径锁复杂度高 |
| Hybrid（SHG + 热点局部细分）| ★★★★ | ★★★★ | ★★★ | ~400 LOC | 工程性价比居中 |
| BVH | ★★ | ★★★ | ★ | - | 动态更新昂贵，不适合高频 move |

## 4. 关键洞察

**密集区域的瓶颈不在索引结构，而在 N² 交互数。**

举例：主城 1000 人每人 AOI 50m 内仍有几百个邻居。即使索引查询从 O(log N) 降到
O(1)，**每 tick 向所有可见邻居同步状态**的总工作量仍然是 O(visible_count)。
换索引结构救不了密集场景。

真正的解在**兴趣管理层**（interest management）：限制每玩家可见邻居数、
对远端降频 tick、对超密场景做 zone sharding。索引层只要能稳定 O(1) 给出
"候选邻居集合"，之后由兴趣层裁剪即可。

## 5. 推荐架构：L1 SHG + L2 兴趣管理

### 5.1 L1 索引层 — Uniform Spatial Hash Grid

**数据结构**（Rust 侧）：

```rust
pub struct SpatialGrid {
    cell_size: f64,                                    // 典型值：50.0（≈ AOI 半径）
    buckets: DashMap<(i32, i32, i32), BucketData>,     // 或 HashMap + parking_lot RwLock
    entities: DashMap<EntityId, EntityState>,         // 实体 → (cell, pos)
}

struct BucketData {
    members: SmallVec<[EntityId; 16]>,                 // 小向量，避免堆分配热点
}
```

**关键 API**：

| NIF | 复杂度 | 用途 |
|-----|--------|------|
| `new_grid(cell_size)` | O(1) | 构造 |
| `insert(grid, id, pos)` | O(1) | 新 entity 入场 |
| `update(grid, id, new_pos)` | O(1) 摊还 | 位置变化；只在跨 cell 时迁移桶 |
| `remove(grid, id)` | O(1) | 离场 |
| `query_range(grid, center, radius)` | O(cells × k) | AOI 查询，遍历覆盖的 cells |

**并发模型**：

- 按 cell key 做桶级锁（DashMap 内部分片），多 cell 操作天然并行
- 与 rayon 兼容：range query 可 parallel 扫描覆盖的 cells
- 没有树结构，不存在 rebalance / split / merge 的全局锁瓶颈

**参数选择**：

- `cell_size = AOI_radius`：每次 query 固定扫描 27 个 cell（3³ 立方体覆盖）
- 场景地图大小假设 2km × 2km × 高度差小 → cell 数量 40×40×few = 数千个 bucket

### 5.2 L2 兴趣管理层 — 密集场景的真正解

在 `agent_server` / `scene_server` 的 AOI 订阅逻辑中加一层：

**策略 A：Top-K 邻居截断**
```elixir
candidates = Grid.query_range(grid, pos, aoi_radius)
visible = candidates |> sort_by_distance(pos) |> Enum.take(50)
```

每玩家最多可见 50 个邻居，超出按距离丢弃。这是密集场景最重要的一道闸。

**策略 B：LOD tick 降频**
```
|距离| < 20m  → full tick (10 Hz)
|距离| 20-50m → half tick (5 Hz)
|距离| > 50m  → 不广播
```

与 Top-K 叠加使用：核心圈用满带宽，外圈用摘要。

**策略 C：主城 zone sharding（按需启用）**

同一逻辑主城物理上分多个 scene_server 实例，跨 shard 只同步位置摘要或不同步。
默认不开；当单 scene 达到密度阈值时由世界管理器触发。

## 6. 为什么不是 Octree

| 维度 | Octree | SHG + 兴趣层 |
|------|--------|-------------|
| 数据特征匹配 | 适合非均匀稀疏几何 | 适合"均匀玩家点 + 局部密集" |
| 密集场景 | 自适应细分但仍受 N² 限制 | 索引不变，密集问题交给兴趣层 |
| 并发 | 需要复杂路径锁或整树锁 | 天然分桶，锁粒度细 |
| 代码量 | ~500 LOC + 复杂边界条件 | ~120 LOC + 直观 |
| 与 rapier3d 重合度 | rapier3d 内部也是 BVH，有冗余 | 职责清晰解耦 |
| 现状风险 | 3 处正确性 bug 待修 | 新结构，可从白盒开始 |

## 7. 实现路径

采用 **POC-first**，不直接全量替换：

### Phase A — SHG Rust crate POC（~1 天）

- 新 crate：`apps/scene_server/native/spatial_grid/`
- 实现 `new_grid / insert / update / remove / query_range`
- Elixir 封装：`SceneServer.Native.SpatialGrid`
- 单元测试：插入、跨 cell 迁移、半径查询、并发竞争

### Phase B — 兴趣层接入（~1 天）

- 在 `scene_server` AOI 订阅路径加 Top-K 截断 + LOD
- 保留 octree 旧路径开关（feature flag），便于对比

### Phase C — Benchmark 对比（~0.5 天）

合成三组场景：

1. **Baseline**：10k 玩家均匀分布，move tick 10 Hz
2. **Crowd**：1k 玩家挤在 100m × 100m 主城，move tick 10 Hz
3. **Mixed**：5k 玩家 + 500 密集集群

测量：
- 单 tick 平均 query_range 耗时
- 单 tick AOI 计算总耗时
- 内存占用峰值
- 同时跑 SHG / octree（修 bug 后）/ 混合三套做 A/B

**注意**：对比前必须先把 octree 的 3 个 bug 修掉，否则是在和坏数据比，无意义。

### Phase D — 决策

- 若 SHG 在三个场景都不输 octree 且密集场景明显更稳 → 全量切换，octree 归档
- 若密集场景 SHG 明显差 → 升级到 Hybrid（SHG + 热点局部细分）
- 若差距微小 → 保留 octree，但修 bug，兴趣层两者通用

benchmark 和灰度切换。

## 9. 风险与未决问题

- **SmallVec 溢出**：密集 cell 内 entity 数 > 16 会退化到堆分配。需要 benchmark
  观察是否足够
- **Cross-cell query 的 cache miss**：query 27 个 cell 的 bucket 可能分布在内存
  不同位置；若成为瓶颈可考虑 morton code 顺序存储
- **Tick 节奏协调**：兴趣层 LOD 降频会导致客户端外圈插值更粗糙，需要客户端配合
  平滑处理
- **Zone sharding 的跨 shard 可视性**：玩家站在 shard 边界时需要看到对面的人，
  细节需单独设计（shard ghost / boundary 缓冲带）
