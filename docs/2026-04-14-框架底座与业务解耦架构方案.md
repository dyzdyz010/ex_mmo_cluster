# 2026-04-14 框架底座与业务解耦架构方案

## 1. 结论

### 1.1 能不能做成“基础底座 + 上层业务逻辑”？

**能，而且非常适合当前项目继续往这个方向演进。**

当前仓库已经具备了做这件事的几个关键前提：

- gate / auth / scene 已经是分层服务
- movement / combat / npc / AOI 已经开始形成可复用子域
- 玩家与 NPC 都已经在往“权威 actor + 可复用 runtime”收敛
- Rustler 已被限制在性能敏感、纯计算的底层热路径

这意味着现在不是从零开始“发明框架”，而是：

> **把已经出现的 runtime kernel 抽出来，稳定其接口，再让具体游戏业务实现放到它上层。**

---

## 2. 对 mlua 的判断

## 2.1 结论

**不建议把 mlua / Lua 作为这次框架化的第一步主方案。**

更好的路径是：

1. **先做 Elixir/OTP 层的框架化边界**
2. **把业务逻辑抽成上层模块 / app / registry / data-driven content**
3. **如果以后真的需要脚本化，再谨慎引入 Lua 或别的脚本层**

换句话说：

> **先把“框架接口”设计好，再决定是否需要脚本语言；不要反过来为了 Lua 去设计框架。**

---

## 2.2 为什么不建议现在直接上 mlua

### 原因 1：当前权威运行时主要在 BEAM

当前项目的强项是：

- OTP supervision
- GenServer actor 模型
- scene / gate / auth 的服务边界
- 玩家 / NPC 的权威生命周期管理

如果现在把业务逻辑直接塞进 `mlua`：

- 运行时链路会变成：
  - **BEAM -> Rustler -> Lua VM**
- 你会一下子拥有三个状态边界：
  - Elixir 状态
  - Rust NIF 状态
  - Lua 脚本状态

这会明显抬高：
- 调试复杂度
- 崩溃恢复复杂度
- 观测复杂度
- 热更新一致性复杂度

### 原因 2：NIF 不适合承载大量高层业务编排

Rustler 很适合：
- movement integrator
- replay
- spatial math
- serialization hot path

但不适合：
- 复杂脚本状态机
- 长时间行为树 tick
- 大量业务分支逻辑
- 频繁脚本回调与宿主交互

如果 Lua 放在 NIF 里承担太多业务逻辑，就会偏离 Rustler 的最佳使用边界。

### 原因 3：你现在真正要解决的不是“脚本语言缺失”，而是“边界设计”

你想要的是 KBEngine / Skynet 那种：
- 基础设施稳定
- 业务逻辑可替换
- 框架与内容分离

这个目标的关键不是 Lua 本身，而是：

- **接口清晰**
- **runtime kernel 清晰**
- **业务扩展点稳定**

Lua 只是“可能的承载形式之一”，不是根因。

---

## 2.3 如果以后真的要用 Lua，推荐什么姿势？

### 推荐：Lua 只做高层脚本 / 内容脚本

适合 Lua 的场景：
- 地图脚本
- 任务流程
- 副本事件
- NPC 高层配置化行为
- 公式/条件表达式
- 运营活动脚本

不适合 Lua 直接接管的场景：
- actor 生命周期主循环
- movement/combat authority 核心
- AOI 核心
- 高频 tick 纯热路径

### 更稳的姿势

如果以后要用 Lua，更推荐：

1. **业务脚本层只接收纯数据上下文**
2. **返回 declarative intent / command**
3. **真正执行仍然在 Elixir authority runtime 内完成**

也就是：

- Lua 决定“想做什么”
- Elixir/Kernel 决定“能不能做、怎么做、何时广播”

而不是让 Lua 直接成为 authority。

---

## 3. 推荐的框架化目标形态

## 3.1 三层结构

### A. Kernel / Runtime Platform（底座）

职责：
- actor lifecycle
- fixed tick scheduling
- AOI / spatial index
- movement engine 接口
- combat runtime 接口
- replication / protocol contract
- observability / stdio / E2E hooks
- supervision / service topology

特点：
- 不知道“具体这个游戏是什么玩法”
- 只知道“玩家 / NPC / actor / skill / snapshot / event”这些抽象

### B. Game Domain / Business Modules（业务层）

职责：
- 技能定义
- NPC 配置
- AI brain 实现
- 地图规则
- 刷怪规则
- 掉落表
- 任务系统
- 玩法规则（PVE/PVP/副本）

特点：
- 使用 kernel 暴露的行为接口
- 不碰底层 transport / cluster / authority plumbing

### C. Content / Data Layer（内容层）

职责：
- 数值配置
- entity template
- spawn points
- behavior tree / utility 参数
- loot table
- scene composition

特点：
- 尽量数据驱动
- 尽量不把纯配置写死在 actor 实现里

---

## 3.2 这三层在当前仓库里的映射

### 当前已经接近底座的部分

#### gate_server
更接近 **transport kernel**：
- tcp/udp
- fast-lane
- connection session
- protocol fanout
- stdio/observe

#### scene_server
更接近 **simulation kernel**：
- player actor
- npc actor
- movement
- combat
- aoi
- native adapters

#### auth_server
更接近 **identity/access kernel**：
- token issuance
- account/character ownership boundary
- demo seeds support

---

## 3.3 未来应如何拆成“底座 vs 业务”

### 推荐不是立刻大拆 app，而是先做“包内分层”

不要第一步就把工程拆成很多 umbrella app。更合理的是：

### Phase 1：在现有 app 内做明确分层

例如在 `scene_server` 内形成：

- `scene_server/kernel/`
- `scene_server/game/`

或者更直接一些：

- `scene_server/runtime/`
- `scene_server/domain/`
- `scene_server/content/`

当前已经有一些雏形：
- `movement/`
- `combat/`
- `npc/`
- `worker/`

下一步要做的是让这些目录从“实现拆分”继续升级成“框架接口边界”。

### Phase 2：把具体业务逻辑抽到单独 app 或 package

当边界稳定后，再考虑：

- `apps/game_demo`
- `apps/game_pve`
- `apps/game_content`

这时底座 app 可以继续保留：
- `gate_server`
- `scene_server`
- `auth_server`

而业务 app 只注册：
- skills
- brains
- entity templates
- spawn tables
- rule hooks

---

## 4. 当前项目最适合的抽象方式

## 4.1 不要抽象成“继承层”，而要抽象成“行为接口 + 注册表”

推荐的核心手法是：

### 用 Behaviour / Protocol / Registry 组织扩展点

例如：

#### Skill 行为接口
- `Game.SkillBehaviour`

定义：
- skill id
- validation
- target selection policy
- effect application command

#### NPC Brain 行为接口
- `Game.NpcBrainBehaviour`

定义：
- `decide(facts, profile, ctx) -> intent`

#### Scene Rule 行为接口
- `Game.SceneRuleBehaviour`

定义：
- on player join
- on npc spawn
- on actor death
- on tick hooks（谨慎）

#### Spawn Policy 行为接口
- `Game.SpawnPolicyBehaviour`

定义：
- 哪些点刷怪
- 刷怪条件
- respawn 规则

### 用 Registry 做装配

例如：
- `Game.Registry.skills`
- `Game.Registry.npc_brains`
- `Game.Registry.entities`
- `Game.Registry.scene_rules`

业务层向 registry 注册实现，底层通过 registry 查找行为模块。

这样做的好处：
- 高内聚
- 低耦合
- 易替换
- 不需要脚本语言也能实现“底座与业务解耦”

---

## 4.2 你现在最该抽的几个“硬边界”

### 边界 1：Skill System

当前 `Combat.Skill` 还是偏 demo/硬编码。应升级成：

- kernel 只定义 skill runtime contract
- business layer 提供 skill catalog

### 边界 2：NPC Brain / Attack / Template

当前已经比之前好多了，但还可以继续抽成：

- `Npc.Profile` -> content/template
- `Npc.Brain` -> business AI module
- `Npc.Actor` -> kernel runtime actor

### 边界 3：Scene Composition / Spawn Rules

目前 demo runner 直接 spawn training slime。未来应变成：

- map content
- spawn group config
- scene rule module

### 边界 4：Game Events / Hooks

要有统一的 domain event 概念，例如：
- actor_entered_scene
- actor_died
- actor_respawned
- skill_cast_resolved
- damage_applied

这样上层业务逻辑就可以挂在事件上，而不是直接侵入 actor 内部流程。

---

## 5. 推荐的最终技术路线

## 5.1 第一阶段：纯 Elixir/OTP 框架化

这是最推荐的。

做法：
- 继续保留当前 OTP actor 模型
- 抽象行为接口与 registry
- 把当前 demo 业务逻辑往上移
- 让底座只保留 runtime primitive

### 这是最适合当前仓库的做法
因为：
- 当前代码已经走在这条路上
- 风险最小
- 可测试性最好
- observability 保持最好

---

## 5.2 第二阶段：更多数据驱动

把这些逐步从硬编码挪到 content/data：
- npc profile
- skill definitions
- spawn tables
- map composition
- reward tables

这样“业务逻辑”进一步从代码转为“框架 + 配置 + 少量 hook 实现”。

---

## 5.3 第三阶段：如果仍然需要脚本，再引入脚本层

只有当出现这些明显需求时，再考虑 Lua / mlua：
- 强热更新需求
- 非工程人员也要写逻辑
- 任务/副本/活动脚本非常多
- 行为树/剧情脚本需要更强脚本化表达

### 即使引入，也建议：
- Lua 只输出 command / intent
- authority 仍在 Elixir actor
- 高频热路径仍在 Rustler / kernel

---

## 6. 与 KBEngine / Skynet 的相似与不同

## 相似点
我们完全可以实现类似的理念：
- infra / kernel 稳定
- game/business 可替换
- content/data 可配置
- actor/service 解耦

## 不同点
你不一定需要复制它们的具体技术路线。

### 你当前项目更适合：
- **OTP actor model 作为核心 runtime**
- **Rustler 只做性能热路径**
- **业务层用 Elixir module + registry + data-driven 扩展**

而不是：
- 让 Lua 成为第一驱动层
- 或者把高层逻辑硬塞进 NIF

---

## 7. 最推荐的落地顺序

### Step 1
把当前系统中的“kernel 公共抽象”定义出来：
- actor summary contract
- skill runtime contract
- npc brain contract
- scene rule contract
- domain event contract

### Step 2
把当前 demo 业务从 runtime 中抽离：
- Training Slime
- skill_id=1
- demo spawn rules
- demo scene composition

### Step 3
建立 `game_demo` 层或至少 `scene_server/game_demo/` 分层

### Step 4
把 profile/content 逐步数据驱动化

### Step 5
如果以后真的还需要脚本，再评估 Lua / mlua / 其他脚本层

---

## 8. 最终建议

### 你问：我们能不能像 KBEngine / Skynet 一样做“基础设施和业务逻辑分离”？

**能，而且应该这样做。**

### 你问：要不要直接用 mlua？

**不建议作为第一步。**

### 我对当前仓库的最佳建议是：

> **先把现有 Elixir/OTP + Rustler runtime 提炼成 kernel，使用 behaviour + registry + data-driven content 做业务解耦；Lua/mlua 只作为未来可选的脚本层，而不是现在的核心架构基石。**

这会比现在立刻引入 Lua 更稳、更可测试、更符合你当前仓库的演进方向。
