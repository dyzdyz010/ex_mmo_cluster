# 2026-04-13 固定 Tick 权威移动重构

## 背景

此前已经分析出最终位置偏差的系统性根因：

- 服务端把“输入包到达”当成了“推进模拟”的时钟
- 输入停止后，服务端与客户端都容易把 zero-input 当成一次事件，而不是持续控制状态
- 最终导致：
  - 本地最终位置
  - 服务端权威最终位置
  - 远端客户端看到的停点
  不能稳定收口

## 本轮重构目标

从架构层修正这个问题，而不是继续靠“最后补一补 stop 包”维持正确性。

## 重构后的核心原则

### 1. 服务端采用固定权威 tick 驱动 movement

`SceneServer.PlayerCharacter` 不再在收到每个 `MovementInput` 时直接执行一步运动积分。

现在改为：

- 输入到达时：
  - 只做校验
  - 只更新 `latched_input`
  - 返回 `:accepted`
- 真正的权威推进由：
  - `:movement_tick`
  - 固定 `fixed_dt_ms`
  驱动

这意味着：

- 输入 ≠ 模拟时钟
- 输入只是控制状态
- 模拟由服务端自己的权威 tick 推进

### 2. `latched_input` 作为持续控制状态

`PlayerCharacter` 现在维护：

- `latched_input`
- `last_input_seq`
- `last_ack_seq`
- `last_input_received_at_ms`

这样 zero-input 不再是“一次 stop 事件”，而是：

> 被服务端固定 tick 持续消费的控制状态

### 3. 输入超时后自动回退到 zero-input

如果一段时间没有新的输入包：

- 服务器会把有效输入退化为 zero-input / brake

这避免了：

- 丢包后沿用陈旧非零输入继续推进

### 4. 服务端即使“不再推进位置”，也会 ack 新输入

这是一个关键修正。

在角色已经停下时，如果客户端又发送了新的 zero-input 帧：

- 服务器即使不再推进 movement state
- 也会回一个新的 authoritative ack

这样客户端就能知道：

- 新的输入 seq 已经被服务器消费
- 当前停点已确认

否则客户端可能继续保留多余的本地预测输入，导致最终位置偏差。

### 5. 客户端 MovementAck 去重改为 `(ack_seq, auth_tick)` 二元判定

不再只靠 `ack_seq` 去重。

现在逻辑是：

- `ack_seq` 更大 => 接受
- `ack_seq` 相同但 `auth_tick` 更大 => 也接受

这为固定 tick 下“同一输入 seq 被多个权威 tick 消费”提供了正确语义。

## 关键改动点

### 服务端

- `apps/scene_server/lib/scene_server/worker/player_character.ex`
  - movement 输入从“收到即 step”改为“latch + fixed tick”
  - 新增 movement timer
  - 新增 `latched_input`
  - 新增 `last_ack_seq`
  - 新增 zero-input timeout 回退
  - 新增“无位移推进但仍 ack 新输入”逻辑

- `apps/gate_server/lib/gate_server/worker/tcp_connection.ex`
  - movement 输入支持 `:accepted`
  - movement ack 改为支持异步 push

- `apps/gate_server/lib/gate_server/worker/udp_acceptor.ex`
  - UDP movement 输入支持 `:accepted`

### 客户端

- `clients/bevy_client/src/net.rs`
  - MovementAck 去重规则升级为 `(ack_seq, auth_tick)`
  - 新增测试：同一 `ack_seq` 但更新 `auth_tick` 的 ack 也会被接收

## 验证结果

### 测试

- `mix test`

全部通过。

结果：

- client / server / fast-lane / bot 联调通过

  - `0.00742579281980047`

这说明：

- 最终停点已经重新收口
- 且这次不是仅靠 stop-sync 的表面补丁，而是因为服务端权威推进模型已经改成固定 tick + latched input

## 当前结论

这轮之后，移动系统在架构层已经从：

- **packet-driven stepping**

转为：

- **authoritative fixed-tick stepping with latched input state**

这是本轮最核心的系统性修正。
