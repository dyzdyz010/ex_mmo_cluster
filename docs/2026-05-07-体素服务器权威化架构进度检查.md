# 2026-05-07 体素服务器权威化架构进度检查

本文记录当前体素链路相对下一阶段目标的真实进度和架构缺口。目标不是给局部代码打补丁，而是明确哪些基础设施必须先成为服务端权威链路的一部分，避免客户端和服务端场景继续分叉。

## 当前目标

1. 体素全面服务器权威化：所有 confirmed voxel truth 只能来自服务端 snapshot、delta、object/catalog/result，不允许遗漏其他本地写入路径。
2. 网页客户端左右键增减体素体能力完善：macro cell 和 prefab/refined micro cell 一视同仁，都可以摆放和破坏。
3. 准备为体素格填充属性：第一步覆盖温度、湿度、密度和导热性，并为后续燃烧、冻结、结构、魔法传播留出权威规则空间。

## 当前真实进度

当前主线已经达到 **server-authoritative macro voxel MVP**：

- 网页客户端默认 `voxel_sync=server-authoritative`，在线模式下 `WorldStore` 只应用服务端 `ChunkSnapshot / ChunkDelta / VoxelIntentResult`。
- `VoxelImpactIntent` 已能提交整宏格放置和破坏。
- `PrefabPlaceIntent` v1 已能把服务端 catalog 中的 macro blueprint 放到场景里。
- dev seed、中心 chunk 订阅、HUD / CLI / observe 验证面已经可用。

但这还不是全面权威化：

- refined micro 数据仍主要是浏览器离线路径能力。
- 服务端 snapshot 协议虽然定义了 `RefinedCells / AttributeSets / TagSets / ObjectRefs / EnvironmentSummaries` section，但 refined/catalog 目前仍是空池或预留字段。
- Gate 对 `VoxelImpactIntent` 的语义仍是 macro-only：`impact_kind == 0` 表示破坏，非 0 表示 material id 并写整宏格。
- `PrefabPlaceIntent` v1 是逐 macro cell 写入，失败不回滚，不具备跨 chunk 原子性。
- 属性链路只有温度/湿度的局部字段和环境摘要雏形，密度/导热性还没有类型化属性目录、存储和 delta 规则。

## 目标一：全面服务器权威化缺口

### 缺口 A：canonical truth 还没有覆盖所有 cell mode

设计文档要求 `ChunkSnapshot` 包含：

- `MacroHeaders`
- `NormalBlocks`
- `RefinedCells`
- `AttributeSets`
- `TagSets`
- `EnvironmentSummaries`
- `ObjectRefs`

当前只有 macro headers、normal blocks 和部分 environment summaries 真正进入 encode/decode 热路径。refined cells、attribute sets、tag sets、object refs 仍没有完整类型和 wire payload。

需要补齐：

- `RefinedCellData` 服务端 domain module，包括 occupancy mask、micro material、micro state、micro part、owner object provenance。
- `AttributeSet` / `TagSet` 服务端 domain module 和 catalog snapshot/patch。
- `ChunkObjectRef` 从摘要结构升级为可支持 object/part/provenance 查询的权威引用层。
- chunk hash / cell hash 覆盖 refined、attribute、tag、environment、object ref，不能只覆盖 normal block。
- DataService 持久化必须保存同一份 canonical snapshot，不允许只保存 macro 子集。

### 缺口 B：离线路径需要硬隔离

浏览器离线 microgrid/refined prefab 能力是有价值的调试面，但它不能在在线模式中伪装成 MMO 权威真相。

需要保持的边界：

- `VITE_VOXEL_SYNC=offline` 才允许本地写 confirmed voxel storage。
- 在线模式下所有编辑只能发 intent；本地只能做 preview / ghost / pending UI。
- CLI / HUD / observe 必须继续显示 `voxel_sync`、`voxel_truth_source`、last snapshot/delta/result 和 edit rejection reason。
- 回归测试要覆盖在线 adapter 不调用本地 refined mutator。

### 缺口 C：跨 chunk 一致性还不是事务模型

当前 Gate 的 prefab v1 是 cell-major loop：每个 cell 单独 route chunk 并调用 `ChunkDirectory.apply_intent`。如果前面写成功、后面失败，已写入部分不会回滚。

这会直接破坏“不要遗留小尾巴导致场景不一致”的目标。

需要把 prefab/build/edit 多 cell 写入改为：

1. Gate 只做协议 decode、鉴权状态检查和请求转发。
2. World 负责确定涉及的 chunks、lease、parcel/build permission、transaction id。
3. Scene `BuildTransactionApplier` 对各 chunk prepare。
4. 全部 prepare 成功后 commit；任一失败则 abort 已 prepared chunk。
5. `VoxelIntentResult` 返回 authoritative summary，客户端只消费 result + delta/snapshot。

## 目标二：macro 与 prefab micro 一视同仁的编辑缺口

### 缺口 A：服务端没有 refined 写操作

当前 Scene 支持：

- `put_solid_block`
- `break_block`

下一阶段至少需要：

- `put_micro_block`
- `clear_micro_block`
- `merge_refined_cell`
- `split_solid_macro_to_refined`
- `clear_refined_cell_if_empty`
- 可选的 `collapse_full_refined_to_solid`

这些操作必须在 Scene 侧完成冲突检查和版本推进，并通过 `CellRefined` delta 下发。

### 缺口 B：intent 需要类型化，不能继续挤在 impact_kind 里

macro-only 的 `VoxelImpactIntent` 可以保留为兼容路径，但下一阶段应该引入或升级为 typed edit intent：

```text
VoxelEditIntent {
  request_id
  logical_scene_id
  actor_cid
  action              // place | break | damage | replace | attribute_patch
  target_granularity  // macro | micro | object_part
  target_world_micro
  face_normal
  material_id?
  blueprint_ref?
  object_ref?
  part_ref?
  attribute_patch_ref?
  expected_chunk_version?
  expected_cell_hash?
}
```

这样左键/右键可以统一表达：

- 命中普通 macro：破坏 macro 或在邻接 macro 放置。
- 命中 refined micro：破坏 micro slot 或在邻接 micro slot 放置。
- 命中 prefab part：按 owner object + part 结算局部破坏。
- 选中 prefab：提交 blueprint/object placement，而不是客户端先改 chunk truth。

### 缺口 C：prefab server rasterizer 仍是 macro blueprint

服务端 `PrefabRaster` v1 只把 blueprint cell 写成 `NormalBlockData`。要支持 refined prefab，需要 rasterizer 输出 per-chunk patch：

```text
PrefabRasterPatch {
  chunk_coord
  local_macro
  refined_cell_patch
  object_ref_patch
  attribute_set_refs
  tag_set_refs
}
```

同时要补：

- rotation 对 macro offset 和 micro occupancy 的一致变换。
- boundary snap / socket snap 的服务端校验版本。
- overlap check 基于服务端 canonical chunk truth。
- prefab instance / object instance 的 id 分配和 covered chunks 记录。

### 缺口 D：局部破坏需要 provenance

**状态：已实现（2026-05-13 verifier 审计通过）**。本段保留作为历史背景，实际落地能力见 README Phase 4 / 4-bis / A4 / A4-4 段。

浏览器本地已有 `microPartIds` 概念，但文档已经指出它不足以支持嵌套 prefab 和多 object 共处同一 macro。服务端需要每个 occupied micro slot 能反查：

- owner object id
- assembly id 或可反查 assembly 的压缩引用
- part id
- part tags / affordances
- runtime state，例如 health、damage mask、open/closed、burning、ownership

没有 provenance，左键破坏 prefab 微格只能变成“清一个 slot”或“清整个 macro”，无法进入后续战斗、魔法、资源和结构系统。

## 目标三：温度、湿度、密度、导热性属性缺口

### 当前已有基础

- `NormalBlockData` 已有 `temperature_delta` 和 `moisture_delta`。
- `NormalBlockData` 已有 `attribute_set_ref` 和 `tag_set_ref`。
- `MacroEnvironmentSummary` 已有 default/current temperature 和 moisture。
- Web material catalog 已能把 temperature/moisture delta 用于表现色调。

### 还缺的基础设施

### 缺口 A：属性目录

需要明确 attribute catalog 的第一版 schema：

```text
AttributeDefinition {
  id
  name
  unit
  value_type        // i16 | u16 | fixed32 | enum | bitset
  default_value
  min_value
  max_value
  merge_rule        // override | add_delta | max | min | material_default
  dynamic?          // true 表示运行时可变
}
```

第一批建议：

| 属性 | 建议归属 | 原因 |
| --- | --- | --- |
| temperature | dynamic environment/current field | 会随规则帧扩散和变化。 |
| humidity / moisture | dynamic environment/current field + material response | 会受环境、液体、材料吸湿影响。 |
| density | material/static attribute，必要时 cell override | 更像材质/结构参数，不应每帧扩散。 |
| thermal_conductivity | material/static attribute，参与 heat diffusion | 决定热传播，不是当前温度本身。 |

### 缺口 B：属性存储粒度

必须先决定每类属性写在哪里：

- material default：材质目录默认值，例如 density、thermal conductivity。
- normal block override：单个 macro block 的局部覆盖。
- refined micro override：单个 micro slot 的局部覆盖。
- object/part attribute：prefab part 或 object runtime 状态。
- environment summary：宏格聚合后的当前温度/湿度。

如果不先定粒度，后续会出现同一属性同时写在 material、block、environment、object 四处而没有优先级。

### 缺口 C：模拟器和边界传播

温湿度不是单次 edit intent 的附属字段，而是服务器规则帧的状态更新。需要 Scene 侧体素状态模拟基础设施：

- dirty cell / dirty bounds 标记。
- 每 tick 或低频规则帧处理温度和湿度扩散。
- chunk 边界事件，跨 chunk 传播时带 lease / owner_epoch fence。
- deterministic hash，用于回归验证相同输入产生相同 delta。
- `EnvironmentUpdated` / `CatalogPatch` delta 下发。

## 不建议走的补丁路线

- 不要在 `OnlineVoxelWorldAdapter.placeMicroBlock` 里直接调用本地 `WorldStore`，这会破坏在线 confirmed truth。
- 不要把 micro slot 编进 `impact_kind` 或临时复用 material id，后续 object part、attribute patch、damage intent 都会被挤爆。
- 不要只在客户端启用 refined prefab 热栏，服务端不认识的 blueprint 会继续制造表现和权威不一致。
- 不要在 Gate 里继续扩展 prefab cell loop，跨 chunk 原子性必须由 World/Scene transaction 基础设施承载。
- 不要把密度、导热性直接塞进 `NormalBlockData` 固定字段；更适合走 attribute catalog 和 material defaults。

## 建议实施顺序

### Phase 1：权威 truth contract 收口

交付物：

- `RefinedCellData` 服务端类型、TS 类型、binary codec。
- `AttributeSet` / `TagSet` 空间和 catalog patch 的最小类型。
- snapshot / delta golden fixtures，覆盖 macro、refined、environment、attribute/tag refs。
- `chunk_hash` / `cell_hash` 覆盖全部 truth section。

验收：

- Elixir codec roundtrip。
- TS decoder roundtrip。
- 相同 fixture 的服务端和客户端 hash 一致。
- 在线 Web CLI 可读取 refined cell，但不能本地伪造写入。

### Phase 2：服务端 refined micro edit

交付物：

- Scene storage refined mutation API。
- `CellRefined` delta encode/decode/apply。
- typed edit intent 第一版。
- CLI：`micro_cell` 读取、`micro_place` / `micro_break` 提交服务端 intent。

验收：

- 左键破坏 refined micro slot 后，只移除目标 slot。
- 右键放置 refined micro slot 后，snapshot/delta 回推可读。
- 在线模式刷新后状态来自服务端持久化，不来自浏览器 local storage。

### Phase 3：prefab v2 事务化

交付物：

- 服务端 refined blueprint catalog。
- Prefab raster 输出 refined patches + object refs。
- World transaction coordinator 接入 Gate prefab path。
- prepare/commit/abort observe events。

验收：

- 跨 chunk prefab 要么全部落地，要么全部不落地。
- 中途 lease/route 失败不会留下部分 chunk 小尾巴。
- 浏览器右键 prefab、CLI prefab_place、服务端 observe 三条入口结果一致。

### Phase 4：object provenance 与局部破坏

**状态：已实现（2026-05-13 verifier 审计通过）**。详见 `apps/scene_server/lib/scene_server/voxel/audit/2026-05-13-phase4-verifier-audit.md`。已落地能力以 `apps/scene_server/lib/scene_server/voxel/README.md` Phase 4 / 4-bis / A4 / A4-4 段为权威记录。

**Phase 4 未关闭的 gap（不阻塞验收，列入 Phase 5 / A4-bis-cluster 跟踪）**：

1. damage dispatch `Task.start` 丢弃路径缺少 dropped-damage observe counter（chunk_process.ex 异步 dispatch best-effort）。
2. owner-lookup cold-start 退化 `%{owner_key => covered_chunks}`，A4-bis-cluster 加 `MapLedger.region_for_chunk` 后退役。
3. PartState health 初始值统一用 ratio=1.0，Phase 5 引入 `PartDefinition.default_health_ratio` 协议字段后改 per-part。
4. e2e 测试用 `assert_eventually` 1s deadline，偶发 CI 慢机器 transient flake（15+ 次连续通过未复现）。

交付物：

- `microOwnerObjectIds` 或等价压缩结构。
- ObjectInstance / PartRuntimeState 最小模型。
- object_at / part_at / object_damage_preview CLI。
- 左键命中 prefab part 时提交 object-part damage/break intent。

验收：

- 命中同一 macro 内不同 prefab part 能得到不同 owner object / part。
- 局部破坏只影响目标 object/part 的 slots。
- delta 中包含 geometry 变化和 object state 变化。

### Phase 5：属性目录和温湿度基础模拟

交付物：

- AttributeCatalogSnapshot / CatalogPatch 最小实现。
- density、thermal_conductivity material defaults。
- temperature、humidity/moisture environment current field。
- Scene 低频规则帧，对 dirty cells 生成 `EnvironmentUpdated` delta。

验收：

- CLI 可读 cell effective attributes。
- 修改材质或环境后，服务端生成确定性 environment delta。
- 客户端只根据权威 delta 更新表现，不本地计算 confirmed 温湿度。

## 回归验收口径

每个阶段完成后，都需要同时覆盖：

- 服务端测试：Scene storage / codec / transaction / DataService persistence。
- Gate 协议测试：WS 与 TCP 同语义。
- Web 单测：decoder、online adapter、CLI、world edit controller。
- 浏览器 smoke：HUD/CLI 显示 `voxel_sync=server-authoritative`，编辑结果通过 snapshot/delta/result 读回。
- observe 日志：记录 request id、actor、target、chunk list、transaction id、result code、conflict reason。

## 当前结论

下一步不要先做“让按钮能点”的 UI 补丁。正确的第一块地基是 **服务端 refined cell + typed edit intent + CellRefined delta**。这三者完成之前，macro 与 prefab micro 一视同仁只会停留在客户端离线能力，不能成为 MMO 权威链路。
