## 1. 定位与职责边界

UE5 `test1` 项目已交付：

- 量化参数 `MicroPerMacro=4`，`ChunkSizeInMacro=16³=4096`
- `FChunkStorageData` / `FNormalBlockData` 等数据结构
- 三态单元 `EVoxelCellMode::Empty | SolidBlock | Refined`
- 客户端 `UProceduralMeshComponent` 渲染
- 单机本地编辑（破坏 / 放置）

UE 项目**明确排除**：网络同步、多人编辑协作、服务端持久化、Prefab 跨账号共享。
ex_mmo_cluster 本期定位即补齐这层"**权威 + 广播 + 持久化 + 冲突解决**"。

## 2. 量化约定（与 UE 端严格一致）

| 参数 | 值 | 说明 |
|------|----|------|
| `MicroPerMacro` | 4 | 单 Macro 内 Micro 排列 4×4×4 |
| `ChunkSizeInMacro` | 16³ = 4096 | 单 Chunk 宏观单元总数 |
| `FNormalBlockData` | 10 bytes | 与 UE 结构体布局一致 |
| `FChunkCoord` | 3 × i32 | 有符号，允许负象限 |
| `FMacroCoord` | 3 × u8 | 0..15 |
| `FMicroCoord` | 3 × u8 | 0..3 |
| Chunk 非 Refined 上限 | ≈ 40 KB | 4096 × 10 B |
| Chunk 全 Refined 上限 | ≈ 2.6 MB | 4096 × 64 × 10 B |

字节序：**小端**，与 UE 保持一致，避免客户端解码负担。

## 4. 数据层

### 4.1 PostgreSQL Schema（在 `data_service` 中落地）

```text
voxel_chunks
  id              bigserial PK
  coord_x         int       NOT NULL
  coord_y         int       NOT NULL
  coord_z         int       NOT NULL
  schema_version  int2      NOT NULL  -- 与 UE FChunkStorageData::Version 同步
  flags           int4      NOT NULL  -- bit0: has_refined
  data            bytea     NOT NULL  -- FChunkStorageData 序列化
  updated_at      timestamptz NOT NULL
  UNIQUE (coord_x, coord_y, coord_z)

voxel_prefabs
  id              bigserial PK
  name            text      NOT NULL
  owner_id        bigint    NULL       -- NULL = 系统内置
  bbox_macro      int4[3]   NOT NULL
  data            bytea     NOT NULL   -- 占用掩码 + macro/micro 分布
  visibility      int2      NOT NULL   -- 0=private 1=guild 2=public
  created_at      timestamptz NOT NULL

voxel_edit_log
  id              bigserial PK
  chunk_id        bigint    NOT NULL REFERENCES voxel_chunks(id)
  actor_id        bigint    NOT NULL
  op_kind         int2      NOT NULL   -- 1=break 2=place 3=prefab_place
  payload         bytea     NOT NULL
  applied_at      timestamptz NOT NULL
  INDEX (chunk_id, applied_at DESC)
```

设计要点：

- Chunk 本身按二进制整块写入，利用 PostgreSQL TOAST 自动压缩
- `edit_log` 仅记录权威结果，不用于回放重建（重建以 `voxel_chunks.data` 为准）
- 审计/封禁/回滚场景才读取 `edit_log`，索引只覆盖 `chunk_id + applied_at`

### 4.2 Elixir 侧二进制编解码（与 UE 字节级一致）

```
FChunkStorageData 线格式
  u16  version
  u16  flags              -- bit0: has_refined
  u8[1024] cell_mode_bitmap  -- 4096 cells × 2 bits
  u32  refined_header_count
  refined_header[] {
    u16 macro_index        -- 0..4095
    u64 micro_solid_bitmap -- 64 bits
  }
  u32  block_record_count
  block_record[] {
    u16 cell_index         -- macro_index × 64 + micro_index 或 macro_index（非 refined）
    u8  block_id
    u8  flags
    u16 reserved
    u32 extra              -- 与 UE FNormalBlockData 对齐
  }
```

codec 与 `GateServer.Codec` 分层放置：`VoxelServer.Codec.Chunk` 专注 Chunk；`GateServer.Codec` 只负责 opcode + 长度封装。

## 5. 协议扩展

在 `docs/2026-04-10-线协议规范.md` 的 opcode 编号空间中保留 `0x60..0x6F` 给体素系统：

| opcode | 方向 | 名称 | 载荷要点 |
|--------|------|------|----------|
| `0x60` | C→S | ChunkSubscribe | `FChunkCoord` |
| `0x61` | C→S | ChunkUnsubscribe | `FChunkCoord` |
| `0x62` | S→C | ChunkSnapshot | coord + `FChunkStorageData` |
| `0x63` | S→C | ChunkDelta | coord + `u8 op_count` + ops[] |
| `0x64` | C→S | BlockBreak | coord + macro + micro? + `u32 base_hash` + `u32 seq` |
| `0x65` | C→S | BlockPlace | coord + macro + micro? + `u8 block_id` + base_hash + seq |
| `0x66` | C→S | PrefabCreate | bbox + 掩码 + 名称 |
| `0x67` | C→S | PrefabPlace | prefab_id + origin + yaw + seq |
| `0x68` | S→C | EditAck | seq + result_code + authoritative_cell? |
| `0x69` | S→C | ChunkInvalidate | coord（服务端主动驱逐订阅） |

`micro?` 表示 `u8 present_flag` + 可选 `FMicroCoord`，对齐 UE 的"Macro 整体" vs "Micro 精修"两种编辑粒度。

## 6. 进程模型：ChunkProcess

**一个 Chunk = 一个 GenServer**。通过 `Horde.Registry` 在集群内注册，key 为 `{:chunk, {cx, cy, cz}}`。

### 6.1 状态

```elixir
%ChunkProcess.State{
  coord:          {cx, cy, cz},
  storage:        %Chunk.Storage{},  # UE 结构体的 Elixir 镜像
  subscribers:    MapSet.t(player_id),
  pending_ops:    :queue.new(),      # 批量落盘缓冲
  dirty_since:    integer | nil,
  last_persist:   integer,
  schema_version: pos_integer
}
```

### 6.2 生命周期

- **启动**：`ChunkProcess.ensure_started/1` 由 `VoxelServer.Aoi` 订阅时触发；若 PostgreSQL 有记录则加载，否则用 UE 约定的"Empty 全空"初始化
- **落盘策略**：
  - 批量阈值：累计 ≥ 32 条 op 或 `dirty_since` 距今 ≥ 30 s
  - 使用 `poolboy` 借用 `DataService` connection 写入
- **终止**：5 min 无订阅者且 `dirty_since == nil` → 自动 terminate
- **崩溃恢复**：`chunk_sup` 为 `:one_for_one`；重启后从 PG 重载

### 6.3 命名与路由

```elixir
via = {:via, Horde.Registry, {VoxelServer.ChunkRegistry, {:chunk, coord}}}
GenServer.call(via, {:apply_op, op})
```

不硬编码节点；`BeaconServer.Client.lookup(:voxel_server)` 返回的节点列表供客户端请求分流（未来可按 chunk hash 选节点，此期只在单节点落地）。

## 7. 并发编辑冲突策略

玩家 A / B 同时编辑同一 micro：

1. 客户端本地预测并带 `base_hash`（基于该 macro 当前 `cell_mode + micro_bitmap + block_id` 的 `xxhash64` 低 32 位）
2. `ChunkProcess` 收到 op：
   - `base_hash` 匹配 → 接受、广播 `ChunkDelta`、ack `seq` = applied
   - `base_hash` 不匹配 → 拒绝，回 `EditAck(seq, conflict, authoritative_cell)`
3. 客户端收到 `conflict` 后回滚预测、采用权威状态

此策略等价于轻量 OT：**不合并，拒绝并以权威覆盖**。后续若冲突率 > 3% 再考虑升级到 CRDT/OT。

## 8. AOI 集成

体素 AOI 与实体 AOI 分离：

- **实体 AOI**（已有）：`scene_server.AoiManager`，半径/视锥剔除
- **体素 AOI**（新增）：`VoxelServer.Aoi`，订阅集合 `{c : ‖c - player_chunk‖∞ ≤ R_voxel}`

订阅 diff：

```elixir
def on_player_moved(player_id, from_chunk, to_chunk) do
  R = @voxel_aoi_radius
  old = chunks_within(from_chunk, R)
  new = chunks_within(to_chunk, R)
  Enum.each(MapSet.difference(old, new), &unsubscribe(player_id, &1))
  Enum.each(MapSet.difference(new, old), &subscribe(player_id, &1))
end
```

默认 `R_voxel = 3`（7³ = 343 chunks），后续根据 V-6 压测调整。

## 9. Rust NIF：voxel_core crate（V-6 落地）

下沉到 Rust 的高频计算：

| 函数 | 用途 |
|------|------|
| `occupancy_mask_of_macro(micro_bitmap) -> u64` | 渲染辅助，批量计算 |
| `neighbor_signature(side_bytes) -> u64` | Chunk 边界缝合 hash |
| `pack_prefab(macros) -> bytes` | Prefab 紧凑打包 |
| `unpack_prefab(bytes) -> macros` | Prefab 解包 |
| `apply_ops_batch(storage, ops) -> storage'` | 可选：批量 op 应用（当 Elixir 版跑不满 10 Hz 时启用） |

crate 布局与 `movement_core` / `octree` 对齐；`Rustler 0.36` + `#[rustler::nif]` + `#[rustler::resource_impl]`。

## 10. 实施阶段

| 阶段 | 周期 | 交付 |
|------|------|------|
| V-0 类型与 codec | 1 周 | `%Chunk.Storage{}`、`VoxelServer.Codec.Chunk`、gate_server opcode stub、codec round-trip 测试 |
| V-1 PostgreSQL 落盘 | 1 周 | schema/migration、`VoxelChunk`/`VoxelPrefab`/`VoxelEditLog` Ecto schema、`VoxelServer.DbOps` CRUD |
| V-2 ChunkProcess | 2 周 | DynamicSupervisor + Horde Registry、惰性加载、批量落盘、基础订阅/广播 |
| V-3 Edit 流 + 冲突 | 2 周 | `BlockBreak`/`BlockPlace` 全链路、`base_hash` 校验、客户端回滚协议、压力测试 100 player × 1 chunk |
| V-4 AOI | 1.5 周 | `VoxelServer.Aoi` worker、跨 Chunk diff、gate_server 合流、卸载路径 |
| V-5 Prefab | 1.5 周 | `PrefabCreate`/`PrefabPlace`、三级可见性、运行时 instancing 缓存 |
| V-6 NIF + 基准 | 2 周 | `voxel_core` crate、criterion + benchee 双端压测、与移动基线合并发布 |

总单人工作量约 **11 周**；V-2 与 V-1 可部分并行，V-4 与 V-5 可交错，实际压缩到 **7-8 周**。

## 11. 开放问题与风险

1. **边界缝合权威**：UE 端在客户端做 neighbor_signature 拼接，服务端是否缓存？初版存 signature、不主动解决，压测后决策
2. **Chunk 进程数爆炸**：活跃 Chunk 10k+ 时 Horde 的 handoff 延迟需实测；必要时按 cluster-wide shard 切分
3. **Schema 演进**：`schema_version` 字段保留向后软降级；迁移通过 `ChunkProcess.init` 惰性触发
4. **冲突率**：依赖 V-3 埋点；> 3% 则升级到 OT；< 1% 保持当前"拒绝覆盖"策略
5. **UE 端网络层缺失**：本规划默认 UE 侧会补 `UnrealNet` 客户端，接入 `VoxelServer.Codec`；若未补则 V-2 前需先补 UE 端 `UVoxelNetSubsystem`（**不在本仓库范围**）
6. **物理破坏**（爆炸/连锁）：延后到独立周期，不在 V-0..V-6 范围内
7. **Mod 方块 ID 扩展**：V-6 之后再考虑，当前 `block_id: u8` 保留 `0x00..0xEF` 给系统，`0xF0..0xFF` 给未来扩展槽

## 12. 验收标准（V-6 末）

- 1 个节点承载 ≥ **1000 active Chunks** 且空闲 CPU > 40%（对应 ≈ 40 MB Chunk 数据驻留 + Aoi 广播）
- 100 玩家并发编辑同一 Chunk，`EditAck` p99 < 50 ms
- 单 Chunk 从 PG 冷加载 → 推送到客户端 p95 < 200 ms
- 服务端崩溃重启后 Chunk 状态无损（PG 落盘保证）
- 与 `docs/2026-04-20-movement-perf-baseline.md` 合并为统一性能报告
