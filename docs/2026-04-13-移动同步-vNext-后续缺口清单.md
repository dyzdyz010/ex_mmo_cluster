# 2026-04-13 移动同步 vNext 后续缺口清单

本文档用于记录 **已经识别、当前刻意暂缓** 的移动同步后续工作，避免这些缺口只存在于对话上下文里而在后续开发中丢失。

当前系统已经完成：

- 本地 `MovementInput -> MovementAck`
- 本地预测 + 基础 reconciliation
- 远端 `RemoteSnapshot`
- 服务端 Rustler movement engine 主路径
- stdio + observe E2E 验证

以下内容原本是 deferred gap。  
截至 `2026-04-13`，其中前三项已经完成第一轮系统性落地，本文转为记录：

- 已完成到什么程度
- 还剩哪些更深层的增强项

---

## 1. rollback / replay 窗口治理（第一轮已完成）

### 当前状态
- 已新增：
  - `ReplayGovernance`
  - `ReplayGovernanceStats`
  - `ReplayAction`
- 已支持：
  - soft correction
  - hard snap
  - replay window trim

### 仍可继续增强
- replay 统计接入 HUD / observe
- 长时间公网环境下的 replay 水位治理
- correction 频率退化策略

### 主要影响
- 在高延迟、抖动、丢包环境下，replay 行为还不够“工业化”
- 未来加入复杂运动模式后，history/replay 管理成本会上升

### 推荐落点
- `clients/bevy_client/src/world/local_player.rs`
- `clients/bevy_client/src/sim/history.rs`
- `clients/bevy_client/src/sim/reconcile.rs`

### 推荐实现方向
- 为 `LocalPredictionRuntime` 增加：
  - replay window 限制
  - correction 计数器
  - history 水位指标
- 将 correction 分为：
  - soft correction
  - hard correction
- 增加 observe 字段：
  - replayed_frames
  - correction_distance
  - replay_window_size

---

## 2. 远端 snapshot buffer + 插值窗口（第一轮已完成）

### 当前状态
- 已从单 latest snapshot 升级到：
  - snapshot ring buffer
  - interpolation delay
  - buffered sampling
  - 有界 extrapolation

### 仍可继续增强
- 更强的 velocity-aware / Hermite 插值细化
- 更完整的全局远端时间基治理
- 更细的 jitter 适应策略

### 主要影响
- 现在的远端表现已经比之前好，但仍属于轻量级实现
- 在远端更新频率更低或抖动更明显时，观感还有继续优化空间

### 推荐落点
- `clients/bevy_client/src/world/remote_player.rs`
- 后续建议新增：
  - `clients/bevy_client/src/presentation/interpolation.rs`

### 推荐实现方向
- 建立每个远端玩家的 snapshot ring buffer
- 引入：
  - interpolation delay
  - playback timestamp / cursor
- 后续可升级到：
  - velocity-aware interpolation
  - Hermite interpolation

---

## 3. camera / animation 更彻底绑定 presentation state（第一轮已完成）

### 当前状态
- 已新增独立 presentation 模块：
  - `presentation/smoothing.rs`
  - `presentation/camera.rs`
  - `presentation/animation.rs`
- 当前 app 已通过 presentation helpers 驱动：
  - smoothing
  - camera target
  - velocity-derived visual animation scale

### 仍可继续增强
- 抽出更独立的 presentation state resource
- 更正式的 animation input pipeline
- UE5 locomotion 对接前的 contract 固化

### 主要影响
- 对当前 Bevy demo 阶段不是硬阻塞
- 对以后复杂角色表现、动画状态机、UE5 迁移很重要

### 推荐落点
- `clients/bevy_client/src/app.rs`
- 后续建议新增：
  - `clients/bevy_client/src/presentation/camera.rs`
  - `clients/bevy_client/src/presentation/animation.rs`

### 推荐实现方向
- 明确拆分：
  - predicted state
  - authoritative state
  - presentation state
  - camera target
  - animation input
- camera / animation 只读取 presentation side 的数据

---

## 4. Rustler / 客户端 movement profile 参数一致性治理

### 当前状态
- 服务端 authoritative step/replay 已使用 Rustler
- 客户端仍保留 Rust 本地 profile + predictor

### 目前缺什么
- 没有显式的“客户端 / 服务端 profile 一致性治理点”
- 后续如果 profile 参数变化，可能出现两端行为漂移风险

### 推荐落点
- `clients/bevy_client/src/sim/profile.rs`
- `apps/scene_server/lib/scene_server/movement/profile.ex`
- `apps/scene_server/native/movement_engine/src/types.rs`

### 推荐实现方向
- 统一 profile 的字段清单
- 增加 profile parity 测试
- 必要时增加 profile version / hash

---

## 5. 何时应该优先继续补这些缺口

如果出现以下情况，就建议把这些 deferred gap 提升优先级：

1. 公网延迟 / 抖动测试开始明显暴露 correction 问题
2. 远端玩家观感成为实际产品问题
3. 开始接复杂动画或 locomotion
4. 准备 UE5 客户端迁移
5. movement profile 频繁调参，开始担心两端漂移
