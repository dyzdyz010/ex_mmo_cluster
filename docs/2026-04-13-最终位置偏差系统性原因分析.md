# 2026-04-13 最终位置偏差系统性原因分析

## 1. 结论

“移动一段时间后，两边最终位置开始偏差”的问题，**根因不是少发了一帧 stop 包这么简单**。  
之前的 stop-sync 修复能把症状压住，但它本质上是在弥补当前移动架构里的一个更深层问题：

> **当前系统把“输入包到达”当成了“推进模拟”的时钟，而不是把输入当成一个在固定权威时钟上采样的控制状态。**

在没有 acceleration / jerk 时，这个问题不容易暴露；一旦运动模型变成“需要多帧减速才能停下”，这个架构缺口就会表现为最终位置收不齐。

---

## 2. 为什么之前那个 stop-sync 修复只是缓解，不是根治

之前的问题表现是：

- 客户端方向归零后，只发一次 zero-input / brake 帧
- 但角色此时并没有真正停下
- 因为 acceleration / jerk 限制下，“停止”不是事件，而是一个需要持续推进的减速过程

所以 stop-sync 修复的本质是：

- 继续发 zero-input / brake 帧
- 直到速度真的收敛

这个修复是有效的，但它更像：

> **用持续补帧来弥补“模拟推进时钟绑定在输入发包上”的结构缺陷**

也就是说：
- 它是对当前架构的正确补救
- 但不是最终理想形态

---

## 3. 系统性原因

## 3.1 模拟时钟绑在输入包到达上，而不是权威固定 tick

当前权威路径在：

- `apps/scene_server/lib/scene_server/worker/player_character.ex`

里，`{:movement_input, frame}` 到达时才会：

1. `sanitize_input_frame`
2. `Engine.step(...)`
3. 更新 SceneOps
4. 回 `MovementAck`

也就是说：

- **服务端并不是按自己的固定权威 tick 持续模拟**
- 而是**输入包来了才前进一步**

这会导致两个问题：

1. 当输入停止时，模拟也跟着停住，除非客户端继续补 zero-input 帧
2. acceleration / jerk 这种本应由固定时间线自然演化的状态，被错误地建模成“收到几个输入包就演化几次”

### 本质问题

当前系统把：

- “输入帧”  

同时当成了：

1. **控制状态**
2. **模拟推进指令**

这两个语义不应该绑死在一起。

---

## 3.2 客户端和服务端并没有共享同一个真正的推进时间线

客户端本地预测：

- 在 `clients/bevy_client/src/net.rs`
- 收到 `MoveInputSample` 时立刻 `apply_local_input`
- 预测时用的是客户端发送的 `dt_ms`

服务端权威路径：

- 在 `player_character.ex`
- `sanitize_input_frame` 里会把 `dt_ms` 改成：
  - `now_ms - last_server_input_at_ms`
  - 再 `clamp_server_dt`

这说明：

- 客户端用的是“本地设定 tick”
- 服务端用的是“包到达间隔的近似值”

### 结果

即使输入方向完全一样，两边积分也不是严格跑在同一条时间轴上。  
在连续移动时，这个差异会被频繁 ack/reconcile 掩盖；  
在停止阶段，如果没有继续推进直到静止，就会把差异留在最终停点。

### 本质问题

当前系统还是：

> **client time-stepping + server arrival-time stepping**

而不是：

> **shared logical tick + authoritative fixed-step sim**

---

## 3.3 “停止”被建模成一次事件，而不是一个持续控制状态

在现有系统里，方向变成零时，客户端之前的做法是：

- 发一帧 brake
- 然后认为“停止命令已经发完了”

但在 acceleration / jerk 模型下：

- 停止不是“按钮事件”
- 停止是“持续保持 zero-input 控制状态，直到速度衰减到 0”

所以从系统建模上说，之前的问题不是“少同步了最终位置”，而是：

> **把 zero-input 错误地当成了离散事件，而不是持续控制状态**

---

## 3.4 客户端本地显示态没有严格区分 predicted / authoritative / presentation

当前本地路径里：

- `NetworkEvent::LocalPosition`
  既可能来自本地预测发送时
  也可能来自权威 ack
- `WorldState.local_position`
  被直接覆盖

这使得客户端有时在同一个字段里混合了：

1. 本地预测位置
2. 权威确认位置
3. 最终展示位置

虽然目前这样能工作，但它会让“最终是否真正收口”这个问题变得不透明。

### 本质问题

客户端还缺一个明确的三层分离：

1. `predicted_state`
2. `authoritative_state`
3. `presentation_state`

如果没有这层分离，类似“最终位置对齐”就会一直通过外围补丁去解决，而不是在状态模型层面清楚地表达。

---

## 3.5 远端角色还是“单最新快照 + 外推”，不是完整插值时间线

当前远端路径：

- `clients/bevy_client/src/world/remote_player.rs`

只保存一个最新 snapshot，然后做短时外推。

这意味着：

- 如果最后一个 snapshot 仍带残余速度
- 远端就会继续按这个速度外推
- 直到下一次更新来覆盖它

所以 stop-sync 之所以能显著改善问题，是因为它增加了“最终零速 snapshot”的机会。  
但这也暴露出一个系统性事实：

> **远端目前还不是完整的 snapshot buffer + interpolation window**

因此它对“最终静止状态”的表达能力还不够强。

---

## 3.6 SceneServer 里存在“双重运动表示”

当前服务端实际上有两套相关状态：

1. `SceneServer.Movement.State`
   - 用于权威 step/replay
2. `SceneOps` 里的 legacy movement / physics component
   - 用于真实场景对象位置存储与读取

虽然当前主路径已经尽量让两者同步，但从架构上看，它仍然意味着：

- 一套“逻辑权威运动状态”
- 一套“底层物理/场景状态”

如果未来不明确谁是真正 source of truth，就很容易再次出现：

- 某边已经认为停下了
- 另一边还在按自己的时间基继续推进或持有旧速度

### 本质问题

需要明确：

> `Movement.State` 是唯一权威状态  
> `SceneOps` 只是权威状态落地到场景/物理容器的执行器

而不是两个都像“运动真相”。

---

## 4. 真正的系统性修复方向

如果要根治，而不是继续“最后补一补”，正确方向是：

## 4.1 改成“服务器固定 tick + 输入状态锁存”

服务端不应该只在收到输入包时 `step`。  
更合理的是：

1. 客户端发来的 `MovementInput` 表示：
   - 当前控制意图
   - 或当前 tick 的输入采样
2. 服务端 `PlayerCharacter` 维护：
   - `last_input_state`
   - `movement_state`
3. 服务端在**固定权威 tick** 上持续执行：
   - `Engine.step(...)`
4. 即使玩家松手了，只要速度还未归零，服务端也会继续推进 zero-input 状态直到真正静止

### 这样有什么好处

- 最终位置收口是模拟自然结果，不需要客户端“补 stop 包”
- acceleration / jerk 的物理语义才真正成立
- 以后 rollback / replay 也更标准

---

## 4.2 输入包应该表达“控制状态”，不应该直接承担“推进模拟”的责任

也就是说，协议语义要再明确：

- 输入帧 = 当前控制采样
- 权威 tick = 服务器自己的时钟

而不是：

- 收到一个输入帧 = 模拟前进一步

---

## 4.3 客户端本地状态要三层彻底分离

建议最终拆成：

1. `predicted_state`
2. `authoritative_state`
3. `presentation_state`

其中：

- 网络 ack 只更新 `authoritative_state`
- 本地输入只推进 `predicted_state`
- 画面和相机只跟 `presentation_state`
- 最终位置一致性检查对比：
  - local authoritative
  - server authoritative

---

## 4.4 远端角色要升级到 snapshot buffer

远端最终应该变成：

- snapshot ring buffer
- interpolation delay
- playback cursor
- 静止状态用最终零速 snapshot 明确表达

这样“最终停点一致”就不会依赖最后有没有恰好多发几帧。

---

## 4.5 明确 SceneOps 的角色：执行器，不是第二权威源

需要逐步收敛成：

- `Movement.State` 才是权威
- `SceneOps` 只是把权威状态投射到物理/场景世界

否则以后只要有任意一侧还保留自己的隐式时间推进逻辑，最终偏差问题就会反复出现。

---

## 5. 对当前实现的判断

### 当前 stop-sync 修复的定位

它是：

- **正确的症状级修复**
- **短中期可用**
- **能显著减少最终位置漂移**

但它不是根治，因为根治需要：

- authoritative fixed tick
- latched input state
- clearer state separation
- remote snapshot buffer

### 所以现在最准确的结论是

> 当前问题的系统性原因，是**移动系统仍然把“输入包”当成“模拟时钟”，而不是把输入当成在固定权威时钟上消费的控制状态**。

这才是最终位置不容易自然收口的根因。

---

## 6. 下一步建议（按优先级）

### P1
把 `PlayerCharacter` 改为：

- 固定 tick 驱动 `Engine.step`
- 输入包只更新 `last_input_state`

### P2
客户端明确拆开：

- predicted
- authoritative
- presentation

### P3
远端玩家升级为：

- snapshot buffer
- interpolation window

### P4
收敛 SceneOps 的职责边界，避免双重权威表示
