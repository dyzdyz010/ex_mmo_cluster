# 2026-04-13 最终位置同步收口

## 问题

在 vNext 输入流移动已经接通后，连续移动一段时间再停止，可能出现：

- 本地客户端位置
- 服务端权威位置
- 其他客户端看到的最终停点

逐渐产生偏差。

表面上看移动是同步的，但**停止后的最终位置**没有完全收口。

## 根因

客户端停止移动时，之前的逻辑只发送一次 zero-input / brake 帧：

- 方向变成 `Vec2::ZERO`
- 发一次带 brake flag 的 `MovementInput`
- 然后停止继续发送

但当前运动模型包含：

- acceleration
- jerk 限制

因此在一次 brake 帧之后，速度并不一定已经收敛到零。

结果是：

1. 服务端可能停在一个**残余速度尚未完全归零**的 snapshot 上
2. 远端客户端还会根据这个 snapshot 的速度继续做短时采样
3. 本地客户端与远端/服务端最终停点逐渐出现偏差

## 修复

客户端现在会在方向归零后继续发送 stop-sync 帧，直到本地速度真正衰减到阈值以内：

- `WorldState` 新增 `local_velocity`
- `NetworkEvent::LocalPosition` 新增 `velocity`
- `movement_sender` 不再只靠 `stop_sent` 判断是否停止发包
- 当：
  - `direction == Vec2::ZERO`
  - 但 `local_velocity` 仍大于阈值
  时，会继续发送 zero-input / brake 帧
- 当速度衰减到阈值内时，再额外发送一次最终 stop sync，然后停止

## 效果

这样可以保证：

- 本地预测真正完成减速
- 服务端权威位置继续推进到最终停点
- 远端收到最终零速 snapshot
- 最终位置重新收口
