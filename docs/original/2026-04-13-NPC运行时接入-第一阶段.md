# 2026-04-13 NPC 运行时接入（第一阶段）

## 本轮目标

在完成 NPC 服务端表示设计之后，把 NPC 从“仅有顶层模型”推进到：

- 已进入 scene supervision
- 已成为活跃运行时 actor
- 已接入 AOI / combat targeting
- 已能在 demo 运行时被生成

## 本轮完成

### 1. NpcActor 从骨架升级为运行时 actor

`SceneServer.Npc.Actor` 现在不再只是 brain skeleton，而是已经具备：

- AOI 注册
- `:await_ready`
- `:get_state_summary`
- `{:apply_skill_hit, ...}`
- HP / alive / respawn timer
- 周期性 brain tick

### 2. NPC 已进入 AOI 系统

通过：

- `SceneServer.AoiManager.add_aoi_item(..., actor_pid)`

活跃 NPC 现在会以 AOI item 的形式进入空间系统。

这意味着：

- 玩家已经可以把 NPC 当成空间中的一个权威实体
- NPC 也能收到 AOI 相关广播

### 3. Combat.Targeting 已从“只看玩家”升级为“看 combat actor”

`SceneServer.Combat.Targeting` 现在基于：

- `AoiManager.get_nearby_actor_pids`
- `:get_state_summary`

来寻找可战斗目标，而不再只依赖 `PlayerManager`。

这一步非常关键，因为它意味着：

- 玩家可以打 NPC
- NPC 也可以打玩家
- 战斗目标已经进入统一 actor 模型

### 4. Demo 运行时会自动生成一个训练 NPC

`Demo.Runner` 现在会在 demo 启动时额外生成一个：

- `npc_id = 90001`
- `name = "Training Slime"`

它位于玩家初始附近，方便后续继续做：

- AOI 可见性
- NPC 攻击
- 玩家打 NPC

### 5. PlayerCharacter 已开始兼容“目标可能是 NPC”

玩家技能命中链路现在通过 `Combat.Targeting.nearby_combatant_pids` 获取目标，  
因此已经不再只限定“玩家打玩家”。

## 这轮的架构意义

这一轮的核心不是“把 NPC 做成会动的怪”这么简单，而是：

> **把 NPC 正式纳入和玩家同一条权威 actor + AOI + combat targeting 体系中。**

也就是说，从系统边界看：

- 玩家和 NPC 开始共享同一套“可被 targeting / 可被伤害 / 可参与 AOI”的基本抽象
