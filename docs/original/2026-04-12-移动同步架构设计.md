# Autopilot Spec — 移动同步架构设计（含加速度 / 可选 jerk）

## 1. 目标

为 `ex_mmo_cluster` 设计一套面向 MMO/动作场景的现代移动同步架构，使其同时满足：

1. **本地手感快**：按键后立即动，不等服务器回包。
2. **服务器权威**：最终位置、速度、碰撞、合法性由服务器决定。
3. **纠错不抖**：服务器修正后尽量通过重演与表现层平滑消化，不硬跳。
4. **远端平滑**：别的玩家看起来连续，而不是按网络快照跳点。
5. **可增量迁移**：能从现有 `location/velocity/acceleration` 协议逐步演进。
6. **可扩展到加速度，且保留加加速度（jerk）选项**。

## 2. 术语与时间线

### 2.1 三条时间线

借鉴 Valve / Unity / Unreal 的成熟做法，明确拆分三条时间线：

1. **服务器权威时间线**
   - 唯一真实世界状态。
   - SceneServer 负责玩家运动积分、碰撞、约束、合法性。

2. **客户端预测时间线（本地玩家）**
   - 客户端以本地输入立即预测未来状态。
   - 服务器回包后进行 reconciliation（校正）与 replay（重演）。

3. **客户端插值时间线（远端玩家）**
   - 远端玩家不走本地预测，走 snapshot interpolation。
   - 只在必要时做有限 extrapolation。

### 2.2 四层状态模型

客户端本地玩家必须拆成四层：

1. **Input State**
   - 玩家输入：方向、技能键、冲刺键等。
2. **Predicted Simulation State**
   - 客户端根据输入积分出的本地预测结果。
3. **Reconciled Authoritative State**
   - 服务器确认后的权威状态。
4. **Presentation State**
   - 用于渲染 sprite / camera / animation 的平滑表现层。

> 关键原则：**相机和动画永远跟表现层，不直接跟权威状态跳点。**

## 3. 为什么“动作队列”不够

用户提出的“动作队列”适合：

- 技能释放
- 交互
- 离散状态变更

但对于移动并不够，因为移动不是一个个离散动作，而是连续输入流。

### 3.1 对移动的正确抽象

移动的主同步结构应该是：

- **输入历史环形缓冲区（input history ring buffer）**
- **预测状态历史环形缓冲区（predicted state history）**
- **服务器 ack + authoritative correction**

而不是：

- 一次移动 = 一个动作
- 一旦纠错 = 丢弃所有后续动作

因为那会造成：

- 本地手感迟钝
- correction 时大回弹
- 连续按键被吞

## 4. 建议的权威运动模型

### 4.1 优先推荐：输入驱动的运动 profile

对本项目，建议采用 **输入驱动、服务器积分** 的运动模型：

#### 输入层（玩家意图）
- `move_dir: Vec2/Vec3`
- `run_modifier`
- `brake_modifier`
- `timestamp/tick/sequence`

#### 运动状态层
- `position`
- `velocity`
- `acceleration`
- `facing`（可选）
- `movement_mode`（grounded / air / dash / root-motion 等）

#### 配置层
- `max_speed`
- `max_accel`
- `max_decel`
- `max_jerk`（可选）
- `friction`
- `turn_response`

### 4.2 jerk 是否值得引入

#### 建议
- **第一阶段：先做 acceleration，不把 jerk 作为网络同步字段。**
- **第二阶段：如需更高级手感，再把 jerk 作为本地/服务端共享运动 profile 的派生控制项。**

#### 原因
jerk（加加速度）主要是“手感塑形参数”，用于：
- 起步不突兀
- 停止不生硬
- 转向更顺滑
- 摄像机/动画更自然

但 jerk 不一定必须成为协议字段。

### 4.3 最佳实践

先网络同步：
- `input`
- `position`
- `velocity`
- `acceleration`

再本地/服务端共用 profile 参数：
- `max_jerk`
- easing / response curve

即：

- jerk **优先做本地与服务器一致的积分规则参数**
- 不要一上来把 jerk 直接塞进每个 movement packet

## 5. 协议设计建议

### 5.1 Client -> Server：输入流而不是位置流（最终形态）

当前项目已有：
- `request_id`
- `timestamp`
- `location`
- `velocity`
- `acceleration`

长期建议演进为：

```text
MoveInput {
  seq: u32
  client_tick: u32
  dt_ms: u16
  move_dir_x: f32
  move_dir_y: f32
  want_run: bool
  want_brake: bool
  movement_mode_hint: u8
}
```

#### 为什么不用“客户端上传位置”作为终态
因为上传位置更像：
- 客户端在提议结果
- 服务器在做纠正

而更现代的做法是：
- 客户端上传“输入”
- 服务器计算“结果”

这样才能稳定支持：
- rollback / replay
- 防作弊
- 多物理约束
- 加速度 / jerk 一致积分

### 5.2 过渡期兼容方案

为了兼容当前代码，可以分两步：

#### Phase A（兼容现网）
继续发送：
- `location`
- `velocity`
- `acceleration`

但新增：
- `seq`
- `client_tick`
- `input_dir`

服务器先继续吃旧字段，同时逐渐把“以输入为主”的积分逻辑接进来。

#### Phase B（稳定后）
服务器只信：
- `seq`
- `client_tick`
- `input_dir`
- movement modifiers

客户端上传的 `location/velocity` 仅作为 debug/telemetry，最终可删除。

### 5.3 Server -> Client：ack + authoritative state

建议权威回包至少包含：

```text
MoveAck {
  ack_seq: u32
  auth_tick: u32
  position: Vec3
  velocity: Vec3
  acceleration: Vec3
  movement_mode: u8
  correction_flags: u32
}
```

### 5.4 远端广播

远端广播建议包含：

```text
RemoteMoveSnapshot {
  cid: i64
  server_tick: u32
  position: Vec3
  velocity: Vec3
  acceleration: Vec3
}
```

远端客户端据此做：
- 速度感知插值
- 或 Hermite 插值

## 6. 本地玩家算法（推荐）

### 6.1 输入采样
每个固定 tick：
1. 采样输入
2. 生成 `InputFrame(seq, tick, input)`
3. 放入本地 `input_history`
4. 立刻执行本地积分 -> `predicted_state`
5. 发送给服务器

### 6.2 本地积分

建议服务端和客户端都用**同一套 deterministic-ish kinematic integrator**：

```text
wanted_velocity = normalize(input_dir) * max_speed
velocity_delta = wanted_velocity - current_velocity
accel_target = clamp_length(velocity_delta / dt, max_accel_or_decel)
accel = smooth_toward(prev_accel, accel_target, max_jerk, dt)
velocity = clamp_speed(current_velocity + accel * dt)
position = current_position + velocity * dt
```

关键点：
- `max_accel` 控制起步
- `max_decel` 控制刹车
- `max_jerk` 控制加速度变化率
- jerk 作用在“accel 的变化速度”而不是直接上网

### 6.3 收到服务器 ack 后

#### 若一致
- 删除 `<= ack_seq` 的输入历史
- 不做硬纠正

#### 若不一致
- 用服务器 `MoveAck` 覆盖 `auth_tick` 的 authoritative state
- 从 `auth_tick+1` 开始，用未确认输入重演
- 重新得到最新 predicted state

#### 表现层
- 小误差：presentation 平滑追近
- 大误差：presentation snap

### 6.4 不要直接丢弃后续输入

这是对用户原始想法最关键的升级：

- **不是 correction 后清空后续动作**
- **而是 correction 后 replay 后续输入**

这是当前主流方案。

## 7. 远端玩家算法（推荐）

### 7.1 Snapshot Buffer
每个远端玩家维护：
- 最近 N 个服务器快照
- 带 `server_tick`

### 7.2 插值
默认：
- 以一个固定 interpolation delay（例如 100~150ms）播放
- 在 `snapshot[t-1]` 与 `snapshot[t]` 之间插值

### 7.3 可选升级
如果要更高质量：
- 用 position + velocity 做 Hermite interpolation
- 比纯线性更自然，尤其在有 acceleration 的场景里

### 7.4 有限 extrapolation
只有在丢包或短时未到包时：
- 用 velocity 做短时 extrapolation
- 时间要很短
- 一旦新 snapshot 到来立即回归插值

## 8. 表现层与动画层设计

### 8.1 本地玩家
必须拆成：
- `authoritative_position`
- `predicted_position`
- `presentation_position`

`presentation_position` 才驱动：
- sprite / mesh
- camera follow target
- animation speed / blend

### 8.2 为什么你之前会抖
因为 Camera 跟了跳变状态，而不是平滑后的表现层。

### 8.3 动画建议
动画不要直接用“相邻两帧坐标差”推速度。
而要优先使用：
- 平滑后的 velocity
- 或输入意图速度

这样才不会因为 correction 导致动画抽搐。

## 9. 针对 `ex_mmo_cluster` 的落地建议

### Phase 1 — 当前代码上的低风险升级
1. 保留现有 `location/velocity/acceleration` 包格式
2. 新增 `seq` 与 `client_tick`
3. 本地增加：
   - input_history
   - predicted_history
4. 服务器 movement ack 返回：
   - `ack_seq`
   - authoritative `position/velocity/acceleration`
5. 本地做 rollback + replay
6. 相机/动画完全跟 presentation layer

### Phase 2 — 远端插值升级
1. 为 `PlayerMove` 广播增加 `server_tick`
2. 客户端远端玩家改用 snapshot buffer
3. 从线性插值升级到 velocity-aware/Hermite 插值

### Phase 3 — 输入驱动协议收敛
1. 客户端 movement packet 逐步从“位置提议”迁到“输入流”
2. 服务器只信输入，不信客户端位置
3. `location/velocity` 字段降级为 debug telemetry，最终删除

### Phase 4 — jerk/profile 细化
1. 在客户端与服务器共享 movement profile
2. 把 jerk 作为 profile 参数而不是网络字段
3. 调整：
   - 起步曲线
   - 刹停曲线
   - 转身曲线
   - 空中控制曲线

## 10. 推荐参数（初始建议）

### Tick
- simulation tick: `30Hz` 或 `60Hz`
- render/update: 帧率自由
- interpolation delay: `100~150ms`

### Movement
- `max_speed`
- `max_accel_ground`
- `max_decel_ground`
- `max_accel_air`
- `max_jerk_ground`
- `max_jerk_air`

### Correction
- `small_error_threshold`
- `snap_error_threshold`
- `presentation_smoothing_speed`
- `camera_smoothing_speed`

## 11. 最终推荐

对这个项目，**最先进且现实可落地**的路线是：

1. **本地玩家**：输入历史 + 客户端预测 + 服务器 ack + rollback/replay
2. **远端玩家**：snapshot buffer + interpolation（后续可升 Hermite）
3. **表现层**：presentation smoothing + camera smoothing + animation smoothing
4. **协议演进**：从位置提议逐步迁移到输入流协议
5. **加速度支持**：立刻支持
6. **jerk 支持**：作为共享运动 profile 参数逐步加入，而不是直接上网
