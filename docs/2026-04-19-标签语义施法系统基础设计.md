# 2026-04-19 标签语义施法系统基础设计

## 1. 文档目标

本文档用于为后续“以 Sevara 魔法语言为核心的开放式施法系统”建立统一概念底座。

本轮先解决的是：

- 世界对象如何被语义化描述
- 法术如何在不预枚举全部对象类型的前提下选择目标
- 生成类法术如何统一处理“空地生成”和“附着于对象生成”
- “标签越具体效果越强”应该如何被工程化表达
- 距离、熟练度、目标数量、对象可操作性如何一起结算
- 这套系统如何挂接到当前 `scene_server` / `combat` / `AOI` 底座

本轮不解决的是：

- 完整 Sevara 语法细节
- 全量魔法学校/词库设计
- UI 美术表现
- 数值平衡

---

## 2. 核心结论

## 2.1 魔法的对象不是实例 id，而是语义描述

施法时，玩家不必总是点选一个确定对象。

更一般的情况是：

> 玩家给出一组对象描述，系统在有限空间内查找“最符合这组描述、且允许被该操作作用”的对象集合。

因此，法术目标选择本质上是：

**空间筛选 + 语义筛选 + 操作合法性筛选**

---

## 2.2 “具体性”不是单个更深的词，而是跨维度约束的集合

本系统采用如下判断：

- `solid -> stone -> statue -> fire_god_statue` 这样的谱系仍然有意义
- 但真正的“具体”，不应只理解为某个单独标签更深
- 更准确的定义是：**施法描述在多个语义维度上同时给出了更多约束**

例如，下面三种描述的具体性逐步升高：

1. `solid`
2. `stone + statue`
3. `stone + statue + fire_god_related + temple_centerpiece`

也就是说：

> 具体性 = 约束维度更多 + 每个维度约束更深 + 约束之间更能共同缩小候选集合

这不是“运行时模糊猜测”，而是**确定性的结构化约束增强**。

---

## 2.3 运行时不依赖开放式模糊语义推断

为保证权威服可解释、可复现、可调试，运行时不采用开放式 embedding/LLM 式模糊匹配。

运行时只允许三种匹配来源：

1. **精确命中**
2. **标签谱系继承命中**
3. **编译期或内容期定义的别名归一化**

例如：

- 对象被标注为 `granite`
- 标签图中定义 `granite -> stone -> solid`
- 则搜索 `stone` 时该对象可命中

这仍然是确定性的，不属于不可解释的模糊匹配。

---

## 2.4 “更具体”带来的主要收益是聚焦，而不只是数值加成

“标签越具体效果越强”在系统中不应简单实现为一个粗暴乘数。

更合理的解释是：

- 约束越少，候选对象或位点越多，法术能量越分散
- 约束越多，候选对象或位点越少，法术能量越集中
- 约束越准确，单目标获得的有效作用越强

因此，更具体带来的收益主要体现在：

- 更高的命中稳定性
- 更高的语义匹配分
- 更小的候选集合
- 更高的能量集中度

---

## 2.5 对象必须显式声明“允许哪些操作”

对象不能只靠“它是什么”来决定能否被施法作用。

还必须声明：

> **它允许被哪些操作类型作用**

例如：

- 门可以 `open` / `close` / `lock` / `unlock`
- 石头可以 `move` / `break`，但通常不能 `grow`
- 植物可以 `grow` / `wither`
- 雕像可以 `heat` / `move` / `break`，但不能 `heal_as_living`

这类能力在本文档中称为：

**可操作性（affordance）**

---

## 2.6 生成类法术的对象是“生成锚点”，而不只是已有目标

当前文档主要讨论的是：

- 找到已有对象
- 对已有对象施加操作

但生成类 / 召唤类法术并不总是如此。

它们常见有两种输入：

- 选择一块没有现成物体的空间，在那里生成火墙、法阵、召唤物
- 选择一个已有对象，并在其表面、挂点或内部生成圣火、符印、附着体

为了把这两类情况统一到同一套模型里，建议额外引入三层概念：

- `host`：可选的已有对象，用来提供附着、承载或语义上下文
- `anchor`：最终执行生成的权威锚点，可能是点位、表面、socket、容器内部
- `product`：法术最终创建出的新实例，如火墙、圣火、召唤狼

这样可以统一表达为：

- “空地生成” = `host = nil`，直接解析 `anchor`
- “对物体生成” = 先解析 `host`，再从 `host` 派生 `anchor`

这里还要再明确一层：

- `host` 可以是“附着承载体”
- 也可以只是“参考系提供者”

例如：

- “在雕像顶部点燃圣火”里，`host` 同时提供语义上下文和附着位
- “在某物体上方 5 米生成岩石”里，`host` 只提供参考系，不提供附着点

界定规则也应一并明确：

> **只要法术的权威结算结果会创建新的世界实例，而不是仅修改已有对象状态，就归入生成类操作器。**

例如：

- 让树木进入“繁茂”状态，仍属于 `grow`
- 在树上生成一个独立可拾取果实实体，则属于生成类

## 3. 设计目标

## 3.1 目标

- 允许对象类型开放增长，而不是靠硬编码对象枚举
- 允许法术以“语义描述”选取目标，而不是只能点选实例
- 让“知识/命名能力”成为角色成长的一部分
- 让服务端保留确定性、权威性和可观测性
- 让客户端能解释“为什么命中/为什么失败”

## 3.2 非目标

- 不做通用自然语言理解器
- 不让运行时根据自由文本随意猜语义
- 不把所有交互都变成魔法系统
- 不为了开放性牺牲调试能力

---

## 4. 核心术语

| 术语 | 含义 |
| --- | --- |
| 标签（tag） | 描述对象某一语义属性的离散标识 |
| 维度（axis） | 标签所属的语义方向，如材质、形态、功能、状态 |
| 标签图（tag graph） | 标签间的继承、别名、相关性关系图 |
| 对象描述（descriptor） | 一个对象在多个维度上的标签集合 |
| 可操作性（affordance） | 对象允许被施加的操作类型 |
| 选择器（selector） | 法术用于筛选候选对象、位点或宿主的条件集合 |
| 锚点规格（anchor spec） | 描述锚点如何被生成出来的结构，如显式点、相对偏移、宿主位点 |
| 操作器（operator） | 法术最终施加的动作，如 `open`、`move`、`ignite` |
| 空间位点（site） | 可被法术选中的有限空间候选，如点位、表面、socket、容器内部 |
| 参考系（reference） | 锚点计算所依附的参考对象或参考点，如施法者、宿主、目标、光标点 |
| 原始锚点（raw anchor） | 根据 `anchor spec` 计算出来、尚未经过投影和合法性校验的锚点 |
| 宿主对象（host） | 为生成提供附着、承载或上下文的已有对象 |
| 生成锚点（anchor） | 法术最终执行生成或放置的权威位置与朝向 |
| 产物（product） | 法术创建出的新实例，如火墙、圣火、召唤物 |
| 作用场（field） | 法术的空间影响形状与距离衰减模型 |
| 解析（resolution） | 从施法输入到候选筛选、评分、结算、广播的完整过程 |

## 5. 世界对象语义模型

## 5.1 对象不是一个类别，而是一组跨维度标签

每个对象都由多个维度共同描述，而不是单一类型。

建议至少拆成以下维度：

| 维度 | 作用 | 示例 |
| --- | --- | --- |
| `ontology` | 本体/存在类型 | `solid`、`living`、`artifact` |
| `material` | 材质 | `stone`、`wood`、`iron`、`flesh` |
| `form` | 形态/构型 | `statue`、`door`、`container`、`pillar` |
| `function` | 功能/角色 | `barrier`、`altar`、`weapon_mount` |
| `state` | 当前状态 | `closed`、`locked`、`broken`、`heated` |
| `relation` | 关系/归属 | `temple_owned`、`quest_bound`、`fire_god_related` |
| `affinity` | 元素/学派相关性 | `fire_aspected`、`life_aspected` |
| `affordance` | 可被作用的操作 | `openable`、`movable`、`flammable`、`growable` |

其中最关键的是：

- `tag` 说明“它是什么”
- `affordance` 说明“你可以对它做什么”

---

## 5.2 例子：火神雕像

一个“火神雕像”不应只写成一个标签。

更合理的对象描述类似：

```text
object: shrine_fire_statue_01

ontology:
  - solid
  - artifact

material:
  - stone

form:
  - statue

function:
  - shrine_centerpiece
  - decoration

relation:
  - fire_god_related
  - temple_owned

state:
  - intact
  - anchored

affordance:
  - movable
  - heatable
  - breakable
  - inspectable
```

这样它既可以被：

- `stone`
- `statue`
- `fire_god_related`
- `movable`

等不同方向的法术所选择，又不会把所有语义压缩成一个不可拆分的“超级标签”。

---

## 5.3 例子：门

“打开门”这个例子说明：

对象必须同时携带分类与可操作性信息。

```text
object: temple_gate_03

ontology:
  - solid
  - artifact

material:
  - wood
  - iron_fittings

form:
  - door
  - panel

function:
  - barrier
  - entry

state:
  - closed
  - unlocked

affordance:
  - openable
  - closeable
  - lockable
  - unlockable
  - breakable
```

如果法术是 `open`：

- `door` 不是唯一必要条件
- `openable` 才是最低合法性条件
- `closed`、`locked` 等状态会进一步影响结果

---

## 5.4 空间位点也需要被语义化

现有的 `ObjectDescriptor` 足以表达“对什么做事”，但生成类法术还需要表达“在哪里生成”。

这里不建议把“空空间”视为一个无限连续、无约束的模糊概念，而应把它收敛为**有限的可候选位点集合**。

这些位点可以来自：

- 玩家显式点选的空间点位
- 地面 / navmesh 上的可落点
- 某个对象表面的可附着区域
- 某个对象预定义的 socket / 挂点
- 容器或结构内部的合法放置位
- 根据施法者 / 宿主 / 目标的 `reference + offset` 在运行时计算出的相对点

建议引入 `SiteDescriptor`，把“空间中的可施法位点”也作为可解释候选来处理。

这里的 `SiteDescriptor` 不一定都来自静态场景数据。

它也可以是运行时临时构造出的**虚拟位点**，例如：

- 施法者面前 2 米的空气点
- 某物体上方 5 米的空间点

这些点先作为 `raw anchor` 被计算出来，再在解析流程里被包装成可校验的 site candidate。

一个位点不一定有宿主对象：

- 空地位点的 `host = nil`
- 表面 / socket / 容器内部位点的 `host = object_id`

位点自身也需要最小语义：

- 它位于什么介质中，如 `medium: air`
- 它是否有承托，如 `support: ground`
- 它是否空闲，如 `occupancy: free`
- 它是否允许生成 / 附着 / 召唤

这样：

- “在空地升起火墙”
- “在雕像顶部点燃圣火”

就能落在一套统一的解析模型里，而不是拆成两套互不兼容的特殊逻辑。

## 6. 标签图与具体性

## 6.1 标签图不是单树，而是多维有向图

单个维度内部可以有谱系：

```text
solid -> stone -> carved_stone -> granite
```

但整体系统不能只有一棵树，因为对象同时存在于多个维度。

推荐模型：

- 维度内：使用继承/上位概念链
- 维度间：通过对象描述组合，而不是互相继承
- 别名：在内容编译期做归一化

例如：

- `granite` 属于 `material`
- `statue` 属于 `form`
- `fire_god_related` 属于 `relation`

它们共同构成“更具体的对象描述”。

---

## 6.2 具体性分数

为支持“越具体越强”的设计，建议为选择器计算一个**具体性分数**，但该分数只来自结构化信息，不来自自由文本长度。

建议具体性由三部分组成：

1. **维度覆盖度**
   - 给出了多少个不同维度的约束
2. **层级深度**
   - 每个约束在本维度内有多深
3. **约束强度**
   - 这些约束在多大程度上缩小了候选集合

可以先用一个简单、可调试的实现：

```text
specificity(selector) =
  axis_coverage_score +
  taxonomy_depth_score +
  hard_constraint_score
```

其中：

- `axis_coverage_score`：每命中一个新维度加分
- `taxonomy_depth_score`：越深的标签加分越高
- `hard_constraint_score`：`required` / `exclude` / `required_affordance` 这类硬条件比 `prefer` 更重

这套分数的用途不是直接决定胜负，而是：

- 影响语义聚焦度
- 参与能量分配
- 参与 UI 解释

## 7. 施法语义模型

## 7.1 一个法术至少包含四部分

一个完整法术建议拆成：

1. **选择器（selector）**
2. **锚点规格（anchor spec）**
3. **操作器（operator）**
4. **作用场（field）**

可抽象为：

```text
spell = selector + anchor_spec + operator + field + resource_cost + mastery_constraints
```

---

## 7.2 选择器

选择器用于描述“想找什么对象、位点，或两者组合”。

建议至少支持：

- `subject_domain`
- `required_tags`
- `preferred_tags`
- `excluded_tags`
- `required_affordances`
- `target_limit`
- `sort_mode`
- `spatial_anchor`
- `shape`
- `radius`

一个概念例子：

```text
selector:
  subject_domain: object
  required_tags:
    - form: door
  required_affordances:
    - openable
  preferred_tags:
    - state: closed
  excluded_tags:
    - state: sealed
  spatial_anchor: cursor_point
  shape: sphere
  radius: 6m
  sort_mode: best_match_then_nearest
  target_limit: 1
```

其中：

- `subject_domain = object`：普通“对已有对象施法”，也是默认模式
- `subject_domain = site`：直接寻找空间位点，常用于火墙、法阵、召唤落点
- `subject_domain = hybrid`：先解析宿主对象，再从宿主派生合法锚点
- `required_affordances`：用于过滤对象或宿主候选
- `shape` / `radius`：定义选择器用于查询候选的几何包络
- `spatial_anchor`：定义选择器从哪里开始做候选查询，如施法者、自身点选、当前光标点
- `selector` 负责“选谁 / 选哪个宿主 / 选哪个查询参考点”，不负责生成相对偏移锚点

这样可以保持：

- 普通对象操作法术继续沿用现有平面模型
- 生成类法术通过扩展 `selector` 接入，而不是成为系统外特例

---

## 7.3 锚点规格（AnchorSpec）

`anchor_spec` 描述的不是“锚点是否合法”，而是：

> **锚点是如何被生成出来的。**

建议至少支持：

- `source_kind`
- `reference_kind`
- `basis`
- `offset`
- `projection`
- `anchor_selector`

其中：

- `source_kind = explicit_point`：直接使用显式点选输入
- `source_kind = host_site`：从宿主对象派生表面、socket、内部位
- `source_kind = relative_offset`：相对于参考系做偏移计算
- `source_kind = raycast`：沿某个方向发射射线后取命中点
- `source_kind = projected_point`：先算点，再投影到地面 / navmesh / 自由空间

- `reference_kind = caster`：以施法者为参考系
- `reference_kind = host`：以宿主对象为参考系
- `reference_kind = target`：以当前目标为参考系
- `reference_kind = cursor_point`：以光标点或显式点为参考系

- `basis` 用于定义偏移方向解释方式，如 `world`、`caster_local`、`host_local`、`facing`、`surface_normal`
- `offset` 用于定义 `forward` / `right` / `up` 等相对位移
- `projection` 用于定义后处理投影，如 `none`、`to_surface`、`to_navmesh`、`to_free_space`
- `anchor_selector` 用于校验生成后的锚点是否合法

一个概念例子：

```text
anchor_spec:
  source_kind: relative_offset
  reference_kind: caster
  basis: facing
  offset:
    forward: 2m
    up: 1.2m
  projection: none
  anchor_selector:
    allowed_anchor_kinds:
      - point
    required_site_tags:
      - medium: air
    require_free_space: true
```

这个结构就能表达：

- “在我面前 2 米生成一个火球”
- “在某物体上方 5 米生成一块岩石”

而不要求锚点必须来自既有附着位。

---

## 7.4 操作器

操作器描述“对命中对象做什么”。

建议分成稳定的大类，而不是每个法术发明一套新行为：

- `open`
- `close`
- `move`
- `push`
- `pull`
- `ignite`
- `cool`
- `grow`
- `bind`
- `break`
- `sense`
- `manifest`

每个操作器都声明：

- 需要哪些 affordance
- 能修改哪些状态
- 允许作用于哪些对象维度
- 是否会创建新实例，以及新实例属于什么类型
- 是否对活体/死物有不同结算路径

例如：

- `grow` 需要 `growable`
- `open` 需要 `openable`
- `ignite` 通常需要 `flammable` 或 `heatable`
- `manifest` 通常需要位点具备 `spawnable`、`attachable`、`summonable` 中的一类或多类

对生成类 / 召唤类法术，建议不要为每个产物发明一个全新操作器，而是统一归入 `manifest` 家族，再通过参数区分：

- `product_kind`：`effect` / `object` / `actor`
- `lifecycle_kind`：瞬时、持续、可回收、可召回
- `cleanup_policy`：超时消失、被销毁、手动解除、区域离开后清理

这样：

- “空地升起火墙”
- “在雕像顶部点燃圣火”
- “召唤狼到指定地点”

都属于同一类操作器，只是 `product` 模板不同。

---

## 7.5 作用场

作用场定义空间限制。

这轮采用你的核心想法：

> 法术影响范围受熟练度约束，距离越远效果越弱，越近效果越强。

建议初版使用：

- 以施法者为锚点的球形或扇形区域
- 半径由法术等级与熟练度共同决定
- 区域内按距离连续衰减，不优先采用硬边界

可抽象为：

```text
field:
  shape: sphere
  radius: mastery_scaled
  falloff: smooth
```

这里建议明确：

- `selector.shape` / `selector.radius` 负责“查哪里”，也就是候选查询包络
- `field.shape` / `field.radius` / `field.falloff` 负责“生效时怎样衰减”

如果业务上不需要把两者拆开，初版可以让 `selector` 默认继承 `field` 的空间配置。

对普通法术，`field` 主要约束“能作用到哪些对象”。

对 `manifest`，`field` 主要约束施法距离、默认查询范围与后续强度衰减；**锚点本身如何生成**则交给 `anchor_spec`。

---

## 8. 目标/锚点解析流程

## 8.1 总流程

建议目标解析流程固定为：

1. **空间预筛选（对象或位点）**
2. **可操作性 / 可生成性门禁**
3. **语义匹配评分与宿主解析**
4. **锚点生成（AnchorSpec）**
5. **锚点合法性校验**
6. **候选排序、强度分配与合法性结算**
7. **权威广播与可观测输出**

---

## 8.2 第一步：空间预筛选

依托现有 AOI / 空间索引能力，在有限半径内取候选对象或候选位点。

这一层只解决：

- 哪些对象在施法范围内
- 哪些点位 / 表面 / socket / 容器内部可以成为候选位点
- 距离是多少
- 是否可作为下一步候选

这一步不做复杂语义推理。

对生成类法术尤其要注意：

> “空空间”不应被视为无限候选集合，而应来自显式点选输入或有限的 site sampling。

---

## 8.3 第二步：可操作性门禁

在进入语义评分前，先按操作器过滤不合法对象或位点。

例如：

- `open` 过滤掉所有不具备 `openable` 的对象
- `grow` 过滤掉所有不具备 `growable` 的对象
- `heal` 过滤掉所有不具备 `living` 或 `healable` 的对象
- `manifest` 过滤掉所有不具备 `spawnable` / `attachable` / `summonable` 的位点
- `attached manifest` 还要检查宿主对象是否允许派生对应锚点

这样能避免出现：

- “门可以被生长”
- “石像可以被治疗成活物”
- “任何空气点都能无条件召唤实体”
- “任何雕像表面都能附着圣火”

这类概念错误。

---

## 8.4 第三步：语义匹配评分与宿主解析

语义匹配采用**可解释的确定性打分**。

建议匹配关系只允许：

- 精确命中：`1.0`
- 继承命中：`lineage_decay(depth_delta)`
- 缺失：`0.0`

可定义：

```text
semantic_score(candidate) =
  weighted_required_match *
  preferred_bonus *
  exclusion_gate *
  descriptor_specificity_bonus
```

其中：

- `weighted_required_match`：必需标签的加权命中率
- `preferred_bonus`：偏好标签的加分
- `exclusion_gate`：命中排除标签则直接归零
- `descriptor_specificity_bonus`：选择器越具体，聚焦上限越高

这不是开放式模糊匹配，而是**标签图上的可解释匹配**。

对 `subject_domain = object` 或 `hybrid` 的法术，这一步还会确定：

- 主目标是谁
- 宿主对象是谁
- 后续 `anchor_spec.reference_kind = host` 时应绑定哪个参考对象

---

## 8.5 第四步：锚点生成（AnchorSpec）

在宿主 / 参考系确定后，开始执行 `anchor_spec`，生成 `raw anchor`。

可抽象为：

```text
raw_anchor_i = build_anchor(anchor_spec, reference_i)
```

典型来源包括：

- 显式点选
- 宿主表面或 socket
- 施法者面前 / 身后 / 头顶的相对偏移
- 宿主上方固定高度
- 射线命中后的位置

例如：

- “我面前 2 米生成火球” = `reference = caster` + `basis = facing` + `offset.forward = 2m`
- “某物体上方 5 米生成岩石” = `reference = host` + `basis = world` + `offset.up = 5m`

这一步生成出来的还只是 `raw anchor`，尚未保证它一定可用。

---

## 8.6 第五步：锚点合法性校验

`raw anchor` 生成后，要再经过 `anchor_selector` 与必要投影步骤，才能变成真正可执行的 `resolved anchor`。

典型校验包括：

- 是否落在允许的 `anchor_kind`
- 是否满足 `required_site_tags`
- 是否具备 `required_site_affordances`
- 是否有足够自由空间
- 是否满足视线约束
- 是否需要投影到地面 / navmesh / 表面

可抽象为：

```text
resolved_anchor_i =
  raw_anchor_i
  |> apply_projection(anchor_spec.projection)
  |> validate_with(anchor_spec.anchor_selector)
```

这一步会把：

- 显式点
- 相对偏移点
- 表面 / socket 点

统一收敛为可执行的锚点结果。

---

## 8.7 第六步：候选排序与能量分配

命中多个对象或位点时，不建议简单地“全部吃满伤害/效果”。

更合理的规则是：

- 法术总能量有限
- 候选越多，能量越分散
- 候选越少，能量越集中

因此建议引入**聚焦分配**：

```text
focus_share_i = score_i / Σ(all_candidate_scores)
```

每个对象目标或锚点候选获得的最终有效强度由其 `focus_share` 决定。

这样自然得到：

- `推动 solid` 会影响很多东西，但每个目标受力较小
- `推动 stone + statue + fire_god_related` 候选很少，主目标受力很强

这正是“越具体越强”的工程化表达。

---

## 8.8 第六步补充：距离衰减与熟练度

距离衰减和语义聚焦应是同一层逻辑的两个因子。

建议：

```text
spatial_score_i = falloff(distance_i, mastery_radius, mastery_curve)
```

则单对象目标或单锚点结果的最终有效强度可表达为：

```text
final_power_i =
  base_power
  * semantic_score_i
  * focus_share_i
  * spatial_score_i
  * mastery_factor
  * operator_compatibility_i
  * resistance_factor_i
```

这里：

- `semantic_score_i` 解决“你说得准不准”
- `focus_share_i` 解决“你是不是把能量分散给了很多目标”
- `spatial_score_i` 解决“距离远近”
- `mastery_factor` 解决“你能不能稳住这个法术”

对 `manifest` 来说，这个 `final_power_i` 不一定直接表现为伤害。

它也可以进一步映射为：

- 生成物持续时间
- 生成物尺寸或强度
- 召唤物初始品质
- 附着效果稳定度

---

## 8.9 第七步：操作结算

通过上述筛选后，再进入具体操作逻辑。

例如：

- `open` 修改对象状态：`closed -> open`
- `ignite` 施加状态：`burning`
- `move` 产生位移/冲量
- `break` 走耐久或结构判定
- `manifest` 在合法 `anchor` 上创建新实例，并登记其生命周期、复制范围与清理规则

这一步建议沿用当前项目已经形成的：

- authoritative resolution
- effect/event 分离
- AOI 广播
- client consume cue

思路，不重新发明一整套复制链路。

---

## 8.10 生成类法术的补充流程

当 `operator = manifest` 时，解析结果不再只是“命中了哪些已有对象”，而是要输出：

- `host`
- `reference`
- `raw_anchor`
- `anchor`
- `product`

建议固定处理为：

1. 根据 `CastRequest` 先确定 seed
2. `target_mode = :point` 时，把显式点位作为 reference 或 raw anchor seed
3. `target_mode = :actor` 时，把显式对象作为 host seed
4. `target_mode = :auto` 时，在 `field` 范围内采样有限对象 / 位点候选
5. 先用 `selector` 解出 host / target / site candidate
6. 再用 `anchor_spec` 生成 `raw anchor`
7. 对 `raw anchor` 做投影、占用、承托、视线、navmesh、attachment 合法性校验
8. 输出 `resolved host + reference + raw_anchor + anchor + product_spec`

统一后可得到：

- 点空地生成：`host = nil`
- 点物体附着生成：`host = 某对象`，`anchor` 来自 `host_site`
- 相对偏移生成：`host` 可空，`anchor` 来自 `reference + offset`
- 自动寻找可落点生成：`host` 可空，`anchor` 来自有限候选集或投影结果

这样“空地生成”和“选择某物生成”就不再是两套系统，而只是 `anchor` 来源不同。

---

## 9. 匹配不是“模糊猜”，而是“祖先闭包 + 多维约束”

为了避免把“具体性”重新拉回不可控模糊，需要明确以下规则：

## 9.1 运行时匹配单位是规范化标签

内容生产期应把：

- 别名
- 近义命名
- 旧标签迁移

统一编译成规范标签。

运行时只消费规范标签，不直接消费任意自然语言。

---

## 9.2 对象 / 位点标签应在加载期展开祖先闭包

例如对象显式标签是：

- `granite`
- `statue`

加载后可得到闭包：

- `granite`
- `stone`
- `solid`
- `statue`

位点标签也应采用同样策略。

这样运行时查找 `stone` 或 `support: ground` 时无需做高成本推理，也更便于日志解释。

---

## 9.3 选择器的“具体”来自 conjunction，而不是模糊相似度

例如：

- `stone` 是单条件
- `stone + statue` 是双条件
- `stone + statue + fire_god_related` 是三条件

系统应把它理解为：

> 约束越来越多，候选越来越窄，法术越来越聚焦

而不是：

> 系统自由猜哪个东西“看起来也像”

---

## 10. 可解释性与可观测性

这个系统是否可玩，关键不只是算法，而是**解释面**。

## 10.1 必须能回答的四个问题

每次施法后，系统至少要能解释：

1. 搜索了什么范围
2. 应用了哪些语义条件与锚点约束
3. 参考系与锚点是如何生成出来的
4. 为什么某个对象 / 位点命中或被排除，以及最终强度、生成位置或失败原因为什么是这个结果

---

## 10.2 建议的结构化日志

结合仓库现有 CLI / observe 约定，建议新增以下可观测事件：

- `magic_cast_started`
- `magic_candidate_scored`
- `magic_candidate_rejected`
- `magic_anchor_built`
- `magic_anchor_candidate`
- `magic_anchor_rejected`
- `magic_anchor_resolved`
- `magic_cast_resolved`
- `magic_product_spawned`
- `magic_cast_failed`

建议默认写入：

- `.demo/observe/`
- 或显式配置的 observe 目录

单条候选评分可包含：

```json
{
  "event": "magic_candidate_scored",
  "cast_id": "cast-123",
  "candidate_id": "temple_gate_03",
  "distance": 1.8,
  "required_match": 1.0,
  "preferred_bonus": 0.15,
  "specificity_bonus": 0.25,
  "focus_share": 0.62,
  "operator_gate": true,
  "final_score": 0.74
}
```

其中 `candidate_id` 可以是对象 id，也可以是 site id，取决于当前 `subject_domain`。

对生成类法术，还应额外记录：

- `host_object_id`
- `reference_kind`
- `basis`
- `offset`
- `projection`
- `raw_anchor_position`
- `anchor_kind`
- `anchor_position`
- `product_kind`
- `product_template_id`
- `reject_reason`

---

## 10.3 面向玩家的解释面

玩家侧不需要看全量原始分数，但至少需要：

- 当前法术在找什么
- 命中的主目标或生成锚点是什么
- 为什么失败

例如：

- “未命中：附近没有可开启对象”
- “已命中：木门，距离 1.8m，匹配度高”
- “效果减弱：你选择的是 `solid`，命中了 7 个目标，能量已分散”
- “已生成：火球出现在你面前 2 米处，因为锚点采用了 `caster + facing + forward 2m`”
- “生成失败：目标雕像存在，但顶部没有可附着的合法位点”

## 11. 与现有项目架构的挂接方式

## 11.1 与当前 `combat` 底座的关系

当前仓库已经具备：

- `Combat.CastRequest`
- `Combat.Skill`
- `Combat.Targeting`
- `Combat.Executor`
- AOI 广播
- effect cue 复制

因此这套系统不应绕开现有底座，而应新增一层“语义目标解析”。

建议关系为：

```text
Sevara / Magic Intent
  -> Magic Selector Resolver
  -> resolved targets / host
  -> AnchorSpec Resolver
  -> reference + raw_anchor + resolved anchor + resolved operation parameters + product_spec
  -> Combat / Gameplay Executor
  -> effect cues + state updates + AOI fanout
```

也就是说：

- `Combat` 继续负责权威执行
- `Magic` 负责“语义选谁 / 选哪里、如何生成锚点、为什么选、怎么分配”

对普通法术，解析结果主要是 `resolved_targets`。

对生成类法术，解析结果主要是 `host_object_id? + anchor_reference + raw_anchor + anchor + product_spec`。

这与当前 `combat` 已有的 `:point` / `:actor` / `:self` 输入模式是兼容的：

- 显式点位天然可以成为 anchor seed
- 显式对象天然可以成为 host seed

## 11.2 建议的模块边界

建议先在 `scene_server` 内以 gameplay kernel 方式落位：

```text
apps/scene_server/lib/scene_server/gameplay/magic/

```

职责建议：

- `tag.ex`：规范标签定义

- `descriptor.ex`：对象语义描述

按照仓库已有“kernel + business/content”方向，建议分层如下：

- affordance gate
- 距离衰减

### Business

- 法术模板

这样能保持：

## 12. 推荐的最小数据结构

## 12.1 对象描述

```text
ObjectDescriptor
  object_id
  scene_id
  position
  tags_by_axis
  affordances
  dynamic_states
  relation_facts
```

其中：

- `tags_by_axis`：静态或半静态语义标签
- `affordances`：允许操作集合
- `dynamic_states`：运行时可变化状态
- `relation_facts`：归属、绑定、阵营等上下文

---

## 12.2 空间位点描述

```text
SiteDescriptor
  site_id
  scene_id
  site_kind
  position
  normal
  host_object_id
  tags_by_axis
  affordances
  occupancy
```

其中：

- `site_kind`：`point` / `surface` / `socket` / `inside`
- `host_object_id`：空地位点为 `nil`，附着位点则指向宿主对象
- `tags_by_axis`：例如 `medium: air`、`support: ground`、`surface: stone`
- `affordances`：例如 `spawnable`、`attachable`、`buildable`、`summonable`
- `occupancy`：表达阻挡、拥挤、是否有足够空间等状态

---

## 12.3 施法选择器

```text
MagicSelector
  subject_domain
  required_tags
  preferred_tags
  excluded_tags
  required_affordances
  spatial_anchor
  radius
  shape
  sort_mode
  target_limit
  split_mode
```

其中：

- `subject_domain`：`object` / `site` / `hybrid`
- `required_affordances`：过滤对象或宿主；位点相关合法性放入 `anchor_selector.required_site_affordances`
- `spatial_anchor`：定义候选查询相对哪个参考点展开
- `shape` / `radius`：定义候选查询包络；若省略可默认继承 `field`
- `split_mode` 用于决定多目标能量如何分配
- 初版可以只支持 `weighted_focus`

---

## 12.4 锚点规格

```text
AnchorSpec
  source_kind
  reference_kind
  basis
  offset
  projection
  anchor_selector
```

其中：

- `source_kind`：`explicit_point` / `host_site` / `relative_offset` / `raycast` / `projected_point`
- `reference_kind`：`caster` / `host` / `target` / `cursor_point`
- `basis`：`world` / `caster_local` / `host_local` / `facing` / `surface_normal`
- `offset`：定义相对位移，可进一步表达为：

```text
OffsetVector
  forward
  right
  up
```

- `projection`：`none` / `to_surface` / `to_navmesh` / `to_free_space`
- `anchor_selector`：用于过滤可接受的位点/锚点，可进一步表达为：

```text
AnchorSelector
  allowed_anchor_kinds
  required_site_tags
  preferred_site_tags
  excluded_site_tags
  required_site_affordances
  require_free_space
  require_line_of_sight
```

---

## 12.5 操作模板

```text
MagicOperator
  operator_type
  required_affordances
  base_power
  state_transitions
  effect_channel
  product_spec
```

其中：

- 普通操作器可以不填 `product_spec`
- `manifest` 家族必须提供 `product_spec`
- `product_spec` 至少应包含 `product_kind`（`effect` / `object` / `actor`）与 `template_id`

这样操作器就可以和目标解析解耦。

---

## 12.6 解析结果

```text
ResolvedMagicCast
  resolved_targets
  host_object_id
  anchor_reference
  raw_anchor
  anchor
  operator_type
  resolved_operator_params
  product_spec
```

其中：

- 普通法术主要消费 `resolved_targets`
- 生成类法术主要消费 `host_object_id + anchor_reference + raw_anchor + anchor + product_spec`
- `anchor_reference` 用于说明锚点依附哪个参考系
- `raw_anchor` 建议统一表达为：

```text
ResolvedRawAnchor
  position
  normal
  built_from
```

- `built_from` 可指向显式点选、相对偏移、宿主位点、射线命中
- `anchor` 建议统一表达为：

```text
ResolvedAnchor
  kind
  position
  normal
  derived_from
```

- `derived_from` 可指向显式点选、宿主表面、socket 或容器内部

## 13. 典型例子

为保持示例可读性，下面各例子会按需要省略一部分默认项。

普通对象操作法术尤其常省略：

- `subject_domain: object`
- 默认 `anchor_spec`（及其内含的默认 `anchor_selector`）
- 与 `field` 一致的默认 `shape` / `radius` 继承关系

位点 / 生成类例子也可能省略默认的 `spatial_anchor` 或默认查询包络，只保留最关键的结构字段。

## 13.1 例子一：打开门

```text
selector:
  required_tags:
    - form: door
  required_affordances:
    - openable
  preferred_tags:
    - state: closed
  excluded_tags:
    - state: sealed

operator:
  open
```

结果：

- 普通门能命中
- 封死的石门会被排除

## 15. 当前设计决策

本轮先明确以下决策：

1. **具体性定义为跨维度约束集合的强度，不等于单标签名称更长。**
2. **运行时不做开放式模糊语义匹配，只做确定性的标签图匹配。**
3. **对象必须声明 affordance，操作合法性优先于语义相似性。**
4. **效果增强主要来自聚焦与能量集中，而不是简单的“更具体乘一个大数”。**
5. **空间限制采用连续衰减，与语义聚焦共同决定最终强度。**
6. **必须提供 CLI/observe 可解释面，不能把命中逻辑做成黑箱。**
7. **生成类法术统一通过 `host + anchor + product` 建模，不把空地生成和附着生成拆成两套系统。**
8. **锚点生成（AnchorSpec）与锚点合法性校验（AnchorSelector）必须分离。**

---

## 16. 当前主要风险

即使概念成立，落地时仍有几个风险需要持续控制：

- **标签漂移**：多人加内容后，标签命名可能失控
- **作者负担**：对象 / 位点标注体系太重会降低内容生产速度
- **玩家认知门槛**：太自由但解释不足，会显得像玄学
- **性能风险**：大范围查询 + 多候选评分可能放大 AOI 压力
- **位点枚举爆炸**：surface / socket / navmesh 采样若不收敛，会放大解析成本
- **参考系歧义**：`facing`、`host_local`、`world_up` 等偏移基准若定义不清，会导致法术位置不可预测
- **生成物治理**：产物模板与生命周期规则增多后，清理和复制链路容易失控
- **数值风险**：聚焦、距离、抗性、熟练度之间可能互相放大

因此实现时必须坚持：

- 先做少量高质量标签轴
- 先做强解释面
- 先做有限对象集验证
- 先跑 CLI/observe 回归

---

## 17. 结论

这套系统的关键，不是“标签多”，而是：

> **把对象、位点、锚点生成、操作、空间、熟练度统一纳入一套可解释、可扩展、可权威结算的语义施法模型。**

如果这个底座成立，那么后续扩展法术时，就不再依赖“继续添加更多预制技能原语”，而是转向：

- 扩展对象与位点语义
- 扩展操作器与产物模板
- 扩展 Sevara 表达能力
- 扩展内容标注与学派知识

这会让魔法系统从“有限技能组合”转向“开放世界语义操控”。
