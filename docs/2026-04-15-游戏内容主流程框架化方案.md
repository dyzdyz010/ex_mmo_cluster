# 2026-04-15 游戏内容主流程框架化方案（排除 auth_server 分层）

## 1. 这轮的范围裁决

本方案只讨论：

- **游戏内容主流程** 如何从当前项目中抽成 framework/kernel + business/content 两层
- **scene_server / gate_server / bevy_client** 中与 gameplay 主流程直接相关的抽象边界

明确不讨论：
- `auth_server` 的继续分层
- 账号 / 登录 / 角色归属的内核化
- 通用用户系统抽象

在本轮里，`auth_server` 仅保持为：
- 既有身份边界
- token / character ownership 服务
- 不参与 gameplay kernel 的继续设计

---

## 2. 当前主流程已经有什么

截至当前，项目的 gameplay main flow 已经具备这些底座雏形：

### 2.1 authority runtime（scene_server）
- player actor
- npc actor
- movement authority
- combat authority
- AOI
- Rustler hot path
- fixed tick runtime

### 2.2 transport runtime（gate_server）
- TCP control plane
- UDP fast-lane
- stdio / observe / protocol dispatch
- player/NPC/combat fanout

### 2.3 client runtime（bevy_client）
- local prediction
- authoritative ack reconcile
- remote snapshot interpolation
- NPC identity awareness
- stdio/headless/E2E

### 2.4 现在真正缺的不是“功能能不能跑”
而是：

> **如何把“主流程运行时底座”与“具体游戏内容逻辑”分离。**

---

## 3. 推荐的目标形态

## 3.1 Gameplay Kernel（底座）

Kernel 负责：
- actor lifecycle
- fixed tick scheduling
- movement primitive
- combat primitive
- AOI primitive
- event dispatch primitive
- replication contract
- observability / stdio / E2E hooks

Kernel 不负责：
- skill 的具体业务语义
- 某种怪的 AI 细节
- 某张图刷什么怪
- 掉落、任务、副本规则

---

## 3.2 Business Layer（业务实现）

Business 负责：
- skill catalog
- NPC brain implementation
- scene rule implementation
- spawn policy implementation
- loot / progression / quest / dungeon rules

也就是说：
- Kernel 只定义扩展点
- Business 提供具体实现

---

## 3.3 Content Layer（数据/内容）

Content 负责：
- npc template
- scene composition
- spawn groups
- wave definitions
- reward table
- skill tuning values

### 关键原则
尽量先把“业务”和“内容”分开：
- 业务 = 规则与行为模块
- 内容 = 配置与模板数据

---

## 4. 当前项目最适合的抽象方式

## 4.1 不是继承，而是 behaviour + registry + event contract

推荐抽象方式：

### A. Behaviour
定义这些稳定接口：
- `SkillBehaviour`
- `NpcBrainBehaviour`
- `SceneRuleBehaviour`
- `SpawnPolicyBehaviour`

### B. Registry
由 registry 管理具体实现：
- `Gameplay.Registry.skills`
- `Gameplay.Registry.npc_brains`
- `Gameplay.Registry.scene_rules`
- `Gameplay.Registry.spawn_policies`

### C. Domain Event Contract
定义 gameplay runtime 产生的标准事件：
- actor_joined_scene
- actor_died
- actor_respawned
- skill_cast_resolved
- damage_applied
- npc_spawned
- npc_target_changed

### 为什么这样做比先上 Lua 更合理
因为它直接解决的是：
- 扩展点清晰
- 业务逻辑可替换
- kernel 稳定

而不是先增加一层脚本状态边界。

---

## 5. 当前仓库中应该如何落位

## 5.1 第一阶段不要马上大拆 umbrella app

当前最稳的方式是：

### 在 `scene_server` 内先做清晰分层
建议新增：

- `scene_server/gameplay/`
  - `behaviours/`
  - `registry/`
  - `events/`
- `scene_server/game_demo/`
  - `skills/`
  - `npc_brains/`
  - `content/`
  - `spawns/`

### 这样做的好处
- 不需要第一刀就重排整个 umbrella
- 风险小
- 现有 movement/combat/npc runtime 可继续保留
- 但边界会开始变清楚

---

## 5.2 kernel 中应保留的部分

这些模块仍应属于 kernel：

### scene_server
- `movement/`
- `combat/` primitive
- `npc/actor.ex`（runtime shell）
- `worker/` 中 player actor shell
- `aoi/`
- runtime event primitive
- native adapters

### gate_server
- transport
- protocol
- fast-lane
- fanout
- stdio/observe

### bevy_client
- protocol
- prediction/reconcile
- remote snapshot interpolation
- actor identity
- stdio/headless/E2E

---

## 5.3 该上移为 business/content 的部分

这些当前还偏硬编码，应逐步移出 kernel：

- `skill_id=1` 的具体业务定义
- `Training Slime` 的具体 brain
- demo spawn 逻辑
- scene composition
- drop / quest / dungeon-specific rules

---

## 6. 与 mlua / Lua 的关系（限主流程）

## 6.1 现在不建议直接引入到主流程 kernel

原因：
- 先要稳定 kernel 接口
- 现在还没到“脚本表达能力是主要瓶颈”的阶段
- 过早引入 Lua 会增加：
  - Elixir 状态
  - Rustler 状态
  - Lua VM 状态

这三层边界会明显增加：
- 调试成本
- 热更新一致性难度
- authority 保证成本

## 6.2 如果以后引入 Lua，推荐放到 business/content 层

适合：
- 地图脚本
- 副本事件脚本
- 任务脚本
- 活动脚本
- 高层 NPC 行为脚本

不适合：
- movement authority
- combat authority
- actor lifecycle core
- AOI core
- 高频 tick 主循环

### 最佳姿势
Lua：
- 输入上下文
- 输出 intent / command

Kernel：
- 做合法性校验
- authority 执行
- 状态落地
- 广播

---

## 7. 现在最值得先抽的几个边界

## Priority 1 - Skill Runtime Contract
当前 `Combat.Skill` 还偏硬编码。

应先抽成：
- kernel 的 skill runtime contract
- business 层的 skill catalog

## Priority 2 - NPC Brain Registry
让 `Training Slime` 从“默认实现”变成“注册到系统中的第一批 demo business brain”。

## Priority 3 - Scene Content / Spawn Content
让 `Demo.Runner` 里的 training slime spawn 不再代表 runtime 本身，而是：
- demo content
- spawn policy
- scene bootstrap data

## Priority 4 - Domain Event Layer
让玩法逻辑逐步挂在事件上，而不是侵入 actor 核心逻辑。

---

## 8. 推荐阶段计划

## Stage A - 抽 gameplay kernel 扩展点
输出：
- behaviours
- registry
- event types

## Stage B - 把当前 demo gameplay 迁移到上层
输出：
- demo skill catalog
- training slime brain
- demo spawn content
- demo scene rule

## Stage C - 内容驱动
输出：
- npc template registry
- scene content definition
- spawn groups
- wave data

## Stage D - 如果边界稳定，再考虑 package/app 继续拆分
例如：
- `apps/game_demo`
- `apps/game_pve`
- `apps/game_content`

---

## 9. 当前的最终建议

如果只给一句最重要的建议：

> **先把 gameplay kernel 的扩展点抽象成 behaviour + registry + event contract，再把 `Training Slime`、`skill_id=1`、spawn/content 这些现有业务逻辑迁移到上层 demo business；auth_server 暂时不动，Lua/mlua 也暂时不进主流程内核。**

这条路线最符合当前仓库现状，也最适合平滑演进成真正的“底座 + 业务”结构。
