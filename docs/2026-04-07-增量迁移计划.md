## 阶段 1：TCP 分帧 + 自定义二进制协议（P0）

**问题**：

1. `tcp_acceptor.ex` 监听 socket 时使用 `packet: 0`（raw 模式），在 TCP 分段/粘包时会导致 protobuf decode 失败。
2. `protox`（纯 Elixir protobuf）对高频游戏消息过慢。位置同步在 10–30Hz/玩家时，需要更轻的序列化开销。

**目标**：

- 先修正 TCP 分帧
- 再逐步把 protobuf 替换成自定义二进制协议
- 使用 Erlang binary pattern matching 作为最终高性能路径

**设计决策**：

所有消息长期都迁移到自定义二进制格式，不保留 protobuf 作为长期主路径。协议格式如下：

```text
线格式：<<length::32-big, msg_type::8, payload::binary>>
  - length：由 Erlang {packet, 4} 自动处理
  - msg_type：1 字节消息类型 ID
  - payload：各消息特定二进制布局（可尽量固定长度）
```

### 步骤 1.1：监听 socket 改为 `packet: 4`

- **文件**：`apps/gate_server/lib/gate_server/worker/tcp_acceptor.ex`
- **改动**：在 `:gen_tcp.listen/2` 中把 `packet: 0` 改为 `packet: 4`
- **验证**：`mix compile && mix test apps/gate_server/`
- **不变量**：保留 `:binary` 与 `active: true`；accepted socket 继承 listener 选项；`packet: 4` 下 `:gen_tcp.send/2` 自动带长度头

### 步骤 1.2：定义自定义二进制 codec 模块

- **文件**：`apps/gate_server/lib/gate_server/codec.ex`（新增）
- **改动**：定义消息类型常量与基于 binary pattern matching 的 encode/decode，先覆盖热点消息
- **验证**：对每种消息做 encode → decode roundtrip 测试，并校验字段值
- **不变量**：旧 protobuf 路径仍保留；codec 先独立存在，不立刻接到运行时

### 步骤 1.3：热点消息接到自定义 codec

- **文件**：`apps/gate_server/lib/gate_server/worker/tcp_connection.ex`、`apps/gate_server/lib/gate_server/message.ex`
- **改动**：`handle_info({:tcp, _, data}, state)` 按首字节判断是否走 `GateServer.Codec.decode/1`，否则回退到 `GateServer.Message.decode/1`；`send_data/3` 对热点消息改用 `GateServer.Codec.encode/1`
- **验证**：集成测试中直接发送自定义 movement 消息，确认服务端能处理并返回自定义二进制响应
- **不变量**：protobuf 仍可为其他消息服务；这是双协议过渡期

### 步骤 1.4：其余消息全部迁移到自定义 codec

- **文件**：`apps/gate_server/lib/gate_server/codec.ex`、`apps/gate_server/lib/gate_server/message.ex`
- **改动**：把 enter_scene、time_sync、heartbeat、广播类、结果类消息全部迁入自定义 codec
- **验证**：每种消息都有 roundtrip 测试；每个 dispatch 路径都有集成测试
- **不变量**：所有消息类型最终都能被自定义 codec 覆盖

### 步骤 1.5：移除 protobuf 依赖

- **文件**：
  - `apps/gate_server/lib/gate_server/message.ex`
  - `apps/gate_server/lib/gate_server/proto/`
  - `apps/gate_server/mix.exs`
  - `.gitmodules`（如果还保留 `mmo_protos` 作为参考，要重新评估）
- **改动**：所有编码/解码都走 `GateServer.Codec`，彻底移除 protox
- **验证**：`mix deps.get && mix compile && mix test`，确认仓库中不再有 protox 引用
- **不变量**：`GateServer.Message.dispatch/3` 仍负责路由，但输入从 protobuf struct 变为 decoded tuple

### 步骤 1.6：文档化线协议

- **文件**：`docs/2026-04-10-线协议规范.md`
- **改动**：文档化完整自定义二进制协议，包括：
  - 分帧格式
  - 消息头格式
  - 所有消息类型表
  - 每个消息的字节级字段说明
- **验证**：文档检查即可，无运行时验证需求

- **验证**：移动玩家 → 等 checkpoint → 杀掉进程 → 验证重启后位置恢复
- **不变量**：首次登录玩家仍可随机出生

### 步骤 5.6：恢复时重新关联 connection

- **文件**：`apps/scene_server/lib/scene_server/worker/player_character.ex`、`apps/gate_server/lib/gate_server/worker/tcp_connection.ex`
- **改动**：新 `PlayerCharacter` 启动后通知 `TcpConnection` 更新 `scene_ref`；当 `scene_ref` 为 nil 时在 `TcpConnection` 里做消息缓冲
- **验证**：杀掉 `PlayerCharacter` → `TcpConnection` 获得新的 `scene_ref` → 客户端继续能移动
- **不变量**：客户端不应被视为完全断线，只允许短暂停顿

## 风险评估

| 步骤 | 风险 | 缓解策略 |
|------|------|----------|
| 1.1（packet:4） | 客户端需要同步更新分帧行为 | 文档化协议变化；旧客户端可通过 decode 失败快速识别 |
| 1.3（双协议过渡） | 需要同时维护两条路径 | 尽量缩短 protobuf fallback 生命周期，在 1.4 尽快完成全量迁移 |
| 1.5（移除 protobuf） | 会破坏旧客户端兼容 | 必须在所有消息类型都完成迁移后再切掉 |
| 2.5（双写） | Ecto 写失败可能被忽略 | 增加 telemetry 与失败告警 |
| 2.8（移除 Mnesia 写） | 如果迁移不完整，可能丢数据 | 回填至少执行两轮，并做行数核对 |
| 3.5（迁移 Interface） | Horde registry 尚未完全稳定 | `BeaconServer.Client` 在过渡期保留 fallback |
| 4.6（zone handoff） | 玩家重复出现 | 使用 remove-from-old → add-to-new 的两阶段协议 |
| 5.6（connection 重新关联） | 恢复期间可能产生 race condition | 在 `scene_ref == nil` 时缓冲消息，恢复后重放 |
