# 架构设计指导思想：系统正交 + 自维护不变量

> 开工前必读。本文是项目级工程纪律，适用于服务端（Elixir/OTP）与所有客户端
> （bevy / Voxia UE / web）。每个设计稿、每次动手前，先用文末「开工前自查清单」过一遍。
> CLAUDE.md「架构设计指导思想」一节链接本文。

## 一、核心原则

1. **正交分解（Orthogonality）**
   每个系统只负责**一件清晰的事**。系统之间只通过**稳定契约**通信，**绝不共享可变的隐式假设**。
   目标：改一个系统不会「牵一发动全身」。判据：如果改 A 会莫名其妙弄坏 B，而 A、B
   在概念上本应无关，那就是出现了**隐藏耦合**，是设计缺陷。

2. **自维护不变量（Self-maintained invariants）**
   一个系统对外承诺的**所有**不变量，必须由它**自己持续维护**——**包括不显眼的时间性
   不变量**（活性 / 续租 / 超时 / 心跳 / 重连）。
   **绝不**让别的系统的正确性，悄悄依赖一个**没人主动维护**的假设。
   常见漏点：「一次性建立、永不刷新」的资源（订阅、lease、连接、缓存），它们会随时间
   静默失效，而最初建立它的代码以为「建好就一直在」。

3. **显式契约 > 隐式假设**
   跨系统依赖必须是**显式、被强制**的契约。两条路任选其一：
   - **承诺方强制维护**：subscription 保证「在场期间订阅始终活着」，edit 只管发编辑。
   - **依赖方对破坏鲁棒（自愈）**：edit 发现「编辑后该回的 delta 没回」就触发重新同步。
   最糟的是第三种：**依赖方默默假设契约成立、承诺方又没维护**——这正是下面的反面案例。

## 二、反面案例：挖放体素失效（2026-06-27，Voxia）

- **现象**：玩家挖/放方块，服务端 `VoxelIntentResult` 回 `accepted`、chunk_version 在涨
  （服务端确实改了），但客户端**看不到任何变化**。客户端 spawn 后收到 **0 个 0x62/0x63**。

- **最初的错误归因（我犯的）**：「移动后区块订阅没更新」——暗示**移动**与**挖放**之间有耦合。
  **这是错误且不精确的诊断。** 用户的反问完全正确：订阅只管「玩家位置变 → 订阅哪些区块」，
  挖放只管「改当前订阅区域的体素」，这两者本就该正交，凭什么「移动了导致挖放失效」？

- **真因**：near-chunk 订阅是 **spawn 时一次性发出**的，**没有任何活性维护**。服务端 lease
  有 TTL；玩家**站着不动也会**在 TTL 后过期 → ChunkProcess 丢掉订阅者 → 不再向其推
  ChunkDelta(0x63) → 编辑应用在服务端却对客户端不可见。**移动是红鲱鱼**——18 分钟不动也会坏。

- **正交性教训**：
  1. 订阅系统其实有**两个正交职责**：
     - **覆盖（coverage）**：位置驱动——玩家移动 → 订阅新区块。
     - **活性（liveness）**：时间驱动——持续续租，防 lease 静默过期。
     原代码只隐含了「一次性覆盖」，**完全漏了活性**。连「订阅 = 位置驱动的覆盖」这个朴素
     心智模型本身都是**不完整**的——它没意识到活性这一维。
  2. 挖放系统的可见性，**隐式依赖**「订阅活着」这个**没人维护**的假设。这就是隐藏耦合：
     edit 的正确性悄悄挂在 subscription 的一个未维护不变量上。两个本应正交的系统，被一条
     隐式假设串在了一起。
  3. **正确设计**：subscription 自己保证「在场期间订阅始终活着」（自维护活性不变量），
     edit 只依赖稳定契约「订阅区块的变更必回推」，**无需知道订阅机制**。这才真正正交。

- **修复**：`MaybeRefreshSubscription` 让订阅系统**自维护**——跨 chunk 重订阅（覆盖）+ 周期
  keep-alive 重订阅（活性，续租 + 重发当前快照）。edit 一行没动、也不感知订阅。

- **仍未决（需团队拍板）**：活性该谁拥有？客户端轮询续租（现状，band-aid）vs **服务端按连接
  活性维护订阅**（更正交——连接活则订阅活，无需客户端轮询，也不会有 10s 重订阅的重网格代价）。
  当前选客户端侧是因为不动服务端协议；长期应评估服务端方案。

## 三、开工前自查清单

动手改/写任何系统前，先回答：

1. **这个系统对外承诺哪些不变量？** 有没有**时间性**的（活性 / 不过期 / 续租 / 重连）？谁维护？
   有没有「一次性建立、之后再没人管」的资源？
2. **我依赖别的系统的什么行为？** 那是**显式契约**还是**隐式假设**？如果那个假设被打破
   （对方失效、超时、重启），我会怎样？我能否自愈，还是会静默坏掉？
3. **我的改动会不会让两个本应正交的系统产生耦合？** 改 A 会不会莫名弄坏概念上无关的 B？
4. **诊断 bug 时**：先问「这个故障**跟我以为的那个原因真的有因果吗**」。挖放失效那次，
   「移动」与故障毫无因果（站着不动也坏），却差点被当成根因。**红鲱鱼要排除，别顺着第一直觉。**

  不反向写、不本地模拟。
