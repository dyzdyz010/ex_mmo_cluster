# 2026-04-17 服务器部署手册（ex_mmo_cluster — Docker Compose）

对应架构文档：`docs/2026-04-17-上线-Docker-CI-部署方案.md`

## 0. 前提

- 一台 Ubuntu 22.04 或 24.04 的阿里云/国内服务器（≥ 4C 8G）
- 宿主机已装 **nginx**（用于 HTTPS 反向代理；本手册不重复安装步骤）
- 已解析到该服务器的域名 `$DOMAIN`，一条 A 记录即可（如 `mmo.hemifuture.cn`）
  - HTTPS（auth_server）走 `$DOMAIN:443`
  - TCP 游戏网关（gate_server）走 `$DOMAIN:${GATE_TCP_PORT}`（也可直接用 IP）
  - visualize_server（20001）**不暴露公网**，运维用 SSH 隧道访问
- 服务器安全组放通：`80/tcp`、`443/tcp`、`${GATE_TCP_PORT}/tcp`（默认 20002）、`${GATE_UDP_PORT}/udp`（默认 20003）
- 阿里云 ACR 账号已可登录（CI 推送用的相同凭证即可）

## 1. 首次初始化

### 1.1 装 Docker 与 compose 插件

```bash
# Ubuntu 官方源
sudo apt-get update
sudo apt-get install -y ca-certificates curl
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
    -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc
echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] \
    https://download.docker.com/linux/ubuntu $(. /etc/os-release && echo $VERSION_CODENAME) stable" | \
    sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
sudo apt-get update
sudo apt-get install -y docker-ce docker-ce-cli containerd.io \
    docker-buildx-plugin docker-compose-plugin

# 当前用户加入 docker 组（避免每次 sudo）
sudo usermod -aG docker $USER
newgrp docker
```

> 若网络访问 docker.com 不畅，可切阿里云镜像源
> `https://mirrors.aliyun.com/docker-ce/linux/ubuntu`。

### 1.2 登录阿里云 ACR

```bash
docker login registry.cn-hangzhou.aliyuncs.com
# Username: aliyun@xxxxxxxxxx（ACR 独立访问用户名）
# Password: ACR 独立密码（NOT 阿里云主账号密码）
```

凭证写入 `~/.docker/config.json`，后续 `docker compose pull` 自动复用。

### 1.3 准备部署目录

```bash
sudo mkdir -p /data/ex_mmo_cluster
sudo chown $USER:$USER /data/ex_mmo_cluster
cd /data/ex_mmo_cluster

# 拉仓库中 deploy/ 下的部署文件
git clone --depth=1 --sparse <GITHUB_REPO_URL> repo
cd repo
git sparse-checkout set deploy
cd /data/ex_mmo_cluster
cp -r repo/deploy/* .
rm -rf repo

ls -la
# 应当能看到：docker-compose.yml  .env.example  nginx.conf.example
```

> 若不想 git clone 整库，也可以直接 `scp` 三个文件上来。

### 1.4 配置 `.env`

```bash
cp .env.example .env
vi .env
```

**必填**：

| 变量 | 说明 | 生成命令 |
|------|------|---------|
| `IMAGE_TAG` | 完整镜像 URL | `registry.cn-hangzhou.aliyuncs.com/<NS>/ex_mmo_cluster:latest` |
| `WEB_CLIENT_IMAGE_TAG` | 可选；网页客户端静态资源镜像 URL | `registry.cn-hangzhou.aliyuncs.com/<NS>/ex_mmo_web_client:latest` |
| `DOMAIN` | 裸域 | `example.com` |
| `PHX_HOST` | Phoenix URL host（与 `DOMAIN` 一致） | `mmo.example.com` |
| `SECRET_KEY_BASE` | 64+ 字符 | `openssl rand -base64 64 \| tr -d '\n'` |
| `RELEASE_COOKIE` | 节点 cookie | `openssl rand -base64 48 \| tr -d '\n'` |
| `MMO_DB_PASSWORD` | DB 密码 | `openssl rand -base64 24 \| tr -d '\n'` |
| `SCENE_SERVER_COUNT` | scene server 容器数量 | 默认 `1`；按机器容量调大 |

**锁定权限**：

```bash
chmod 600 .env
```

### 1.5 配置宿主 nginx

证书用 **acme.sh** 签发并安装到 `/etc/nginx/ssl/$DOMAIN/cert.pem` + `key.pem`。
模板默认就指这两个路径，无需调整。

```bash
# 1) 首次签发 + 安装（一次性；acme.sh 会注册 cron 每天 00:00 自动续期）
#    具体的 --issue 参数取决于 DNS/HTTP 校验方式，略
sudo acme.sh --install-cert -d $DOMAIN \
    --cert-file      /etc/nginx/ssl/$DOMAIN/cert.pem \
    --key-file       /etc/nginx/ssl/$DOMAIN/key.pem \
    --fullchain-file /etc/nginx/ssl/$DOMAIN/fullchain.pem \
    --reloadcmd      "service nginx reload"

# 2) 确认证书落地
sudo ls -l /etc/nginx/ssl/$DOMAIN/
# 应看到 cert.pem / key.pem / fullchain.pem；key.pem 权限 600，owner root

# 3) 把模板拷到 nginx 配置目录
sudo cp /data/ex_mmo_cluster/nginx.conf.example /etc/nginx/sites-available/ex_mmo_cluster.conf
sudo sed -i "s/\${DOMAIN}/$DOMAIN/g" /etc/nginx/sites-available/ex_mmo_cluster.conf
sudo ln -sf /etc/nginx/sites-available/ex_mmo_cluster.conf /etc/nginx/sites-enabled/

# 4) 验证 + reload
sudo nginx -t
sudo systemctl reload nginx
```

> 续期：`acme.sh --install-cert` 同时会在 root crontab 注册每日任务，续到期后
> 自动刷新 `/etc/nginx/ssl/$DOMAIN/*.pem` 并执行 `--reloadcmd`（即 `service nginx reload`），
> 无需手动干预。若改回 Let's Encrypt+certbot，参考 `nginx.conf.example`
> 顶部注释里的 `/etc/letsencrypt/live/...` 路径替换。

### 1.6 启动容器

```bash
set -a; . ./.env; set +a # 将 SCENE_SERVER_COUNT 导入当前 shell
docker compose pull      # 从 ACR 拉镜像（首次耗时）
docker compose up -d --scale scene=${SCENE_SERVER_COUNT}
docker compose ps        # 应看到 postgres + app + scene 都 running
docker compose logs -f app
```

`SCENE_SERVER_COUNT` 必须至少为 `1`：`app` 内的 World/Gate 启动路径会等待
`:scene_server` 注册。`app` 容器启动 edge/world/data 服务；`scene` 服务使用同一镜像里的
`ex_mmo_scene` release，只启动 scene runtime。调整 `SCENE_SERVER_COUNT` 后重新执行
`docker compose up -d --scale scene=${SCENE_SERVER_COUNT}` 即可增减 scene server 数量。
`scene` 服务不绑定宿主机端口，避免多副本争抢端口；它只通过 Compose 内部网络加入
Erlang/Horde 集群。

每个 `scene` 容器都会启动自己的 `DataService.Repo` 连接池；扩容时按 `app + scene_count`
估算数据库连接数，必要时降低 `MMO_DB_POOL_SIZE` 或提高 Postgres `max_connections`。

### 1.7 跑数据库 migration

首次启动后，pg 已经 ready，但库里是空的。在 app 容器内执行：

```bash
docker compose exec app /app/bin/ex_mmo_cluster eval \
    'DataService.Release.migrate()'
```

`DataService.Release.migrate/0` 已随 release 打包，内部使用 `Ecto.Migrator`，不依赖 Mix。

### 1.8 验证

```bash
# HTTPS 健康（证书在 1.5 步骤已就位）
curl -I https://$DOMAIN

# TCP 连通（游戏网关）
nc -v $DOMAIN $GATE_TCP_PORT
# 或从客户端尝试连接

# Visualize 控制台（通过 SSH 隧道访问）
ssh -L 20001:127.0.0.1:20001 user@$DOMAIN
# 浏览器打开 http://localhost:20001
```

## 2. 日常升级（最常用）

每次 CI 把新镜像推到 ACR 后：

```bash
cd /data/ex_mmo_cluster
./upgrade.sh
docker compose logs -f app
```

`upgrade.sh` 会拉取 `IMAGE_TAG` 对应的 app/scene 镜像、确保 postgres 不被重建、执行 release
migration、按 `SCENE_SERVER_COUNT` 更新 app/scene。若 `.env` 中设置了 `WEB_CLIENT_IMAGE_TAG`，
脚本还会拉取独立的网页客户端静态资源镜像，将其中 `/usr/share/nginx/html` 替换到
`WEB_CLIENT_DIST_DIR`（默认 `/data/ex_mmo_cluster/web_client/dist`），供 nginx `/client/`
alias 生效。旧客户端目录会备份到 `/data/ex_mmo_cluster/backup/web_client-dist-<timestamp>`。

**停机时长**：业务容器约 5-15 秒（旧容器 stop + 新容器 start + BEAM 初始化）。网页客户端替换是
宿主机静态目录切换，通常是毫秒级。

如果要同时更新 compose 配置（docker-compose.yml / .env）或宿主 nginx 配置：

```bash
# 先拉最新部署文件（按 1.3 的步骤重做，或手动编辑）
docker compose up -d           # 会 recreate 变化的服务
```

## 5. 常见问题

| 症状 | 原因 | 处置 |
|------|------|------|
| `docker compose pull` 401 | ACR 登录过期 | `docker login registry.cn-...` 重新登录 |
| nginx 启动报 `cannot load certificate` | `/etc/nginx/ssl/$DOMAIN/{cert,key}.pem` 路径 / 权限不对 | `ls -l /etc/nginx/ssl/$DOMAIN/`，确认两个文件存在且 nginx 可读 |
| 浏览器 SSL 报"证书过期" | acme.sh cron 未跑或 `--reloadcmd` 失败 | `sudo acme.sh --list` 看续期时间；手动触发 `sudo acme.sh --renew -d $DOMAIN --force` |
| app 启动崩 `SECRET_KEY_BASE missing` | `.env` 未正确加载 | 确认 `env_file: .env` + 文件存在 |
| Phoenix 404 / 502 | nginx 反代到了错的端口 | `curl -v http://127.0.0.1:20000` 直通测试，再核对 `nginx.conf` 的 `proxy_pass` |
| TCP 20002 或 UDP 20003 连不上 | 安全组 / compose `ports:` 未声明 | `docker compose ps` 看端口映射 |
| scene 容器没有加入集群 | `DISABLE_CLUSTER` / `RELEASE_DISTRIBUTION` 仍是旧单容器值 | `.env` 应为 `DISABLE_CLUSTER=false`、`RELEASE_DISTRIBUTION=name`，并用 `--scale scene=${SCENE_SERVER_COUNT}` 重建 |

## 6. 备份

### 6.1 PostgreSQL

```bash
# 手动快照
docker compose exec postgres pg_dump -U $MMO_DB_USER $MMO_DB_NAME \
| gzip > /data/ex_mmo_cluster/backup/pg-$(date +%Y%m%d-%H%M).sql.gz

# crontab 每日凌晨
0 4 * * * docker compose -f /data/ex_mmo_cluster/docker-compose.yml exec -T postgres \
pg_dump -U ex_mmo_cluster ex_mmo_cluster_prod | gzip > /data/ex_mmo_cluster/backup/pg-$(date +\%Y\%m\%d).sql.gz
```

### 6.2 TLS 证书

acme.sh 签发的证书部署到 `/etc/nginx/ssl/$DOMAIN/{cert,key,fullchain}.pem`，
原始签发数据在 `~/.acme.sh/$DOMAIN/`（含账号 key、csr、历史）。服务器换机或灾备：

```bash
# 部署后的 nginx 证书目录
sudo tar czf /data/ex_mmo_cluster/backup/nginx-ssl-$(date +%Y%m%d).tgz \
    -C /etc/nginx ssl

# acme.sh 原始数据（含账号私钥；恢复后 cron 续期能直接接上，不用重新签发）
sudo tar czf /data/ex_mmo_cluster/backup/acme-$(date +%Y%m%d).tgz \
    -C /root .acme.sh
```

恢复：解包两个 tar；确认 `key.pem` 权限 600 / owner root；执行
`sudo systemctl reload nginx`。`.acme.sh` 里的 cron 条目需在新机上 `sudo acme.sh --install-cronjob` 补一次。

> acme.sh 自带每日续期 cron，**不要**同时装 certbot 抢同一个域名的 port 80 验证，会相互踩。

## 7. 安全 checklist（首次上线前）

- [ ] `.env` 权限 600，只 root/部署用户可读
- [ ] 服务器 SSH 关掉密码，只留 key 登录
- [ ] 安全组最小开放（80/443/${GATE_TCP_PORT}），禁止 5432 公网
- [ ] `visualize_server` 未反代到公网 nginx（方案 A：只绑 `127.0.0.1:20001`，运维走 SSH 隧道访问）
- [ ] Phoenix 的 admin / dashboard 路由确认未在生产暴露
- [ ] `docker compose logs` 中确认没有 leak 环境变量
- [ ] 第一次上线后 `curl -I https://$DOMAIN` 返回 200 / 404，不是 5xx
