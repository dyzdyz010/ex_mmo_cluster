# ex_mmo_cluster 架构与实现质量审查报告

- 日期:2026-06-04
- 分支:`codex/mmo-architecture-foundation`(审查基于工作区当前代码)
- 范围:整个 umbrella(13 个 app,约 7 万行 Elixir + 7 千行 Rust NIF)
- 方法:8 个专家审查域并行初筛(共 83 个问题)→ 对每域 critical/high 做对抗性"证伪"核验(确认成立 33 个)→ 首席架构师综合。所有 critical/high 结论均经第二轮独立读码核验。

## 一、总体评级

| 维度 | 评级 | 一句话判断 |
|---|---|---|
| 架构质量 | **C+** | 边界与可观测性的"意图"是 A 级,但分布式容错、进程身份/生命周期、并发模型三类"机制不变量"系统性缺位。 |
| 实现质量 | **B- / C+(两极分化)** | `movement_core`/`codec`/体素存储的乐观并发与持久化校验是 A 级;但热点路径串行化、2PC 驱动/恢复、auth 密码绕过、UID 碰撞拉低了均值。 |

**核心判断:这是一个架构意图清晰、局部实现精良、但分布式与并发"机制层"系统性欠账的项目。** 它当前停留在"单节点能跑、多节点 + 崩溃恢复语义未闭合"的阶段。不要急于扩功能,先把四条机制主轴补齐(见第六节路线):**进程身份注册化、持久化/阻塞移出热路径、协调权威内生 liveness、节点故障接入事务/租约生命周期**。

### 真实存在的强项(不是客套)

- **权威边界落到了代码,而非只写在 moduledoc**:权威只在服务端、`ChunkProcess` 拥有 chunk 真相、`lease + owner_epoch + owner_scene_instance_ref` 四元组围栏在 `apply_intent`/`persist`/`init` 三处一致执行,客户端/旧持有者结果无法成为权威。
- **`movement_core` 是教科书级实现**:纯 f64、固定步长、per-tick 位移/速度/jerk 全部有界、speed_scale 服务端 clamp,配 100k-tick 确定性 + 无 NaN + 镜像对称不变量测试,并与 bevy 客户端预测共用,杜绝双端漂移。历史"越拉越远"外推失控在服务端积分层已被结构性消除。
- **wire codec 单一真相源**:`GateServer.Codec` 把字节布局收口,`@protocol_version`/`@movement_wire_schema` 提供握手协商与逐帧 schema 守卫。
- **持久层 Ecto 主路径扎实**:`ChunkSnapshotStore` 用 `pg_advisory_xact_lock` + `FOR UPDATE` + CAS 版本比较,`binary_to_term(_, [:safe])` 防 atom 表溢出,约束下沉到 migration。
- **几乎全路径结构化 `observe` 日志**,`CliObserve` 未启用时短路开销极低,符合"可观测优先"硬边界。

## 二、七大系统性问题(机制层根因 —— 最重要的输出)

这些不是孤立 bug,而是同一种"功能正确但缺少机制不变量"的思维在多处复现。每一条都跨多个域。

### S1【P0】权威进程身份寄生于易失内存 map,与监督树/服务发现脱钩 —— 崩溃即丢真相或产生双权威
同一反模式在 5 处复现:`ChunkDirectory.state.chunks`、`AoiManager.state.aois`、`SceneNodeRegistry.state`(region_assignments/join_order)、`PhysicsManager` 持有的全局 `PhySys`、octree `ResourceArc`,都用普通内存结构当唯一注册表,既无 `Process.monitor`/`:DOWN`,也不在 Registry/Horde.Registry 注册,更不与 `DynamicSupervisor` 的 `:permanent` 重启语义对接。
**后果**:(a) `ChunkProcess` 崩溃重启后以空 `Storage` 起步且目录仍指旧 pid → 同一坐标出现两个权威进程(split-brain);(b) `AoiManager` 重启造出新空 octree,存活 `AoiItem` 仍持旧树句柄 → AOI 静默错乱;(c) `SceneNodeRegistry` 崩溃丢全部 region 归属。
**根因**:进程身份(谁是权威)与监督身份(谁负责重启)放在两个不同步的地方,任一重启即发散。
**机制修复**:以资源 key 注册到 Registry/Horde.Registry(via-tuple 写进 child_spec 的 start 参数,让重启天然去重),把"重启即从权威存储 hydrate"设为 `init` 不变式,目录/管理者退化为无状态 facade。
波及域:voxel-storage、movement-combat、scene-rust、cluster-otp、world-coordination。

### S2【P0】用"全局可变单例 + try_lock + 上层 `Process.sleep` 忙等"冒充并发
`scene_ops` 全部 NIF 对单个全局 `Mutex<PhySys>` 做 `try_lock`,失败返回 `:lock_fail`,Elixir 侧 `*_with_retry` 在 GenServer 回调里 `Process.sleep(5ms)×5` 忙等;最坏 50ms 阻塞远超 16ms tick 预算,退避耗尽后**静默丢 tick(写)或静默回退缓存位置(读)**,把所有玩家物理读写并发度锁死为 1。同源问题在 `ChunkDirectory`(handle_call 内同步下游 + 同步 Postgres)、`AoiManager`(全场景 self_move 同步 call 单进程)、`FastLaneRegistry`(每帧出站同步 call + 每 call O(N) 扫描 = O(N²))重复出现。叠加 **NIF 全无 `DirtyCpu` 标注**,rapier step、octree 递归、field Dijkstra、rayon 直接阻塞普通调度器。
**机制修复**:读写分离(只读走 ETS/`RwLock` 快照,不进写串行结构)、按空间分片所有权、删除回调内 sleep 改异步重排、CPU-bound NIF 标 `DirtyCpu`。
波及域:scene-rust、movement-combat、voxel-storage、gate-connection。

### S3【P0】阻塞/长时工作被塞进延迟敏感进程的同步路径 —— 单个慢下游冻结整条链路
连接进程在 `handle_info` 内对 Scene 做 per-chunk **同步 15s call**(分区刷新 apply),且 socket 全程 `active: true` 无流控 → 慢 Scene 节点可让单连接冻结至 15s,mailbox 无界堆积;战斗结算在 actor 的 `handle_info` 内对目标 actor 同步 `GenServer.call` 互调 → PvP/双 NPC 互攻触发**两进程环形等待死锁**(10s 超时才解开);`ChunkDirectory` 在 handle_call 内同步 persist 到 Postgres 阻塞全目录。
**根因**:权威写本质是单向命令,却被建模成同步请求-应答;延迟敏感角色(socket owner、actor tick)与对慢下游敏感的角色挤在同一进程同一同步路径。
**机制修复**:权威写改单向 `cast` + 状态快照(ETS)只读;订阅/碰撞/持久化移出热路径同步段;socket 改 `active: N` + 在途请求数背压。
波及域:gate-connection、movement-combat、voxel-storage、world-coordination。

### S4【P0】分布式协调/恢复缺少"内生 liveness 与故障联动"
`TransactionCoordinator` 是纯被动状态机,写了 `timeout_at_ms` 却**全程无人消费**(无 `send_after`/周期 sweep 驱动超时);推动 prepare→commit/abort 的 `Executor` **跑在 gate 连接进程里**(与客户端生命周期绑定),客户端断线即事务永久悬挂在 `:prepared`;唯一兜底 `TransactionRecoveryWatcher` 只在 init/重启时 **one-shot 扫一次**,运行期(world 不重启)悬挂事务永不恢复;`commit_decision` 在 participant commit 失败时仍被**无条件记录**(破坏全局原子性);`SceneNodeMonitor` 收到 `:nodedown` 只更新路由成员表,不回收租约、不中止在途事务,产生孤儿 fence(scene 侧 fence 还无 TTL)。
**根因**:把"推进到终态"与"故障检测"当成外部偶发动作而非协调权威的内生职责;节点存活信号(Erlang VM 权威源)只接到路由成员管理一个面,没接到事务与租约两个一致性面。
**机制修复**:per-transaction 受监督 driver(归 world 节点)+ coordinator 内建 deadline 调度消费 `timeout_at_ms` + commit_ack 对称确认 + 独立 fence 对账回路 + membership 事件驱动的 epoch 失效/failover。
波及域:world-coordination、cluster-otp、voxel-field、voxel-storage。

### S5【P1】持久化/全量计算在热路径上同步且无界 —— 缺乏"活跃工作集 vs 历史"与"增量 vs 全量"分离
协调者四张 map 永不裁剪 + 每次状态变更同步 `term_to_binary` 全量历史单行 upsert;体素热路径每改一格就**全量 `normalize!`(4096 header 约 10+ 趟)+ `refresh_chunk_object_refs`(O(headers × `Enum.at` refined_cells))+ 两次全量 snapshot encode**,canonical 用 `List` 做 4096 元素随机写;field kernel 的 `clear_layer_in_aabb` 每 10Hz tick 对全 AABB 做 O(N) 稠密写(与稀疏层设计自相矛盾);`ChunkProcess` 永不回收 + 永久 10Hz 空转 timer,进程数随探索无界增长。
**机制修复**:活跃集与历史物理分离 + 行级增量持久化 + `normalize` 收口到信任边界 + `macro_headers` 改 `:array`/map + delta/rebuild 语义替代"清空再写" + chunk 空闲驱逐与按需 tick。
波及域:world-coordination、voxel-storage、voxel-field、data-and-aux。

### S6【P1】逻辑收口只做到 wire 层,gameplay 路由/分发在传输端逐字复制并已实际漂移
`codec` 把"字节如何编码"收口了(做对了),但"已解码消息如何分发/鉴权/路由/写回"被复制成 `tcp_connection.ex`(3774 行)与 `ws_connection.ex`(3824 行)两份同构副本,`movement_ack` 组帧又第三次复制进 `udp_acceptor`。**已发生真实漂移**:`voxel_field_conduct_intent` / `partition_bootstrap_retry` 只在 WS 端有 handler,同一客户端走 TCP 会被静默丢弃;observe 事件名也出现 `ws_` 前缀分叉。同类"双份真相靠人工同步"还出现在 field 物理权重常量在 `.ex` 与 `.rs` 各写一份。
**机制修复**:抽 `ConnectionCore` behaviour + transport 回调薄壳;用 codec message-type 枚举做"每个可解码类型必有 handler"的 CI 断言;共享常量单一来源(`build.rs` 生成)。
波及域:gate-connection、voxel-field、scene-rust。

### S7【P2】迁移债/未完成模块以"正常 OTP 应用"身份共存,且 dev/test 与生产 release 启动集合漂移
`data_store`/`data_contact` 两个 Mnesia 遗留 app 仍作为完整 OTP 应用启动并向 BeaconServer 注册真实资源(生产 release 白名单已排除,但 dev `iex -S mix` 全启,形成启动集合漂移);`DataService.Dispatcher` 是 Mnesia 时代遗留的"二段 GenServer 串联 poolboy",在 Ecto 自带连接池之上叠加冗余串行点(还留有 `conneddddddddddddddcted` 调试日志);`agent_server.Agent` 是携带 `:pg` 滥用与 `Process.put` 副作用的未实现占位;`visualize_server` 硬编码节点名 + 逐 PID 串行 call 拉位置。
**机制修复**:启动集合在所有环境单点收口、按"无迁移债"纪律彻底删除遗留孤岛(而非半启动打 warning)、auth 数据访问回归无状态 Repo 直调。
波及域:data-and-aux、cluster-otp。

## 六、优先级修复路线(P0 → P2)

> 原则:不打补丁,从机制主轴整体修复;P0 是上线前硬阻断。

1. **P0 — 安全闸门(数小时级,先堵)**:修复 auth 登录密码绕过(persist-1)。`Bcrypt.verify_pass` + `no_user_verify` 防时序;controller 先验证再签 token;密码框 `type=password`、删明文日志、token 改 POST。这是唯一可被立刻利用、影响游戏入场的 critical,改动面最小,优先级最高。
2. **P0 — 进程身份注册化(消除双权威与崩溃丢真相)**:为 `ChunkProcess` 以 `{logical_scene_id, chunk_coord}` 注册 Registry/Horde.Registry,via-tuple 写进 start 参数让重启天然去重;`init`(或带 ready 门闩的 `handle_continue`)以 `ChunkSnapshotStore.get_snapshot` hydrate 为不变式,区分 `:never_persisted` 与 `:loaded`,删除 `Storage.empty` 默认兜底;`ChunkDirectory` 退化为无状态 facade。同一手法套用到 `AoiManager`(octree 句柄移出易崩进程)与 `SceneNodeRegistry`(region 归属落库)。一次消解 voxel-storage-2、cluster-discovery-2/4、aoi-singleton 的重启孤儿/双主。
3. **P0 — 热路径去同步去阻塞(吞吐与死锁)**:删除 finalize 内冗余 rapier readback(权威位置已由 `movement_core`+`VoxelCollision` 算出),随之移除 PhySys 热路径加锁与回调内 sleep;战斗伤害改单向 `cast` + 目标 actor 单一 HP 权威 + AOI/ETS 只读快照供 targeting 读(消除环形死锁);连接进程把分区刷新 apply、体素碰撞、订阅编排移出 handle_info 同步段(每连接受监督 Subscription Worker + 短超时),socket 改 `active: N` + 在途请求数背压;`ChunkDirectory` 改 Registry 直达 + chunk 内 DB 异步化。
4. **P0 — 2PC 协调权威闭环(分布式正确性)**:`Executor` 驱动收回 world 节点的受监督 per-transaction driver(gate 只发起 + 订阅结果);coordinator 内建 deadline 调度消费 `timeout_at_ms` 自我推进到终态;commit 阶段引入 `commit_ack` 对称确认(decision 记录后进入 `:committing`,全 participant ack 才到 `:committed`,commit 失败绝不触发 abort 而是持续重投递);补 begin 阶段持久化 `intents_by_participant`;recovery_watcher 升级为 boot sweep + 周期 reaper + 独立 fence 对账(scene fence 加基于 coordinator deadline 的 TTL 自愈)。
5. **P1 — 节点故障接入一致性面 + NIF 调度**:引入统一 `ClusterMembership`(对齐 BeaconServer,带 incarnation/epoch),membership 事件驱动 `MapLedger` 对死节点 region 置 unreachable + advance owner_epoch + failover,事务侧 driver monitor participant 节点 `:DOWN` 主动 abort;为 CPU-bound NIF(rapier step、octree、field Dijkstra)加 `DirtyCpu` 并加 CI lint,octree insert 改 `upgradable_read` 消 TOCTOU。
6. **P1 — 性能/内存机制(防长跑退化)**:协调者活跃工作集与历史物理分离 + 行级增量持久化;体素 `normalize` 收口到信任边界、`macro_headers` 改 `:array`/map、object_refs 绑定 dirty_bounds 增量、snapshot 全量 encode 仅按需;`ChunkProcess` 空闲驱逐(目录 `terminate_child`,restart 改 `:transient`)+ 按需 tick;field clear 改过滤式 + rebuild 语义;修 `client_ack_ledger` 退订漏清理。
7. **P1 — 逻辑收口与契约防漂移**:抽 `ConnectionCore` behaviour + transport 薄壳,三端共用 `MovementFrames`;用 codec message-type 枚举做"每个可解码类型必有 handler"的 CI 断言(立即修 `field_conduct`/`bootstrap_retry` 的 TCP 缺失)与双端 parity 测试;field 物理权重常量改共享单一来源(`build.rs` 生成)。
8. **P2 — 迁移债清理与正确性补强**:按"无迁移债"纪律彻底删除 `data_store`/`data_contact`/`Dispatcher` 遗留孤岛,启动集合在所有环境单点收口(dev/test 与 release 一致),auth 数据访问回归无状态 Repo 直调;修 `UidGenerator` 三态逻辑(同毫秒递增/跨毫秒重置/回拨告警)+ service_id 集群唯一 + 溢出自旋;`Character` 加外键约束;`visualize_server` 改服务发现 + 异步/批量拉位置。

## 七、方法论与局限

- **覆盖**:8 域覆盖全部 13 app + 全部 Rust crate。每域初筛后,对 critical/high 各最多取 6 个(critical 优先)做对抗性"证伪"核验;`voxel-storage` 有 1 个 high 因上限未对抗验证(已保留在报告)。medium/low 共约 50 个未逐一二次核验,仅作参考。
- **置信度**:第三节 9 个 critical 与第四节 high 均经独立第二轮读码确认成立;核验过程对若干原始判断做了修正(如 voxel-storage-1 的碰撞预算实际是 50ms 而非 1s,危害比初判更大;`Task.start_link` 与 chunk 共命运反而放大丢失概率)。
- **审查者独立基线**:报告作者另行抽样核对了 `transaction_coordinator.ex`(确认 durable persistence + 幂等设计成熟,弱点在故障恢复路径而非正常路径)与 `scene_ops/src/lib.rs`(独立确认全 NIF 无 `DirtyCpu` + `try_lock` 设计),与 agent 结论一致。
- **未覆盖**:运行时性能实测、压测下的真实 mailbox 堆积曲线、netsplit 实景演练、CI 实际通过情况、前端 `clients/web_client` 端到端 parity —— 这些需要在 P0 修复后用 CLI observe + 压测验证(符合本仓"CLI 可观测 + 结构化日志优先"约定)。
- **一处需澄清的统计**:`cluster-discovery-2` 与 `voxel-storage-2`、`scene-physics-mutex-1` 与 `scene-rust-2` 是同一根因被两个审查域独立交叉确认,清单中分别列出以保留各自视角,实际为 7 个独立 critical 根因。
