# 2026-05-22 Web 客户端真 3D 移动同步实测记录

本文记录本轮 Web 客户端移动同步升级后的用户可见结果、当前阶段目标、已完成验证和后续任务。它用于接续后续移动/碰撞工作，避免结论只留在对话上下文中。

## 1. 用户可见结论

当前 Web 客户端已经不再把角色高度交给渲染层按体素列临时推导。

用户现在可以看到：

- 玩家头顶有方块时，角色不会被瞬间抬到上方方块表面。
- 本地跳跃会真实表现为 Y 轴升高再落地。
- 服务端权威回执中的跳跃高度会进入显示链路，不会被客户端渲染层覆盖。
- 另一个网页客户端可以看到远端玩家跳跃，包括 airborne 状态和 Y 轴插值过程。

这意味着当前问题的核心边界已经收口：角色显示高度来自移动同步状态，而不是来自“当前位置 X/Z 上最高实体方块”的渲染期猜测。

## 2. 本轮阶段目标

本轮目标不是扩展新玩法，而是修复移动同步架构中的高度真相源问题：

1. 让 local / authoritative / remote 三类角色显示都读取同步后的 3D 位置。
2. 保留体素表面查询作为出生点、传送点等“落点选择”能力。
3. 禁止渲染层在每帧根据体素列重新决定角色 Y 坐标。
4. 用真实浏览器和真实 WebSocket 验证头顶方块、单客户端跳跃、双客户端远端跳跃三条用户路径。

## 3. 实测证据

### 3.1 头顶方块不会再抬升角色

测试方式：启动真实网页客户端，在玩家当前 X/Z 正上方放置普通方块。

结果：

- 放块前本地 Y：`100`
- 放块后本地 Y：`100`
- 放块后权威 Y：`100`
- 结论：通过

### 3.2 单客户端跳跃能显示 3D 高度

测试方式：真实网页客户端执行 `jump`，采样本地显示高度、权威回执高度和 frame trace。

结果：

- 起跳前 Y：`100`
- 本地显示最高 Y：约 `213.04`
- 权威回执最高 Y：约 `196`
- airborne trace 数量：`80`
- 结束后本地 Y：`100`
- 结束后权威 Y：`100`
- 结论：通过

### 3.3 双客户端能看到远端玩家跳跃

测试方式：同一浏览器自动化会话打开两个真实网页客户端。A 客户端执行 `jump`，B 客户端采样 A 的远端实体。

结果：

- 远端初始 Y：`100`
- 远端最高 Y：约 `197.3`
- 远端 airborne 采样数：`8`
- 采样丢失远端实体次数：`0`
- 代表性采样：
  - `t=100ms y=102.5 mode=airborne`
  - `t=200ms y=148.4 mode=airborne`
  - `t=500ms y=197.3 mode=airborne`
  - `t=900ms y=119 mode=grounded`
  - `t=1000ms y=100 mode=grounded`
- 结论：通过
