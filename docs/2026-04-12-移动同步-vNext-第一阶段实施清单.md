# 2026-04-12 移动同步 vNext 第一阶段实施清单

## 0. 第一阶段目标

第一阶段不是一次性做完整移动系统，而是把 **vNext 的骨架与最关键闭环** 搭起来：

1. 协议从“位置提议”切到“输入流 + 权威 ack”
2. 客户端建立本地预测与回滚/重演骨架
3. 服务器建立权威 movement engine 门面
4. 表现层与模拟层彻底分离
5. 预留 Rustler 计算边界，但第一阶段可先用 Rust 客户端本地实现 + Elixir façade 组织服务端

第一阶段完成标准：
- 本地玩家走 `MoveInputFrame -> MoveAck -> rollback/replay`
- 远端玩家仍可用现有链路，但协议上开始切入 `server_tick`
- 当前 stdio / E2E harness 仍然可用
- 模块边界明确，文件不过载

## 3. 第一阶段实施顺序

### 客户端新增结构

#### `MoveInputFrame`
```rust
pub struct MoveInputFrame {
    pub seq: u32,
    pub client_tick: u32,
    pub dt_ms: u16,
    pub input_dir: Vec2,
    pub speed_scale: f32,
    pub movement_flags: u16,
}
```

```rust

    pub tick: u32,

    pub velocity: Vec3,

}
```

## Step 4 — 客户端 reconciliation / replay 骨架

### 新建
- `sim/reconcile.rs`

### 内容
#### `ReconcileEngine`
核心接口：
- `apply_ack(ack, input_history, predicted_history)`

流程：
1. 删除 `<= ack_seq` 输入
2. 找到 `auth_tick`
3. 比对 predicted 与 authoritative
4. 若超阈值：rollback
5. replay 后续输入

### 完成标准
- 纯单元测试验证：
  - ack 一致时不重演
  - ack 偏差时 rollback + replay
  - replay 结果稳定

## 4. Rustler 第一阶段接入策略

## 4.1 本阶段是否直接用 Rustler
### 建议
- **是，至少把 façade 和 crate 边界建出来**
- 但第一阶段不要求把所有逻辑都下到 NIF

### 最小接入方案

#### Stage 1A
先实现：
- 客户端 Rust 预测器（纯 Rust）
- 服务端 Elixir façade + 纯 Elixir integrator
- 同时建好 `apps/scene_server/native/movement_engine/`

#### Stage 1B
再把：
- `step`
- `replay`

迁入 Rustler

## 4.2 这样做的好处
- 架构边界一次建对
- 实现风险可控
- 单测先跑起来
- 后续替换引擎不动上层业务

## 7. 开始实现前的最后确认

### 这份清单没有结构问题，原因是：
- 它按层拆分，不会把所有逻辑继续堆进 `app.rs` / `player_character.ex`
- 它把 Rustler 边界先立住，避免后续再大拆
- 它把协议、模拟、表现、控制面分开，符合高内聚低耦合
- 它直接面向最终架构，不再绕过渡期

### 真正开始实现时，建议执行顺序
1. **协议 DTO + 编解码**
2. **客户端 history / predictor / reconcile**
3. **服务端 movement domain façade**
4. **接通真实 movement path**
5. **presentation 层彻底脱钩**

这就是 vNext 第一阶段实施清单。
