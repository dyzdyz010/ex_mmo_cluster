# 2026-04-13 NPC 运行时收口

## 本轮目标

把 NPC 从“仅接入 AOI / combat targeting 的第一阶段”推进到：

- 拥有完整的服务端活跃 actor 主循环
- 可被客户端稳定识别为 NPC，而不是匿名 remote player
- 具备 chase / attack / return_home / death / respawn 的最小闭环
- 具备 server stdio 与 E2E 可验证能力

## 最终裁决

### 1. 活跃 NPC 仍然保持“一只一个 GenServer”

本轮没有改变这个顶层设计，而是把它真正跑通：

- `NpcManager` 负责 spawn / 索引
- `NpcActor` 负责单体 NPC 的权威状态、脑逻辑、移动、战斗、生命周期
- `NpcBrain` 继续保持纯决策函数
- `NpcNavigation` 负责把行为意图翻译成固定 tick movement input

### 2. 实时主循环继续使用确定性的 rule-based AI

本轮依然没有把 AI agent / LLM 放入实时环。

实时权威环中运行的是：

- `NpcBrain.decide/2`
- `NpcNavigation.build_input_frame/6`
- fixed-tick movement + combat

高层内容生成或脚本生成未来可以用 AI agent，但不进入实时权威步进。

## 本轮完成

### 1. NPC 拥有完整 fixed-tick movement 主循环

`SceneServer.Npc.Actor` 现在除了 brain tick 之外，还拥有独立的 movement tick：

- chase 时朝目标持续推进
- return_home 时回出生点
- attack / idle / dead 时切为 brake / stop
- movement 通过 `SceneServer.Movement.Engine` 权威积分
- AOI 位置通过 `RemoteSnapshot` 持续广播

这意味着 NPC 不再只是“能打人的静态点”，而是进入了与玩家相同的移动时间线抽象。

### 2. NPC 的攻击配置已从全局玩家技能中解耦

新增：

- `SceneServer.Npc.Attack`

并把以下能力变成 `NpcProfile` 可配置项：

- `skill_id`
- `skill_damage`
- `skill_radius`
- `skill_cooldown_ms`
- `max_hp`
- `respawn_ms`

因此训练 NPC 与玩家技能配置不再强耦合。

### 3. AOI 广播已增加 actor identity 元数据

新增下行语义：

- `ActorIdentity`

AOI enter 时，订阅方现在不仅收到位置进入事件，还会收到：

- `cid`
- `actor_kind`
- `actor_name`

这样客户端可以稳定区分：

- player
- npc

而不是依赖 CID 约定或临时规则推断。

### 4. 客户端已具备 NPC 识别与表现边界

客户端新增：

- `world/remote_actor.rs`

现在会维护 remote actor identity：

- `RemoteActorKind`
- `RemoteActorIdentity`

并且：

- NPC 与玩家使用不同颜色/尺寸表现
- stdio 新增 `npcs` 命令
- snapshot 新增 `remote_npc_count`

### 5. server stdio 已覆盖 NPC 查询

服务端 stdio 新增：

- `npcs`
- `npc <cid>`
- `npc_state <cid>`

这让 agent 可以直接查询：

- NPC 是否存在
- 当前位置
- alive / hp / max_hp
- deaths
- 当前 intent / target

### 6. 训练 NPC demo 已变成稳定 E2E 目标

`Training Slime` 现在是一个明确的训练目标：

- 会 chase 玩家
- 会攻击玩家
- 能被玩家击杀
- 会 respawn
- 数值已调整为适合自动化验证

## E2E 验证点

`scripts/e2e-stdio.ps1` 现在会额外验证：

1. 客户端收到 NPC identity
2. client stdio `npcs` 能看到 `90001`
3. server stdio `npcs / npc / npc_state` 返回训练 NPC
4. NPC 死亡已被观测到（stdio 或 client observe）
5. NPC 最终已 respawn 且满血
6. NPC 对玩家的 `combat_hit` 已被观测到

## 本轮的架构意义

这一轮真正完成的是：

> **NPC 从“服务端可表示的实体”升级为“服务端权威 runtime actor + AOI actor + combat actor + client-identifiable remote actor”。**

也就是说：

- 服务端有独立 actor 生命周期
- 网络层有稳定 identity 语义
- 客户端有稳定类型识别
- QA/E2E 可以自动证明它是活的、会动、会攻击、会死亡、会重生
