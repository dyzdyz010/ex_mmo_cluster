# 2026-04-12 移动同步项目实现蓝图（最终版）

## 1. 文档目的

本文档给 `ex_mmo_cluster` 提供一份**可直接指导实现**的移动同步蓝图，目标不是描述一个过渡方案，而是直接定义最终形态：

- 客户端采用**输入流 + 本地预测 + 服务器权威校正 + rollback/replay**
- 远端玩家采用**snapshot buffer + interpolation**
- 表现层采用**presentation proxy / camera smoothing / animation smoothing**
- 运动学正式支持 **acceleration**，并将 **jerk（加加速度）** 作为运动 profile 参数
- 将计算密集型、纯算法型运动核心下沉到 **Rust/Rustler**
- 保持整体架构**高内聚、低耦合、可测试、可演化**

---

## 2. 顶层设计裁决

### 2.1 直接演进，不保留过渡架构

不再长期保留“客户端上传位置、服务器再修正”的老思路。最终形态直接采用：

1. **Client -> Server 上传输入流**，而不是上传位置结果
2. **Server -> Client 返回权威 ack**
3. **Client 本地根据输入立即预测**
4. **收到服务器权威状态后 rollback + replay**
5. **本地相机 / 动画 /渲染跟随 presentation state，而不是直接跟随权威状态**

### 2.2 角色职责裁决

- **GateServer**：会话、协议、transport、转发、控制面
- **SceneServer**：权威运动模拟、合法性检查、玩家状态
- **Bevy Client**：输入采样、本地预测、回滚重演、表现层
- **Rustler / Rust**：重计算、确定性运动积分、远端插值采样、回放

### 2.3 加速度 / jerk 裁决

- **Acceleration**：作为正式运动状态进入系统
- **Jerk**：不作为网络字段，作为 `MovementProfile` 参数由客户端和服务器共享

原因：
- acceleration 是权威运动状态的一部分
- jerk 更像手感塑形参数，应属于模型规则，而不是每帧协议载荷

## 3. 系统分层

## 3.1 六层模型

### A. Input Layer
负责：
- 采样键盘 / 手柄 / stdio 控制输入
- 量化为稳定的输入结构
- 生成 `MoveInputFrame`

### B. Prediction Layer
负责：
- 本地玩家预测积分
- 输入历史缓冲
- 预测状态历史缓冲

### C. Authority Layer
负责：
- SceneServer 的权威运动积分
- 合法性检查
- 碰撞与约束（未来）

### D. Reconciliation Layer
负责：
- ack 应用
- rollback
- replay / resimulation

### E. Replication Layer
负责：
- 输入流上行
- 权威 ack 下行
- 远端快照下行

### F. Presentation Layer
负责：
- 本地玩家的平滑表现
- 远端玩家的插值表现
- 相机跟随
- 动画融合

## 4. 设计模式与原因

## 4.1 Command Pattern
用于：
- `MoveInputFrame`
- `SkillCommand`
- `InteractionCommand`
- `ChatCommand`
- stdio 命令

理由：
- 天然适合排队、缓存、重放、测试
- 输入与行为统一表达，便于录制与回放

## 4.2 Event-Sourcing-lite / Timeline Buffer
用于：
- 本地输入历史
- 本地预测状态历史
- 远端快照缓冲
- correction 时间线

理由：
- rollback/replay 本质就是沿时间线重算

## 4.3 Strategy Pattern
用于：
- movement integrator strategy
- interpolation strategy
- correction smoothing strategy
- camera smoothing strategy

理由：
- 便于从线性插值升级到 velocity-aware / Hermite
- 便于后续切换不同运动模式（地面/空中/冲刺）

## 4.4 State Pattern
用于：
- `MovementMode`：grounded / air / dash / disabled
- `NetworkState`：authing / in_scene / fast_lane / fallback
- `ReplayState`：clean / diverged / replaying

理由：
- 避免大量布尔标志和分支污染核心逻辑

## 4.5 Ports and Adapters / Hexagonal
用于：
- 输入适配器（键盘 / stdio）
- 网络适配器（TCP / UDP）
- Rustler 计算适配器
- observe / stdio interface

理由：
- 控制面与模拟面解耦
- 调试接口不会直接绑死核心运动引擎

## 4.6 Facade + Anti-Corruption Layer
用于：
- `MovementEngineFacade`
- `ProtocolAdapter`

理由：
- 上层不直接依赖 Rustler / NIF 细节
- 协议演进不直接污染客户端/服务端所有模块

## 5.1 LocalPlayerAggregate

本地玩家应作为一个聚合根：

```text
LocalPlayerAggregate
  - input_history
  - predicted_history
  - authoritative_anchor
  - latest_ack
  - movement_profile
  - presentation_proxy
```

### 责任
- 接受本地输入帧
- 执行本地预测
- 应用服务器 ack
- 发生偏差时 rollback + replay
- 对外提供：
  - predicted state
  - authoritative anchor
  - presentation state

### 高内聚体现
- 本地移动的所有核心状态都放在一个聚合里

### 低耦合体现
- 相机、动画、渲染层不直接读网络包或 ack
- 它们只读 `presentation_proxy`

## 5.2 RemotePlayerReplica

远端玩家单独作为复制聚合：

```text
RemotePlayerReplica
  - snapshot_buffer
  - interpolation_cursor
  - presentation_proxy
```

### 原则
- 远端玩家不参与 rollback/replay
- 远端玩家不做本地预测
- 远端玩家只走快照缓冲 + 插值时间线

Scene 才是 authoritative movement domain。

## 6.3 PlayerCharacter 最终职责

`SceneServer.PlayerCharacter` 应只承担：
- 接收 `MoveInputFrame`
- 调用 movement engine
- 持有权威状态
- 构建 ack / remote snapshots

不要让它既做：
- transport
- 广播
- 大量规则判断
- 手工积分细节

这些要拆出去保持单一职责。

## 7. 最终协议

## 7.1 Client -> Server：输入流

```text
MoveInputFrame {
  seq: u32
  client_tick: u32
  dt_ms: u16
  input_dir_x: f32
  input_dir_y: f32
  speed_scale: f32
  movement_flags: u16
}
```

### 含义
- `seq`：确认/删除输入历史的关键
- `client_tick`：回滚与时序对齐关键
- `dt_ms`：客户端预测步长
- `input_dir_x/y`：方向输入
- `speed_scale`：速度倍率（跑步/减速等）
- `movement_flags`：跳跃、制动、冲刺等扩展位

## 7.2 Server -> Local Client：权威 ack

```text
MoveAck {
  ack_seq: u32
  auth_tick: u32
  position: Vec3
  velocity: Vec3
  acceleration: Vec3
  movement_mode: u8
  correction_flags: u32
}
```

### 用途
- 删除已确认输入
- 找到权威锚点
- rollback + replay
- 做 correction 分类处理

## 7.3 Server -> Remote Clients：远端复制快照

```text
RemoteMoveSnapshot {
  cid: i64
  server_tick: u32
  position: Vec3
  velocity: Vec3
  acceleration: Vec3
  movement_mode: u8
}
```

## 8. 运动学模型

## 8.1 正式状态

推荐权威/预测状态至少包含：

```text
MovementState {
  position: Vec3
  velocity: Vec3
  acceleration: Vec3
  movement_mode: MovementMode
}
```

## 8.2 MovementProfile

```text
MovementProfile {
  max_speed: f32
  max_accel: f32
  max_decel: f32
  max_jerk: f32
  friction: f32
  turn_response: f32
}
```

### acceleration 与 jerk 的关系
- acceleration 是运动状态
- jerk 是控制 acceleration 变化率的 profile 参数

## 8.3 推荐积分器

```text
wanted_velocity = normalize(input_dir) * max_speed * speed_scale
velocity_error = wanted_velocity - current_velocity
accel_target = clamp_length(velocity_error / dt, accel_or_decel_limit)
acceleration = smooth_toward(prev_acceleration, accel_target, max_jerk, dt)
velocity = clamp_speed(current_velocity + acceleration * dt)
position = current_position + velocity * dt
```

### 优点
- 起步、刹车、转向都更自然
- acceleration 可正式成为同步状态
- jerk 提供更高级手感控制

## 9. 回滚 / 重演（Rollback / Replay）

## 9.1 客户端机制

客户端维护：
- `input_history[seq]`
- `predicted_state_history[tick]`

### 收到 ack 后
1. 删除 `<= ack_seq` 的输入
2. 找到 `auth_tick` 对应预测状态
3. 比较 predicted 与 authoritative 的误差
4. 若误差超阈值：
   - rollback 到 `auth_tick`
   - replay 所有后续输入
5. 重新生成当前 predicted state
6. presentation 层平滑追近

### 关键原则
**不能像动作队列那样直接丢弃所有后续移动输入。**

正确做法是：
- correction 后 replay
- 保留玩家输入连续性

## 9.2 服务器 replay 窗口

服务器可以维护小窗口：
- 用于 debug
- 用于 anti-cheat
- 用于未来 rewind features

但不要求像客户端那样高频完整 replay。

## 10. 远端插值

## 10.1 推荐最终方案

- snapshot buffer
- interpolation delay（100~150ms）
- velocity-aware interpolation
- 之后可升级为 Hermite interpolation

## 10.2 原则

- 远端玩家优先插值，不参与本地预测
- 只在缺包时做有限 extrapolation
- 一旦新包到来立即回到插值轨道

## 11. 表现层（重点）

## 11.1 本地玩家

必须拆成：
- `authoritative_state`
- `predicted_state`
- `presentation_proxy`

### presentation_proxy 驱动
- sprite / mesh
- camera follow target
- animation blend

## 11.2 相机

### 硬性原则
相机永远跟随：
- `presentation_proxy`

而不是：
- `authoritative_state`
- `raw ack position`
- `raw local_position`

## 11.3 动画

动画驱动信号优先使用：
- smoothed velocity
- input intent
- movement_mode

不要直接用 ack 间坐标差作为动画速度来源。

## 12. Rustler 使用边界

## 12.1 最适合下沉到 Rustler 的内容

### A. 统一运动积分器
- `step(state, input, profile, dt)`

### B. rollback / replay
- `replay(anchor_state, inputs[], profile, dt)`

### C. 远端插值采样器
- `sample(snapshot_a, snapshot_b, t)`
- 可做 Hermite / velocity-aware interpolation

### D. 批量轨迹预测
- 用于 debug / QA / future tooling

### E. 重型空间计算
- 如果 movement 未来与 AOI / spatial query 更深耦合

## 12.2 不适合放 Rustler 的内容

- GenServer 生命周期
- 会话管理
- supervision
- 网络 IO
- 协议编排
- stdio / observe 控制面

## 12.3 推荐模式

### Elixir Facade

```text
SceneServer.Movement.Engine
  -> RustlerMovementEngine.step(...)
  -> RustlerMovementEngine.replay(...)
  -> RustlerMovementEngine.sample_remote(...)
```

这样：
- 上层永远依赖 Elixir facade
- NIF 细节被包在 adapter 后面

## 12.4 风险控制

- NIF 输入输出必须是纯 DTO
- 不把复杂 Elixir 状态图直接交给 NIF
- 对 replay 做窗口限制
- 对重计算函数优先考虑 dirty CPU schedulers

## 12.5 最终裁决

**本项目的 movement integrator / replay / interpolation 非常适合放 Rustler。**

因为它们：
- 计算密集
- 纯数据导向
- 逻辑清晰
- 易做 property/fuzz 测试
- 对性能和一致性收益很大

## 17. UE5 客户端迁移可行性裁决

### 17.1 是否可以迁移到 UE5？
可以，而且**核心策略本身与引擎无关**。

真正与引擎耦合的只有：
- 输入采样接口
- 表现层（动画蓝图 / 相机 /动画节点）
- 物理角色组件接口

而以下这些是**可以直接迁移概念和协议**的：
- 输入流协议 `MoveInputFrame`
- 服务器权威 `MoveAck`
- rollback / replay 思路
- remote snapshot interpolation
- acceleration / jerk movement profile
- presentation proxy 分层

### 17.2 UE5 中 locomotion 是什么？
是的，UE5 里通常会把“locomotion”理解为：
- 角色移动相关的**动画表现系统**
- 包括 BlendSpace、State Machine、Orientation Warping、Stride Warping、Motion Matching 等

但要注意：

- **locomotion != 网络同步核心**
- locomotion 更偏向**表现层 / 动画层**
- 网络同步核心仍然是：
  - movement simulation
  - prediction
  - correction
  - interpolation

也就是说，在 UE5 里应当坚持：

- **Movement simulation / network prediction** 决定“角色真实应该怎么动”
- **Locomotion / animation graph** 决定“角色看起来怎么动得自然”

### 17.3 UE5 里的推荐对应关系

#### 本地玩家
- `UCharacterMovementComponent` 或 `Mover + Network Prediction` 负责：
  - 输入驱动移动
  - 预测
  - correction
  - authoritative movement state
- `Animation Blueprint / locomotion graph` 负责：
  - 根据速度、方向、movement mode 驱动动画
  - 用 Orientation Warping / Stride Warping / Motion Matching 做自然表现

#### 远端玩家
- replicated movement snapshots + UE5 proxy smoothing
- 动画层读取：
  - smoothed velocity
  - acceleration
  - facing
  - movement mode
- 使用：
  - BlendSpace
  - Orientation Warping
  - Stride Warping
  - 必要时 Motion Matching

### 17.4 UE5 最适合映射的层

#### 可直接迁移的“逻辑层”
- input timeline
- authoritative state model
- rollback/replay
- snapshot interpolation
- movement profile

#### 需要引擎适配的“表现层”
- camera system
- animation blueprint
- locomotion graph
- warping nodes

所以本蓝图天然适合未来迁到 UE5，只要坚持“协议/模拟/表现分离”。

## 18. UE5 中远端角色如何实现自然移动

### 18.1 原则
远端角色不要做本地玩家那种重 replay 预测，而是：

1. **接收服务器快照**
2. **缓存在 snapshot buffer 中**
3. **按插值时间线播放**
4. **把插值后的状态喂给 locomotion / animation graph**

### 18.2 三层远端表现模型

#### A. Replicated State
来自网络：
- position
- velocity
- acceleration
- movement_mode
- facing / root heading

#### B. Interpolated Presentation State
客户端本地计算：
- interpolated_position
- interpolated_velocity
- interpolated_acceleration

#### C. Animation Locomotion State
动画层使用：
- speed
- local move direction
- turn angle
- gait
- grounded/air

### 18.3 为什么这样最自然
因为：
- 网络层只负责“正确状态”
- 插值层负责“平滑时间线”
- 动画层负责“动作自然”

不把这三者混在一起，远端角色才不会出现：
- 走路像飘
- 转向抽搐
- 脚步和位移不同步

### 18.4 UE5 推荐动画技术组合

对远端角色，推荐组合：

1. **BlendSpace Locomotion**
   - 基础速度/方向混合
2. **Orientation Warping**
   - 修正移动方向与角色朝向差异
3. **Stride Warping**
   - 让步幅匹配真实移动速度
4. **Motion Matching（可选）**
   - 若项目追求高品质移动表现

### 18.5 对你的架构的具体映射

#### 本项目中的 `RemotePlayerReplica`
未来迁到 UE5 时，可映射为：
- replicated movement component data
- interpolation proxy
- animation blueprint input struct

#### 本项目中的 `presentation proxy`
未来迁到 UE5 时，可映射为：
- actor visual transform target
- camera follow target（仅本地）
- animation graph locomotion inputs

### 18.6 最终裁决

如果未来有 UE5 客户端，这套蓝图**完全可以迁移**，而且比“直接在 UE5 里边写边猜网络同步”更先进。正确做法是：

- 把本蓝图的**协议和模拟层**保持引擎无关
- 在 UE5 中只重建：
  - movement adapter
  - presentation adapter
  - locomotion adapter

也就是说：

> **UE5 的 locomotion 是表现层，网络同步核心仍然应该沿用本蓝图中的输入流、权威状态、rollback/replay 与 snapshot interpolation。**
