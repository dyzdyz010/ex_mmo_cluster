# 2026-04-15 技能复制与视觉传播最佳实践说明

## 本轮采用的核心原则

### 1. 服务器权威决定 cast / hit / damage / follow-up

本轮没有把 projectile 命中、AOE 判定或 chained follow-up 交给客户端决定。客户端只负责：
- 发起 cast 请求
- 渲染服务器广播的 effect cue

真正的：
- 目标选择
- travel time
- impact timing
- damage 应用
- follow-up 触发

都由服务器权威完成。

## 2. 状态传播与视觉传播分离

本轮明确分离：

### 状态正确性
- `CombatHit`
- `PlayerState`

### 视觉正确性 / 自然度
- `EffectEvent`

这样做的好处是：
- 客户端不会从 HP 变化反推视觉
- 远端也能收到同样的视觉 cue
- 网络状态与视觉效果职责明确分离

## 3. projectile 使用“权威延迟结算 + cue 播放”

对于远程单体和触发技能：
- 服务器根据距离和 projectile speed 计算 `travel_ms`
- 先广播 `projectile` cue
- 再在正确时间点应用 impact 与 follow-up

客户端收到 cue 后：
- 只播放从 origin 到 target 的视觉运动

也就是说：
- projectile 的**命中结果**是权威的
- projectile 的**飞行表现**是 replicated cue 驱动的

## 4. follow-up 也是通用 effect spec，不是专门写技能模块

触发型技能不是靠写一个独立模块实现，而是：
- primary effect
- nested follow-up effects

例如：
- projectile 命中 primary target
- follow-up AOE anchored on target
- follow-up chain single-target anchored on target

都在统一 `EffectSpec` 结构内表达。

## 与业界常见做法的对应

### Valve
经典 latency compensation 资料里，projectile 与 hitscan 的处理是分开的；projectile 不能简单等同于“立即回溯命中”。本轮方案保持：
- authoritative projectile resolution
- authoritative delayed impact

参考：
- https://developer.valvesoftware.com/wiki/Latency_Compensating_Methods_in_Client/Server_In-game_Protocol_Design_and_Optimization

### Unreal GAS
Gameplay Ability System 常见做法是：
- gameplay result 由 authority 确定
- visual cue 通过 GameplayCue 传播

本轮的：
- `CombatHit` / `PlayerState`
- `EffectEvent`

本质上就是同样的职责分离思路。

参考：
- https://dev.epicgames.com/documentation/en-us/unreal-engine/using-gameplay-abilities-in-unreal-engine
- https://dev.epicgames.com/documentation/en-us/unreal-engine/API/Plugins/GameplayAbilities/UAbilitySystemComponent/ForceReplication

### Unity Netcode
Unity multiplayer / Netcode 系的核心原则之一是：
- authority state 与 presentation timeline 分离
- 客户端渲染不应替代 authority result

本轮也遵循这个思路：
- 远端运动由 snapshot interpolation
- 技能视觉由 replicated cue
- 伤害与 HP 仍然走 authority state

参考：
- https://mp-docs.dl.it.unity3d.com/netcode/

## 当前结论

本轮的技能复制与视觉传播策略可以概括为：

> **服务器权威决定结果，客户端消费显式 gameplay cue 做视觉，状态同步与视觉同步职责分离。**

这就是当前 demo 下最稳、最通用、也最接近产业实践的做法。
