# 2026-04-13 NPC 服务端表示设计

## 结论

### 是否可以一个 NPC 一个 GenServer？

**可以，但要加限定条件：**

> **一个“活跃 NPC / AOI 内 NPC / 已激活 NPC”一个 GenServer 是合理的。**

但不应该直接把“全世界所有静态配置里的 NPC”都常驻成进程。

更合理的顶层结构是：

1. **静态 NPC 定义**
   - 配置 / 模板 / 生成点
   - 不直接对应常驻进程

2. **激活中的 NPC Actor**
   - 一只活跃 NPC = 一个 `GenServer`
   - 负责该 NPC 的权威状态与生命周期

3. **NPC Manager / Activation Layer**
   - 决定哪些 NPC 需要被激活
   - 管理 spawn / despawn / reset / 索引

也就是说：

- **活跃 NPC：一只一进程**
- **静态世界配置：数据，不是一只一进程**

---

## 为什么这个模型适合当前项目

你当前项目已经逐渐形成了：

- 权威 fixed-tick movement
- `PlayerCharacter` 作为玩家权威聚合根
- combat domain
- AOI 广播
- stdio / E2E / observe

所以 NPC 最自然的方式不是另开一套“特殊系统”，而是：

> **让 NPC 也成为一种权威 actor / aggregate，只是输入源不是客户端，而是服务器侧 Brain。**

这和玩家模型是同构的：

- 玩家：
  - 输入来自客户端
  - 服务器权威推进
- NPC：
  - 输入来自服务器 Brain
  - 服务器权威推进

---

## 顶层推荐架构

## 1. NPC 静态定义层

建议长期存在：

- NPC 模板
- 出生点
- 基础属性
- 行为树/决策树类型
- 技能配置
- 掉落配置

这一层可以来自：

- 数据表
- json
- 地图脚本

但它**不是运行时 actor 本身**。

---

## 2. NPC 运行时聚合根（NpcActor）

### 推荐：一只激活中的 NPC 一个 `GenServer`

`NpcActor` 应该拥有：

- npc id
- spawn position
- movement state
- combat state
- current target
- brain state
- cooldowns
- 计时器 / fixed tick 状态

这和现在 `PlayerCharacter` 的定位很类似。

### 为什么值得一只一进程

优点：

- 高内聚：一个 NPC 的状态只在一个 actor 里
- 生命周期明确：spawn / death / respawn / leash / reset 清晰
- 易于调试：`GenServer.call(pid, :get_state_summary)` 非常自然
- 易于接入 AOI / combat / fixed-tick
- 与当前 OTP 架构一致

### 什么时候不适合

如果未来某张地图要同时激活：

- 几千
- 上万

只靠“一只一进程 + 各自 timer”会开始有压力。  
到那时就要引入：

- shared tick
- activation partition
- region sharding

但以你当前项目阶段，**一只活跃 NPC 一进程是合理且优越的**。

---

## 3. NPC Brain：建议用“纯函数决策树 / utility AI”，不要让 AI agent 进实时闭环

你问：

> 是否可以通过决策树和 AI agent 来驱动？

### 我的结论

#### 决策树 / 行为树 / utility AI：
**适合，推荐。**

#### LLM/AI agent 直接驱动实时权威行为：
**不推荐进入实时主循环。**

---

## 为什么不推荐 AI agent 直接进实时闭环

因为实时权威 NPC 需要：

- 低延迟
- 稳定
- 可重复
- 可测试
- 可预测

而 AI agent / LLM 天然会带来：

- 时延
- 不确定性
- 难以复现
- 成本波动

这会直接破坏：

- fixed-tick
- 权威一致性
- QA 稳定性

所以：

> **AI agent 可以做“高层规划 / 内容生成 / 离线作者工具 / 非实时导演系统”**  
> **但不应该直接替代 NPC 的实时 Brain。**

---

## 推荐的分层

### 运行时实时闭环

- `NpcActor`
- `NpcBrain`
- `NpcCombat`
- `NpcMovement`

这些都应是：

- 本地纯函数
- 可测试
- 低延迟

### AI agent 的合理位置

AI agent 更适合做：

1. 生成 NPC 配置
2. 生成行为脚本
3. 高层任务规划（例如“今天巡逻这里、明天迁徙那里”）
4. GM/导演系统

而不是每 100ms 决定一次“往左走还是往右走”。

---

## 推荐行为架构

## 1. NpcBrain 用纯函数决策

输入：

- 当前 npc facts
- 当前 profile
- 目标状态
- 距离
- leash 状态

输出：

- `:idle`
- `:chase`
- `:attack`
- `:return_home`
- `:dead`

也就是：

> **Brain 只做“决策”**

不要让它直接改世界。

---

## 2. NpcActor 做权威状态机

`NpcActor` 收到 Brain 决策后，再决定：

- movement input
- attack request
- leash reset
- respawn

也就是：

> **Actor 持有状态，Brain 只给出 intent**

这符合：

- State pattern
- Strategy pattern
- Command pattern

---

## 3. fixed tick 最好共享，不要长期每只 NPC 一个独立高频 timer

当前项目阶段可以先：

- 一只 NPC 一个 `GenServer`
- 每只 NPC 一个定时 tick

但从顶层设计来说，长期更优的做法是：

### 推荐最终形态

- `NpcActor` 仍然是一只一进程
- 但 tick 来源尽量统一：
  - scene shared tick
  - npc manager broadcast tick
  - region tick

也就是：

> **一只一进程，但不要一只一高频计时器作为最终形态**

这是为了以后大规模扩展。

---

## 4. 与玩家的关系

NPC 应该尽量复用玩家已有的权威体系：

- movement model
- combat model
- AOI model
- player state / entity state broadcast 思想

这样：

- 玩家和 NPC 共享大部分服务端规则
- 不会变成两套完全不同的战斗/移动世界

最理想的方向是：

> **PlayerCharacter 和 NpcActor 共享一层更高阶的 Actor Domain 语义**

比如以后可以抽出：

- movement state
- combat state
- state summary
- death / respawn

---

## 当前建议实现顺序

### 第一阶段

先实现：

- `NpcProfile`
- `NpcState`
- `NpcBrain`
- `NpcActor`
- `NpcManager`

让服务端先有“NPC 作为权威 actor”的表示。

### 第二阶段

再接入：

- AOI enter/move/leave
- combat hit
- player target selection

### 第三阶段

再做：

- spawn/despawn 激活策略
- shared tick
- region 分片

---

## 最终一句话裁决

> **是的，活跃 NPC 完全可以一只一个 GenServer；但实时驱动应当用可测试的决策树/行为树/utility AI，AI agent 只适合作为离线或高层规划层，而不应进入实时权威主循环。**
