## 1. 环境约束与验证基础设施(动手前必读)

### 1.1 BEAM 无法在 Claude harness 内运行(已实测)
- `erl` / `mix` / `elixir` 在本会话的 PowerShell 与 Bash 工具中**100% segfault**(exit 139 / Windows access violation `0xC0000005`),与沙箱开关、单调度器 `+S 1`、io 重定向均无关。
- 判断:**harness 非交互进程模型 ↔ BEAM SMP 调度器线程不兼容**,非安装损坏。重装 Erlang/Elixir 对此 harness 大概率无效,但不影响用户交互式终端。
- **后果:所有 Elixir 编译/测试验证必须在用户交互式终端执行**(或 Claude Code 提示符下用 `! <cmd>` 前缀,使输出回灌会话)。Claude 负责改代码 + 静态/多 agent 审查 + 生成精确验证命令;运行时验证由用户终端兜底。

## 3. 验证协议(每阶段强制执行)

每个修复项遵循 **green-before → 实施 → green-after + 新测试** 闭环:

1. **green-before**:改动前,在用户终端跑该项涉及 app 的现有测试,确认全绿(记录基线)。
2. **实施**:按本计划的实施步骤改代码;Elixir 改动遵守 AGENTS.md(公共模块补 `@moduledoc`/`@doc`,authority/adapter 边界写明谁拥有状态;子系统目录维护 README)。
3. **新测试**:对每个修复补一条**能复现原问题**的测试(bug → 先红后绿;机制重构 → 不变量/故障注入测试)。
4. **green-after**:跑 ① 的同一组测试 + 新测试,全绿;再跑相邻 app 冒烟,确认无回归。
5. **可观测验证**(AGENTS.md 硬边界):凡涉及运行时行为,用对应 `mix <app>.*_observe` CLI 任务 + 结构化 observe 日志确认行为符合预期,不只靠单测。

### 阶段 1 — 自包含机制修复(不碰 WIP,多数无需 DB)

- **1.1 · persist-4 · UidGenerator 时序逻辑错误**
  - 位置:`apps/data_service/lib/data_service/worker/uid_generator.ex:63-68`
  - 根因:`get_sequence` 比较方向反转 → 同毫秒内序列号恒为 0,主键碰撞;跨毫秒无界累加后被静默截断。
  - 实施:重写三态逻辑 —— 同毫秒递增序列、跨毫秒重置序列、时钟回拨告警/自旋;序列溢出时自旋到下一毫秒;`service_id` 保证集群唯一(配置校验)。
  - 测试:新增 `uid_generator_test.exs` 用例:同一毫秒内连续生成 N 个 → 全部唯一且严格递增;跨毫秒重置;回拨处理。

  - 依赖:无。风险:低(自包含)。

- **1.2 · field-2 · clear_layer_in_aabb 稠密写 → 稀疏删除**
  - 位置(四处同构):`voxel/field/kernels/conduction_path_kernel.ex:437-445`、`electric_discharge_kernel.ex:451-459`、`circuit_current_kernel.ex:158-166`、`voxel/field/electric_field.ex:274-283`;底层 `voxel/field/field_layer.ex`。
  - 根因:每 10Hz tick 用三层 for 对全 AABB 逐格 `FieldLayer.put(_, idx, 0.0)`,与稀疏 delta 存储矛盾(产生 4096 次"写再删")。
  - 实施:在 `FieldLayer` 增 `clear_in_aabb(layer, aabb)`(或 `clear_all/1`),直接从 `values` map 删除落在 AABB 内的 key(稀疏语义),不逐格写 0;四处 kernel 改调该 API。统一抽出避免再次复制(消除四份同构)。
  - 测试:`field_layer` 单测:稀疏层 clear 后 values 不含 AABB 内 key 且语义等价于逐格写 0;kernel tick 前后场值一致性。
  - 验证:`cd apps/scene_server && mix test test/scene_server/voxel/field/ --no-start`(无需 DB)。
  - 依赖:无。风险:低-中(需保证清除语义完全等价,有测试兜底)。

- **1.3 · field-4 · 跨 chunk 两段式场区域原子性**
  - 位置:`voxel/field/field_runtime.ex:766-808`
  - 根因:source shard 成功 / target shard 失败时状态不一致(无回滚/补偿)。
  - 实施:两段式提交化 —— 先在两端 prepare(可校验、可回滚),全部 ok 才 commit;任一失败回滚已 prepare 端;或引入 idempotent 重试 + 对账。复用 world 侧 2PC 思路的轻量版(单 runtime 内)。
  - 测试:故障注入:target 失败时 source 不残留;重入幂等。
  - 验证:scene_server field 测试。依赖:无(但与阶段 4 的 2PC 思路呼应)。风险:中。
  - **完整核实结论(2026-06-04 执行时逐函数读完跨 chunk conduction 协调链 558-808 / 1040-1090)**:审查报告"无回滚/补偿"的描述**不准确**。当前补偿**功能正确**:① 整条 `with` 链前两个校验(`validate_conduction_channel` / `validate_power_source_policy`,均为纯读校验,**不建 region**)用统一 `cleanup_cross_chunk_on_error` 包裹;② source region 唯一在 766 `ensure_field_region(source)` 才建立,故第三步 `validate_cross_chunk_target_boundary`(709,未包裹)失败时 source 尚未建立,**不泄漏**;③ source 建成、target 建失败时 766-808 手写补偿 `release source + destroy target`,与统一 `cleanup_cross_chunk_conduction` 等价(`release_field_region_source` 设计为幂等不失败,`_ =` 未吞真实错误信号)。
  - **唯一真实正确性缺陷**:source 与 target 两次 `ensure` 之间存在**崩溃窗口**——若此刻协调进程或 source/target chunk 崩溃,补偿来不及执行,source region 残留无人回收。根治需要给跨 chunk 场区域引入**带 transaction_id 的 prepare/commit + 崩溃恢复扫描**,这正是阶段 4(world-2pc 闭环 + recovery reaper)的机制。
  - **结论:1.3 重定性,完整原子性归入阶段 4 统一设计**;阶段 1 **不做**局部 DRY 补偿重构(零正确性提升 + 改复杂函数有未测风险 + 与阶段 4 返工)。这符合"从架构层面整体修复,不打补丁"。

- **1.4 · scene-rust-1 · 所有 NIF 加 dirty scheduler**
  - 位置:`scene_ops/src/lib.rs:69-105`、`octree/src/lib.rs:43-82`、`field_kernel/src/lib.rs:20-95`、`coordinate_system/src/lib.rs:330-376`、`movement_engine/src/lib.rs:11-27`(逐函数评估)。

  - 依赖:无。风险:中(需重编 NIF;dirty 标注错配会有性能/正确性影响,逐函数审)。

## 8. 重启/重装后的第一步(给用户)

1. 重装 Erlang/Elixir 后,在**交互式终端**确认 `erl -version`、`mix --version` 正常。
2. `cd ex_mmo_cluster && mix deps.get`;在 **VS Dev Command Prompt** 下 `mix compile`(NIF 需要)。
3. 起 Docker postgres + `mix ecto.create/migrate`(§1.3)。
4. 跑全量测试,把各 app 通过数填进 §7 的 green-before 列(基线)。
5. 从阶段 1.1(UidGenerator,最自包含)开始,逐项 green-before → 实施 → green-after。
6. 若希望 Claude 驱动执行:Claude 负责改代码 + 多 agent 对抗审查 + 出验证命令;**运行时验证由你的终端执行**(或 `! mix test ...` 回灌会话),因为 Claude harness 跑不了 BEAM。
