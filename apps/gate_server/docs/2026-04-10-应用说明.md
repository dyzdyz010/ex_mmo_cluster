# GateServer 应用说明

**更新日期：** 2026-04-10

## 当前定位

`gate_server` 是客户端进入服务集群的 TCP 网关，负责自定义二进制协议编解码、连接状态管理、连接级时间同步，以及 scene/auth 分发。

## 当前已实现

- TCP 监听与 `{packet, 4}` 分帧
- `GateServer.Codec` 自定义二进制协议
- 与 `scene_server` 的进入场景 / movement / AOI 广播链路
- 与 `scene_server` 的 scene-local chat / simple skill AOI 广播链路
- 与 `auth_server` 的 token 验证调用
- 关键消息顺序约束：auth 后才能 `EnterScene`，进入场景后才能 `Movement`
- `TimeSync` 已重构为 gate 级连接协议，不再依赖 scene player 内部占位逻辑
- auth 成功后连接会保存显式 session 上下文（如 `username`、`session_id`、claims）
- `AuthRequest.username` 会与 token claims 中的用户名比对
- 当 token claims 声明了 `cid` 或 `allowed_cids` 时，`EnterScene(cid)` 会执行约束校验
- `AuthRequest` / `EnterScene` / `Movement` / `TimeSync` 现在都使用 request_id 关联语义
- 已实现 UDP fast-lane bootstrap：TCP 可签发 attach ticket，UDP 可完成 attach 握手
- attached UDP peer 现在可以发送 `Movement` 上行，gate 会按会话把它投递到 scene
- attached UDP peer 现在也会通过 UDP 接收 `PlayerMove` 下行广播
- UDP fast lane 现在具备基础运行时治理：重附着会替换旧 peer、空闲 peer 会过期、TCP 关闭会清理 UDP 绑定
- `MovementResult` 现在会回传服务端权威位置，可直接作为客户端的最小纠偏信号
- `ChatSay` / `ChatMessage` 已形成最小可靠聊天切片，可用于当前调试客户端
- `SkillCast` / `SkillEvent` 已形成最小简单技能切片，当前只保留 `skill_id = 1` 的 pulse 广播语义

## 当前限制

- 当前错误回包仍然是现有通用 error 形式，尚未细化到稳定错误码族
- 角色归属现在会经过 auth -> data_service 的真实来源校验，但角色列表/选择本身仍未完全产品化
- 聊天与技能目前都只做到 scene/AOI 级可靠广播，还不是完整战斗/频道系统
- UDP fast lane 仍缺少 sequence / snapshot / resync 等更强恢复机制

## 下一步建议

建议优先进入：

1. 在当前 fast lane 基础上补齐 snapshot / resync / sequence 等恢复机制
2. 再评估是否需要 KCP 层或更强的传输治理
